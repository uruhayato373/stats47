`/design-note-structure` で設計した構成に沿って、note 記事（B/C/D シリーズ）を執筆する。

> **A シリーズ（ランキング記事・量産型）はこの4ステップワークフローの対象外です。** A シリーズは自動生成スキルで処理するため、validate → design → write → edit の手順は不要です。

## 用途

- 構成設計済みの記事を実際に書き進めたいとき
- stats47 データを「読者に刺さるストーリー」に変換したいとき

## フロー

```
/validate-note-idea → /investigate-note-data → /design-note-structure → ★/write-note-section → /edit-note-draft
```

**鉄則**: 書きながら編集しないこと。まず全体を書き切ってから、編集は `/edit-note-draft` で別チャットで行う。

## 引数

- **articleId**: 記事ID（例: `B-1`, `C-1`, `D-1`）
- **シリーズ**: B / C / D
- **構成**: `/design-note-structure` で設計した章構成
- **活用する stats47 データ**: リンク候補リスト

## ディレクトリ構成

```
docs/31_note記事原稿/<slug>/
├── note.md          ← 本文（このスキルで作成）
├── chart-data.json  ← チャート生成用データ（A シリーズ: このスキルで作成）
├── images/          ← 画像（.gitignore 対象）
└── _data/           ← 作業ファイル
    ├── plan.md          ← validate + design の結果
    ├── research.md      ← investigate の結果
    └── chart-data.json  ← チャート生成用データ（B/C/D シリーズ: investigate で作成）
```

`<slug>` は articleId + テーマから生成する URL-safe な文字列。例: `b1-yakiniku-tohoku`, `c1-household-survey`

## 執筆ペルソナ

あなたは「noteで伸びる記事構造を熟知したコンテンツ編集者」であり、「統計データの活用」に詳しいライターです。
公的統計の面白さと実用性を、専門用語を使わずにわかりやすく伝えます。
読者が「自分でもデータを使ってみたい」「stats47 で見てみよう」と思えるように書いてください。

## note おすすめ表示最適化（必須）

note のアルゴリズム方針に沿い、おすすめ表示されやすく読者満足度が高い記事を目指す。

### 重視する要素

- **一次情報**: 元県庁職員としての実体験・検証・観察を必ず含める。「行政の現場で見てきたこと」が stats47 の最大の差別化要素
- **検証可能な内容**: 主張にはデータの裏付けを示し、stats47.jp リンクで読者自身が確認できるようにする
- **独自視点**: 他の統計記事にはない「なぜこの結果になるのか」の考察を厚く書く
- **事実と意見の分離**: データから読み取れる事実と、筆者の解釈・考察を明確に分けて書く
  - 事実: 「〇〇県が1位で△△円です」
  - 意見: 「この背景には〜があると考えられます」
- **初心者でも理解できる構造**: 統計の前提知識なしで読める平易な文章にする
- **引用元の明記**: データの出典は stats47.jp リンクに委ねつつ、どの統計調査のデータかは文中で触れる

### 記事の6部構成（全シリーズ共通の骨格）

シリーズ別テンプレートに沿いつつ、以下の流れを意識すること:

1. **導入**: なぜこのテーマを取り上げるのか。読者が共感できる問題提起
2. **背景**: テーマに関する事実・状況。データや引用で裏付け
3. **一次情報**: 元県庁職員として実際に見てきたこと、検証したこと
4. **結果**: 得られた結果を数値や具体例で提示
5. **考察（独自視点）**: なぜこの結果になったのか。読者が活かせるポイント
6. **まとめ**: 結論と読者への学び

### 禁止事項

- 煽りタイトル（「衝撃！」「ヤバすぎる」等）
- 攻撃的コンテンツ（地域を貶める表現）
- 他記事の単なるまとめ（オリジナルの考察・一次情報がないもの）
- AI 量産記事のような薄い内容（具体例のない抽象的な記述の連続）

## 前提条件（ハードブロック）

**執筆開始前に以下を確認すること。条件を満たさない場合は執筆を開始しない。**

1. `docs/31_note記事原稿/<slug>/_data/research.md` が存在すること
   - 存在しない場合: 「先に `/investigate-note-data` を実行してください」と案内して**停止**
2. `_data/research.md` に「相関分析結果」セクションがあること
3. `_data/plan.md` が存在すること

## 執筆手順

`plan.md` の構成に沿って一括執筆する。文体は「です・ます」調で、読者に問いかけるような引き込まれるトーンにすること。

1. `_data/plan.md` の構成と stats47.jp リンク候補、および `_data/research.md` のデータ分析結果を読み込む
2. Frontmatter（title, description, tags）を作成
3. 各章を構成テンプレートに沿って執筆
4. stats47.jp リンクを配置（UTM パラメータは不要、素の URL で記載）
5. CTA セクションを作成
6. チャート生成（`/generate-note-charts`）を執筆と同時に行うことを推奨
7. `note.md` に保存

### シリーズ別の執筆ポイント

**B シリーズ（ランキング読み物）:**
- フックが命。最初の3文で「え、そうなの？」と思わせる
- 考察は1,000字以内。ダラダラ書かずにキレよく終わる

**C シリーズ（統計調査ガイド）:**
- 「元行政職員の視点」で調査の裏側を語る。これが独自の価値
- e-Stat の探し方セクションで「加工が大変 → stats47 なら簡単」の誘導を自然に

**D シリーズ（深掘り分析）:**
- 複数指標を横断する構成力が差別化ポイント
- 「なぜこうなったのか」の歴史・政策の文脈を厚く書く

### research.md データの活用

`/investigate-note-data` で生成した `research.md` のデータを活用して執筆すること。相関分析の結果を引用するときは「相関係数 r=0.72」のような具体的な数値を記事に含めないこと。読者向けには「強い関連がある」「ほとんど関連が見られない」等の平易な表現に変換する。

### データ主張の制約（必須）

**記事中で2つの指標の関係性に言及する文は、すべて `research.md` の相関分析結果に裏付けがなければならない。**

- **|r| >= 0.5**: 「強い関連がある」「密接に関係している」等の表現を使用可能
- **0.3 <= |r| < 0.5**: 「一定の関連が見られる」「やや関連がある」等の控えめな表現のみ
- **|r| < 0.3**: 関係性の主張は**禁止**。「データ上は明確な関連が見られない」と正直に書くか、言及しない
- **research.md に記載のない指標ペア**: 関係性の主張は**禁止**。推測や一般常識に基づく因果・相関の記述をしないこと

**禁止表現の例:**
- 「〇〇が高い県は△△も高い傾向がある」→ research.md で |r| >= 0.3 の裏付けが必要
- 「〇〇が△△の原因と考えられる」→ 相関から因果を導く記述は常に禁止

### ストーリー・具体例の補強

内容が抽象的になる場合、以下を追加生成する:
- コンセプトを一瞬で理解させる **たとえ話**（3つ生成して最も強い1つを採用）
- 共感できる人物を使った **ケーススタディ**
- 核心コンセプトを裏付ける **データポイント**（数字ではなくストーリーとして提示）

## URL 生成ルール

note 記事の stats47.jp リンクには UTM パラメータを付けない。素の URL をそのまま記載する。

## note 執筆制約（全記事共通）

- **文章はできるだけ短く改行すること。長い段落を避け、読みやすさを重視すること。**
- **太字記法 `**` を使用する際は、括弧（全角・半角問わず）を太字の外に配置すること。**
- **見出し（`##`、`###`など）には絵文字を使用しないこと。**
- **Markdown テーブル（`|` 記法）は使用しないこと**（note 非対応）。
- **全47都道府県データは記事に含めない。** 具体例は 3〜5 個まで。詳細は stats47.jp に誘導する。
- **1 記事あたり stats47.jp へのリンクを 5 本以上設置すること。**
- **数値データの出典は記事に記載しない。** stats47.jp リンクに「詳しくはこちら」と案内する。
- stats47.jp リンクに UTM パラメータは不要。素の URL（例: `https://stats47.jp/ranking/fiscal-strength-index-prefecture`）で記載する。
- **stats47.jp リンクは `### 見出し` + 直下に素 URL の形式で記載すること。** note.com がブログカードとして埋め込み表示する。Markdown リンク記法 `[テキスト](URL)` は使わない。
  ```markdown
  ### 年間日照時間ランキング

  https://stats47.jp/ranking/annual-sunshine-duration
  ```
- **データリテラシーの知識**（人口当たり補正の重要性、相関と因果の違い等）は独立セクションにせず、具体的なデータの文脈で自然に織り込む。

## SEO カニバリ防止ルール

- title に「都道府県 〇〇 ランキング」形式を使わない
- stats47.jp = ファクト検索 + ランキング、note = ストーリー + 考察 + 調査ガイド
- 全47都道府県データを掲載しない（具体例は 3〜5 個まで）
- note 記事のターゲットキーワード:
  - B シリーズ: 「なぜ〜」「意外な〜」「〜の正体」等の読み物型
  - C シリーズ: 「〇〇調査 わかること」
  - D シリーズ: 社会課題系キーワード

## チャートデータ JSON の保存（共通ルール）

**記事で使用するデータは `chart-data.json` として必ず JSON で保存すること。** これにより `/generate-note-charts` でのチャート生成時に DB 再取得が不要になる。

- **B/C/D シリーズ**: `/investigate-note-data` が `_data/chart-data.json` を生成済み。追加データがあれば追記する
- **A シリーズ（深掘り型）**: このスキルの執筆時に `chart-data.json` を記事ディレクトリ直下に保存する
- **A シリーズ（量産型 `/post-note-ranking`）**: `/post-note-ranking` が `chart-data.json` を生成する

## ハッシュタグ生成

執筆完了時に `hashtags.txt` を記事ディレクトリ直下に作成する。

- note は**最大 100 個**のハッシュタグを登録できる
- `#キーワード` 形式で、スペース区切り、1 行にまとめる
- 記事のテーマ・登場する都道府県・関連する統計指標・一般的な発見系ワードを網羅する
- 参考: `docs/31_note記事原稿/b2-sunshine-duration/hashtags.txt`

## 出力

```
docs/31_note記事原稿/<slug>/note.md
docs/31_note記事原稿/<slug>/hashtags.txt
```

全セクション書き終えたら `/edit-note-draft` へ進む（**必ず別チャットで**）。
