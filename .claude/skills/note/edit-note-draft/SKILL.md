note 記事（B/C/D シリーズ）の原稿をチェックし、修正理由付きで改善案を出力する。

> **A シリーズ（ランキング記事・量産型）はこの4ステップワークフローの対象外です。** A シリーズは自動生成スキルで処理するため、validate → design → write → edit の手順は不要です。

## 用途

- `/write-note-section` で書き上げた原稿を公開品質に磨き上げたいとき
- stats47.jp への誘導設計を客観的に検証したいとき

## フロー

```
/validate-note-idea → /investigate-note-data → /design-note-structure → /write-note-section → ★/edit-note-draft
```

**鉄則: 必ず新しいチャットで実行すること。** 執筆したチャットと同じ場所で編集すると、AI が自分の出力に引っ張られて客観的な修正ができなくなる。

## 引数

- **原稿ファイル**: `docs/31_note記事原稿/<slug>/note.md`
- **シリーズ**: B / C / D
- **トーン**: 会話的 / 権威的 / データドリブン
- **ターゲット読者レベル**: 初心者 / 中級者 / 上級者
- **核心メッセージ**: この記事全体を1文で要約

## 手順

原稿を読み込み、以下の軸で編集する。1セクションずつ通すのが推奨。

### 1. 明瞭性チェック

- 2回読まないと理解できない文をすべて特定
- 専門用語をわかりやすい言葉に置き換え
- 各段落が1つの明確な目的に奉仕しているか
- すべての段落の冒頭文と末尾文を引き上げ

### 2. フローとペーシング

- 読者の勢いが止まるポイントを特定
- 長すぎる段落を分割
- 論理が飛躍しているところに接続文を追加
- 受動態を直接的な能動態に置き換え

### 3. データ主張の監査（最重要）

**記事内の全てのデータ関連の主張を抽出し、`_data/research.md` と照合する。**

1. 記事から以下のパターンを全て抽出する:
   - 「〇〇が高い県は△△も高い」系の相関主張
   - 「〇〇が△△の原因/要因/背景」系の因果主張
   - 「〇〇と△△には関連がある」系の関連性主張
   - 具体的な数値データ（ランキング順位、値）

2. 各主張を `_data/research.md` と照合する:
   - 相関主張: `_data/research.md` の相関分析で |r| >= 0.3 の裏付けがあるか確認
   - **|r| < 0.3 の主張は削除または「データ上は明確な関連が見られない」に修正**
   - 因果主張: 相関から因果を導いていないか確認（**全て修正対象**）
   - 数値データ: `_data/research.md` の値と一致するか確認

3. `_data/research.md` に記載のない指標ペアの関係性主張は**全て削除**

**B-3（出生率）の教訓**: 「婚姻率が高い県は出生率も高い」と書いたが、実際の相関は r=-0.09（ほぼ無相関）だった。このような誤りを防ぐために、このステップは省略不可。

### 4. データ提示の品質

- 統計データが「へえ」と思える形で提示されているか
- 数字の羅列になっていないか（ストーリー化されているか）
- 読者が「自分の県はどうだろう？」と思えるか
- まだ具体例が必要な抽象的コンセプトを特定

### 5. 誘導設計チェック（シリーズ別）

#### B シリーズ（ランキング読み物）
- フックが3文以内で読者を引き込めているか
- 考察が冗長でないか（1,000字以内が目安）
- 「もっと詳しく知りたい方へ」のリンク集が充実しているか

#### C シリーズ（統計調査ガイド）
- 「e-Stat での探し方」→「stats47 なら整形済み」の誘導が自然か
- 具体的発見のセクションで stats47.jp リンクが活きているか
- 元行政職員の視点が独自の価値を出しているか

#### D シリーズ（深掘り分析）
- 複数指標の横断分析が説得力を持っているか
- 歴史・政策の文脈が十分に書かれているか
- 「これからどうなるか」のトレンド提示が根拠に基づいているか

#### 全シリーズ共通
- stats47.jp への誘導が自然に組み込まれているか（最低5本）
- 「stats47 で見てみよう」と思わせる動線があるか
- SEO カニバリを起こしていないか（ランキング記事化していないか）
- note 戦略の骨子に沿った構成になっているか

### 6. 読者体験チェック

- オープニングの約束を果たしているか
- 読後に具体的な行動が取れるか
- stats47.jp への誘導が自然に組み込まれているか
- 締めが読者に行動を促す内容になっているか

## 出力形式

すべての重要な修正に **変更理由** を付けて出力すること。

```
#### 修正 1: [箇所]
- 原文: 「...」
- 修正案: 「...」
- 理由: ...
```

修正を反映した原稿を `note.md` に上書き保存する。

## 品質チェックリスト（全記事共通）

- [ ] タイトルに「都道府県 〇〇 ランキング」形式を使っていない（SEO カニバリ防止）
- [ ] ランキング記事・データストーリーになっていない
- [ ] stats47.jp へのリンクが 5 本以上ある
- [ ] 全 stats47.jp リンクに UTM パラメータが付与されていない（素の URL）
- [ ] リンク先のページが実在する（存在確認済み）
- [ ] 記事末尾に stats47.jp リンク集がある
- [ ] 出典はサイト誘導に委ね、記事内に記載していない
- [ ] 文体は「です・ます」で統一
- [ ] 文字数が目安以上（B: 1,500字, C: 2,000字, D: 3,000字）
- [ ] 見出しに絵文字を使用していない
- [ ] Markdown テーブルを使用していない（note 非対応）
- [ ] 太字記法の括弧が太字の外にある
- [ ] 文章が短く改行されている
- [ ] note 戦略の骨子に沿った構成になっている
- [ ] 記事内のデータと `_data/research.md` の分析結果が整合している
- [ ] 相関関係を因果関係として記述していない
- [ ] 一次情報（元県庁職員の実体験・検証・観察）が含まれている
- [ ] 事実と意見が明確に分離されている
- [ ] 煽りタイトル・攻撃的表現がない
- [ ] 初心者でも理解できる平易な文章になっている
- [ ] チャートが記事内容と対応している
- [ ] `hashtags.txt` が存在し、ハッシュタグが 100 個以内である

## 公開準備（編集完了後）

### 1. DB にメタデータ登録

ローカル D1（SQLite）の `note_articles` テーブルに直接 INSERT する:

```sql
INSERT INTO note_articles (
  id, ranking_key, title, summary, file_path,
  status, note_price, created_at, updated_at
) VALUES (
  '<uuid>', '<articleId>', '<タイトル>', '<概要120文字>',
  '31_note記事原稿/<slug>',
  'draft', 0, datetime('now'), datetime('now')
);
```

`ranking_key` フィールドには `articleId`（例: `B-1`）を格納する。
`note_price` は 0（全記事無料）。

### 2. note 戦略の進捗更新

`docs/30_note記事企画/note戦略.md` の進捗サマリーテーブルで、該当記事の「執筆」列を更新する。

### 3. R2 にアーカイブ

公開済み記事を `.local/r2/note/` にコピーし、`docs/` 側を削除する:

```bash
SLUG="<slug>"
# note.md + images をR2にコピー
mkdir -p ".local/r2/note/$SLUG/images"
cp "docs/31_note記事原稿/$SLUG/note.md" ".local/r2/note/$SLUG/"
cp docs/31_note記事原稿/$SLUG/images/*.png ".local/r2/note/$SLUG/images/" 2>/dev/null || true

# docs側を削除（_data含む）
rm -rf "docs/31_note記事原稿/$SLUG"
```

### 4. note.com への投稿手順

1. `docs/31_note記事原稿/<slug>/note.md`（または `.local/r2/note/<slug>/note.md`）の内容を note エディタにペースト
2. `images/` の画像をアップロード（あれば）
3. 無料で公開
4. 公開後、ローカル D1 で status と noteUrl を更新:
   ```sql
   UPDATE note_articles
   SET status = 'published', note_url = '<note.com URL>', updated_at = datetime('now')
   WHERE ranking_key = '<articleId>';
   ```
5. `docs/30_note記事企画/note戦略.md` の進捗サマリーで「公開」列を更新
