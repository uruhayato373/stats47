プロジェクトの **Pre-Mortem（事前検死）** を実施する。
「1年後、このプロジェクトは完全に失敗した。何が起きたか？」を、現在の実態データに基づいて具体的に描写し、各シナリオに対する **予防策・耐性策** を設計する。

**失敗シナリオと対策は 1 Issue の中で 2 セクションに分けて管理する**（旧バージョンは pre-mortem / countermeasures の 2 ファイルだったが、2026-04 以降は 1 Issue に統合）。

## 目的

- 楽観バイアスを打ち破り、見て見ぬふりしているリスクを言語化する
- 「いつかやる」と先送りしている致命的タスクを炙り出す
- 定量データで裏付けた失敗シナリオにより、優先順位の再設定を促す
- 各リスクに対して「予防」と「耐性」の2層で備える

## 手順

### Phase 1: 現在の実態データを収集

以下を**すべて実際のソースから直接取得**する。計画書やロードマップの記載は信用しない。

#### DB（ローカル D1 SQLite）

```bash
DB=".local/d1/v3/d1/miniflare-D1DatabaseObject/baffe56c6b0173e34c63a5333065bcdb6642a01b4c2cfecd70ad3607b00c9972.sqlite"
sqlite3 "$DB" "
  SELECT 'ranking_items' as tbl, COUNT(*) as cnt FROM ranking_items WHERE area_type='prefecture'
  UNION ALL SELECT 'ranking_data', COUNT(*) FROM ranking_data
  UNION ALL SELECT 'ranking_ai_content', COUNT(*) FROM ranking_ai_content
  UNION ALL SELECT 'articles', COUNT(*) FROM articles
  UNION ALL SELECT 'sns_posts', COUNT(*) FROM sns_posts
  UNION ALL SELECT 'correlation_analysis', COUNT(*) FROM correlation_analysis
"
```

#### コンテンツ資産

- `.local/r2/blog/` の記事数と公開状態（publishedAt の有無）
- `.local/r2/sns/` の未投稿コンテンツ数
- `apps/remotion/` のテンプレート種類数

#### 外部指標（取得可能な場合）

- Google Search Console: 直近30日のインプレッション・クリック数（`/fetch-gsc-data` が使える場合）
- GA4: 直近30日の PV・ユーザー数（`/fetch-ga4-data` が使える場合）
- SNS: フォロワー数・投稿数（各 `/fetch-*-data` スキル）

外部 API にアクセスできない場合はスキップし、その旨を明記する。

#### Git 活動

```bash
git log --oneline --since="30 days ago" | wc -l   # 直近30日のコミット数
git log --oneline --since="7 days ago" | wc -l     # 直近7日のコミット数
```

#### インフラ

- Cloudflare D1 の行数（ranking_data が free tier 上限に近いか）
- R2 バケットのサイズ（`/r2-du` の直近結果があれば参照）
- `.git/` のサイズ

#### 既存の対策状況

- `apps/web/src/middleware.ts` のリダイレクト設定
- `apps/web/src/app/sitemap.ts` の動的生成状況
- AdSense / アフィリエイトの設定状況
- 実際に稼働している仕組み（自動投稿パイプライン等）の有無

### Phase 2: 失敗シナリオを生成（Issue 本文 前半セクション）

収集したデータをもとに、**1年後に完全に頓挫した**前提で失敗シナリオを 7〜10 個書く。これは Issue 本文の「## 想定失敗シナリオ」セクションに入れる。

#### シナリオの書き方ルール

1. **見出し**: 失敗の本質を1行で（例: 「配信恐怖症が最後まで治らなかった」）
2. **冒頭**: 現在のデータから、なぜこの失敗が起きうるかを1〜2文で
3. **「具体的に何が起きたか」**: 箇条書き3〜5項目。**過去形**で書く（1年後の視点）
4. **数字を入れる**: 「約」ではなく具体的な数字（現在のデータから外挿）
5. **発生確率・影響度**: 高 / 中 / 低 のいずれかを明記
6. **予兆サイン**: 何を見たら兆候と判断するか

#### 必須カテゴリ（該当する場合）

- **配信・公開の停滞**: 作ったが届けていないコンテンツ
- **SEO・検索流入の崩壊**: インデックス状況、CTR低下リスク
- **収益ゼロの長期化**: AdSense、アフィリエイト、有料記事の状況
- **技術的楽しさへの逃避**: 手段の目的化、過剰な自動化
- **インフラ・技術的負債**: D1上限、SSD容量、依存パッケージ
- **一人プロジェクトの限界**: 稼働時間、モチベーション管理
- **外部環境の変化**: API仕様変更、AI Overview、競合動向
- **データ品質**: AI生成コンテンツの誤り、データ鮮度

### Phase 3: 対策を生成（Issue 本文 後半セクション）

Phase 2 の各シナリオに対して、**予防策（Prevention）** と **耐性策（Resilience）** の2層で対策を設計する。これは Issue 本文の「## 対策」セクションに入れる。

#### 対策の書き方ルール

1. **対応**: シナリオ番号と連動させる（`シナリオ 1 → 対策 1`）
2. **予防策**: そもそも起きないようにする仕組み。具体的な手順・最小実装を含む。チェックボックスで列挙
3. **耐性策**: 起きても致命傷にならない構造。「なぜ大丈夫か」を明示。チェックボックスで列挙
4. **既に対策済みの場合**: 調査結果を明記し、追加で必要な監視・改善を提案
5. **具体的な判断基準を含める**: 「X ヶ月後に Y 未満なら Z する」の形式

### Phase 3.5: ロードマップの Sprint 進捗評価

`docs/02_実装計画/01_実装ロードマップ.md` を読み込み、四半期の区切りとして以下を評価・更新する:

1. **「現在のステータス」テーブル**: Phase 1 で取得した実測値で更新（`/weekly-review` と同じ手順）
2. **Sprint 完了判定**: 現在の Sprint の「完了条件」を実績データと照合。達成/未達を明記
3. **次 Sprint への示唆**: 対策書の提言を踏まえ、次 Sprint で優先すべき事項を提案（ただし Sprint 定義そのものは書き換えない。提案として Issue 本文の末尾に「次 Sprint 提案」サブセクションで記載する）

**変更してよい範囲**: 「現在のステータス」テーブルの数値・状態のみ。「基本方針」「Sprint 定義」「凍結タスク」は変更しない。

### Phase 4: Issue 作成

以下を 1 Issue にまとめて作成する。テンプレは `.github/ISSUE_TEMPLATE/pre-mortem.md` に準拠。

```bash
# 本文を /tmp/pre-mortem-body.md に書き出し後:
gh issue create \
  --title "[Pre-Mortem] YYYY-MM" \
  --label "pre-mortem" \
  --body-file /tmp/pre-mortem-body.md
```

### Phase 5: 報告

生成した Issue の番号・URL と、最も優先度の高い対策3つを報告する。

## トーンと姿勢

### 失敗シナリオ（前半セクション）
- **容赦なく、しかし建設的に**: 問題を美化しない。「まだ大丈夫」は禁句
- **過去形で書く**: 「〜だろう」ではなく「〜だった」「〜してしまった」
- **数字で語る**: 「たくさん」ではなく「146件」「0件」「82%」
- **自分事として書く**: 第三者の分析ではなく、失敗した本人の回顧として

### 対策（後半セクション）
- **現在形・命令形で書く**: 「〜すべきだった」ではなく「〜する」
- **最小実装を示す**: 理想論ではなく、今週中にできる最小ステップ
- **判断基準を明確に**: 「いつ」「何を見て」「どう判断するか」を具体的に
- **既に対策済みのものは正当に評価する**: 無理にリスクを誇張しない

### 共通
- **既存の Pre-Mortem Issue を参照**: `gh issue list --label pre-mortem --state all` の過去 Issue があればトーンを合わせる

## Issue ラベル

- `pre-mortem` — 本スキルが作成する Issue のラベル
- 各 `critical-review` / `weekly-review` / 改善施策 Issue からクロスリファレンスされる

## 参照

- `gh issue list --label pre-mortem --state all` — 過去の Pre-Mortem Issue（トーンや構造の参考）
- `gh issue view <番号>` — 過去の詳細を参照
- `.github/ISSUE_TEMPLATE/pre-mortem.md` — Issue テンプレ（失敗シナリオ + 対策の統合型）
- `docs/02_実装計画/01_実装ロードマップ.md` — Sprint 進捗評価・実測値更新の対象
- `.claude/skills/management/critical-review/SKILL.md` — 批判的レビュー（計画書向け、対になるスキル）
