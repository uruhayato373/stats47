ブログ記事（article.md）の公開前チェックを実行し、問題箇所の報告と修正を行う。

## 引数

```
$ARGUMENTS — チェック対象の記事ファイルパスまたは slug
             パス例: .local/r2/blog/commute-time-ranking/article.md
             slug例: commute-time-ranking
             URL例:  http://localhost:3000/blog/commute-time-ranking
```

### 引数の解決ルール

1. URL が渡された場合: `/blog/{slug}` から slug を抽出
2. slug のみの場合: 以下の優先順で探索
   - `docs/21_ブログ記事原稿/{slug}/article.md`（下書き）
   - `.local/r2/blog/{slug}/article.md`（公開済み）
3. フルパスの場合: そのまま使用

## 概要

記事作成後の公開前チェックを4領域で実施する。問題を検出したら修正案を提示し、承認を得てから article.md に直接適用する。

## チェック項目

### 1. フロントマター

- `publishedAt` が `(未定)` のままでないか
- `description` が 120〜160字であるか
- `tags` が1件以上設定されているか
- `category` が設定されているか
- `subtitle` が設定されているか（OGP 画像のサブテキストに使用）
- `title` に `──` や `｜` が含まれていないか（title / subtitle に分離すべき）
- `rankingKeys` が残っていないか（廃止済み・削除対象）
- `seoKeywords` が残っていないか（廃止済み・削除対象）
- **`title` が「○○ランキング」形式でないか**（ランキングページとの検索カニバリ回避）
  - 検出パターン: `title` が「ランキング」を含む
  - ランキングページ `/ranking/` がデータ参照意図を押さえるため、ブログ記事は洞察・発見・対比をタイトルにすべき
  - 修正方針: 既存の `subtitle` を参考に、記事の切り口を反映したタイトルに変更を提案する
  - 例: `製造品出荷額ランキング` → `愛知58兆円一強の製造業地図`
- **`title` が「○○格差」「○○の地域差」パターンに偏っていないか**
  - 検出パターン: `title` が「格差」「地域差」で終わる
  - 多くの記事が同じパターンだと一覧で単調に見える。意外性・疑問・逆説・自分ごと化など多様なパターンを使う
  - 例（NG）: `失業率2倍差の労働市場格差` → 例（OK）: `失業率が高い県、低い県の構造`
- **slug が `-ranking` で終わっていないか**（`/ranking/` の URL パスと混同される）
  - 検出: ディレクトリ名（slug）の末尾が `-ranking`
  - ⚠️ として報告する（slug 変更はリダイレクト設計が必要なため自動修正しない）
- **`title` に記事の主要指標キーワードが含まれているか**（SEO 検索流入チェック）
  - 記事の `tags` や本文の見出しから主要な統計指標名（例: 「貯蓄率」「日照時間」「出生率」「失業率」）を特定する
  - `title` にその指標名が含まれていない場合、ターゲットクエリ（「○○ 都道府県」「○○ ランキング」等）での検索流入が見込めない
  - 検出: `tags` の第1タグのキーワードが `title` に含まれていない
  - 修正方針: 指標名を自然に組み込んだタイトルを提案する（ランキング形式は避けつつ、検索意図と記事内容が一致するように）
  - 例（NG）: `手取りの何割を貯められるか`（「貯蓄率」がない → 個人向け家計相談記事に見える）
  - 例（OK）: `貯蓄率が高い県、低い県──手取りの何割を貯めているか`（指標名＋地理比較＋自分ごと化）

### 2. 本文・チャート

- フロントマター直後に `# タイトル`（h1）が残っていないか（ページ側で `<h1>` を表示するため二重になる）
- チャートプレースホルダー `<!-- chart:... -->` が残っていないか（SVG画像に置換済みか）
- `![alt](data/file.svg)` の画像ファイルが `data/` ディレクトリに実在するか
- `[チャート: ...]` 形式のプレースホルダーテキストが残っていないか
- 見出し・alt テキスト・本文に「ベスト」「ワースト」が使われていないか
  - Stats47 は統計情報サイトとして中立的なトーンを保つため「上位」「下位」を使う
  - 例外: 記事タイトル・description・URL slug・他記事へのリンクテキストは変更不要
  - 修正例: `ワースト10──死者が多い県` → `死者数が多い上位10県`
- 本文中に未リンクの指標キー記法が残っていないか
  - 対象パターン: `e-Stat: \`ranking-key\`` / `（\`ranking-key\`）` / `使用指標: \`ranking-key\`（日本語名）`
  - 修正: 指標キーを削除し、日本語名を `/ranking/{key}` へのリンクに置き換える
  - 例: `使用指標: \`road-length-per-km2\`（道路密度）` → `使用指標: [道路密度](/ranking/road-length-per-km2)`
  - 例: `**指標名**（\`ranking-key\`）` → `[**指標名**](/ranking/ranking-key)`
- ボールド記法の中に全角カッコ（`「」`・`（）`）が含まれていないか
  - 検出パターン（開き・閉じ両方）:
    - 開き: `**「` / `**（` — ボールド開始直後にカッコがある形
    - 閉じ: `」**` / `）**` — 閉じカッコがボールド終端直前にある形
  - 多くのMarkdownパーサーで太字が正しく認識されないため修正が必要
  - 修正例: `**「規模の不経済」**` → `「**規模の不経済**」`（カッコをボールドの外へ）
  - 修正例: `**右下（税収多・交付税少）**:` → `**右下**（税収多・交付税少）:`
- ボールド終端 `**` の直前が記号（`%`・`円`・数字以外の半角記号など）でスペースなしの場合、パーサーが閉じボールドを認識できないことがある
  - 検出パターン: `%**`（`**` の直前が `%`）
  - 修正: `**` の後にスペースを入れる（`**98%超**とほぼ` → `**98%超** とほぼ`）
- ボールド開始 `**` の直後に特殊記号（`△` `▲` `▽` `▼` `※` 等）がある場合、パーサーが太字の開始を認識できない
  - 検出パターン: `**△` `**▲` `**▽` `**▼` `**※`
  - 修正: 記号を変更するか前後にスペースを入れる（`**△47%**` → `**−47%**`）
  - 減少を示す `△` は `−`（マイナス記号 U+2212）に統一する
- 数値の括弧列挙が冗長でないか
  - 検出パターン: `名詞（数値+任意の単位）` が2つ以上連続する文（括弧内に数値がある要素の連続）
    - 例: `○○県（約510万人泊）、△△県（約396万人泊）`
    - 例: `東京都（80.4%）、大阪府（77.9%）`
    - 例: `北海道（3.4万円）、沖縄県（2.8万円）`
  - **適用範囲**: チャート直下に限らず、本文全体が対象。チャート・ランキング表・マップで視覚的に読み取れる数値を地の文で繰り返すのは冗長
  - **判定基準**: 1文中に `都道府県名（数値）` の括弧付き列挙が **3つ以上** あれば冗長と判定する
    - 例（NG）: `東京（74.5%）・神奈川（71.8%）・埼玉（69.3%）・千葉（67.4%）・群馬（67.0%）が軒並み65%超`
    - 例（OK）: `東京・神奈川・埼玉・千葉・群馬が軒並み65%超`（チャートで数値は確認できる）
    - 例（OK）: `東京が74.5%で最も高く、神奈川・埼玉と首都圏が上位を占める`（1位の数値のみ残す）
  - **修正方針**:
    - 括弧内の数値を削除し、都道府県名のみの列挙にする
    - 1位（または最大・最小）の数値だけ地の文に組み込む
    - 数値の比較が論旨の核心である場合（例: 1位と最下位の差を示す文）は括弧列挙を許容する
  - 金額はカンマ区切りの円表記（`34,497円`）を避け、万円単位（`約3.4万円`）に丸める
- まとめセクションの構成が正しいか（正しい順序: ①導入文 → ②SVG → ③考察文）
  - `## まとめ` 直下に SVG への導入文（1〜2文）を置く
  - その後に `![この記事でわかったこと](data/xxx-summary-findings.svg)` を配置
  - SVG の後にまとめの考察・結論を記述する
  - NG: マークダウンテーブル（`| # | ファインディング |`）とSVGが両方ある / 導入文なしでいきなりSVG
  - **マークダウンテーブルのみでSVGがない場合は ❌ として報告し、SVGを作成して差し替える**
    - 既存の `traffic-deaths-summary-findings.svg` 等を参考に、番号付きカード形式のSVGを `data/{slug}-summary-findings.svg` として新規作成する
    - article.md のマークダウンテーブルを `①導入文 → ②SVG → ③考察文` 構成に置き換える
    - SVG作成は校正の一部として実施する（後回しにしない）
- `data/*.svg` の `<svg>` タグに `width`/`height` 属性が明示されているか（`viewBox` のみだと Obsidian・ブログで横幅いっぱいに表示されない）
  - 判定: SVG 1行目に `width="..."` が含まれているか確認
  - 修正ルール（`viewBox="0 0 W H"` の W・H から算出）:
    - 横長（W > H）: `width="780" height="{round(780*H/W)}"`
    - 縦長（H > W）: `width="{W}" height="{H}"` （元寸法を維持）
    - 正方形（W = H）: `width="700" height="700"`
  - 修正方法: `<svg` の直後に `width="..." height="..."` を挿入（`viewBox` は残す）
- `data/*.svg` のサブタイトル行に出典・データソース名・年度が埋め込まれていないか
  - 検出: タイトル `<text>` の直下にある `font-size="10"〜"13"` かつ `fill="#888"` or `fill="#6b7280"` の `<text>` 要素
  - 出典情報（「国勢調査」「e-Stat」「○○統計」「○○年」等）は SVG に含めず、md 側の `<data-source>` で記載する
  - チャート固有の補足（「全国平均」「男女別」等）のみ SVG サブタイトルに残してよい
  - 修正: 該当行を削除

### 3. データ注記・出典

- 年次ズレ（主要指標と比べて2年以上古い）があるデータを使っている散布図・チャートの直下に `> [!NOTE]` コールアウトが付いているか
  - フッターへの一括記載は不要。文脈に応じた注記を該当箇所に置く
- 各チャート（`![...](data/...)` の画像）の直下に `<data-source>` コンポーネントがあるか
  - 形式: `<data-source url="..." label="..." note="..."></data-source>`
  - `url` と `label` は必須。`note` は任意
  - `year` 属性は **使用しない**（年度情報は不要。既存の `year="..."` は削除する）
  - **`url` 属性の欠落チェック**: `<data-source label="..."` のみで `url=` がない場合は ❌ として報告する
    - 検出パターン: `<data-source` タグに `url=` が含まれていない
    - 修正方法: 記事で使用しているランキングキーから `source_config` の `statsDataId` を DB で取得し、`url="https://www.e-stat.go.jp/dbview?sid={statsDataId}"` を設定する
    - `label` は `e-Stat {source.name}` の形式に統一する（例: `e-Stat 社会・人口統計体系`）
    - `label` に年度情報（`2021年` 等）を含めない（年度は不要）
    - `label` に `総務省「○○調査」` のような調査名のみの形式が残っていたら、DB の `source_config.source.name` に基づく `e-Stat ○○` 形式に修正する
  - 旧形式チェック: 以下のパターンが残っていたら `<data-source>` に変換する
    - `> 出典：[...](URL)` / `> 出典: [...]`
    - `**データ出典:**` / `**出典:**`
    - `出典：e-Stat ...` 等のプレーンテキスト
- 記事末尾に冗長な全体出典 `<data-source>` が残っていないか
  - 各チャート直下に個別の `<data-source>` が付いている場合、末尾の汎用出典は**不要**（削除する）
  - 個別出典がないチャートがある場合のみ、末尾に全体出典を残す

### 4. サイト内リンク

- ランキングページ・相関ページへのサイト内リンクが `<source-link>` コンポーネントを使っているか
  - 形式: `<source-link href="/ranking/xxx">リンクテキスト</source-link>`
  - 旧形式チェック: 以下のパターンが残っていたら `<source-link>` に変換する
    - `[○○ランキングを Stats47 で見る →](/ranking/xxx)`
    - `[○○ →](/ranking/xxx)` / `[... →](/correlation?...)`
    - 末尾の `- [リンクテキスト](/ranking/xxx)` リスト形式
- `### 関連記事` の見出しレベルが `###`（h3）になっているか
  - 検出: `## 関連記事`（h2）や `#### 関連記事`（h4）になっている
  - 修正: `### 関連記事` に統一する
- 本文・末尾に空リンク `()` が残っていないか
- **ランキングチャートに対応する `<source-link>` があるか**（ランキング→リンク対応チェック）
  - 検出: 記事内のランキング棒グラフ（`上位10・下位10` 等の SVG）ごとに、同セクション内に対応する `<source-link href="/ranking/{key}">` が存在するか確認する
  - 散布図は `<source-link href="/correlation?...">` で既にリンクされていることが多いため、棒グラフ・タイルマップ等の単一指標チャートが主な対象
  - 修正: チャートを分析しているセクションの末尾に `<source-link>` を追加する
- **`<source-link>` が末尾に集約されていないか**（インライン分散チェック）
  - 検出: 記事末尾（`## まとめ` より後）に `<source-link>` が2件以上連続している
  - 各 `<source-link>` は、そのランキングデータを言及・分析しているセクションの末尾に配置すべき
  - 修正方針: リンク先の `/ranking/{key}` が本文のどのセクションで使われているかを特定し、該当セクション末尾に移動する
  - `### 関連記事` セクションには `/blog/...` リンクのみを残す
- **リンク先の DB 存在確認**: `<source-link>` や `### 関連記事` のリンク先がローカル DB に存在するか確認する
  - 検出方法: 記事内の `/ranking/{key}` を抽出し、`ranking_items` テーブルに `ranking_key` が存在するか照会する
  - `/blog/{slug}` リンクは `.local/r2/blog/{slug}/article.md` の存在を確認する
  - `/correlation?x={key1}&y={key2}` リンクは両方の `ranking_key` が `ranking_items` に存在するか確認する
  - 存在しないキーが見つかった場合の **類似キー検索**（必須）:
    1. `<source-link>` のリンクテキスト（例: `ゴルフ行動者率ランキング`）からキーワードを抽出する
    2. ローカル D1 で以下のクエリを実行し、類似キーの候補を検索する:
       ```sql
       -- リンクテキストのキーワードで ranking_name を部分一致検索
       SELECT ranking_key, ranking_name FROM ranking_items
       WHERE ranking_name LIKE '%ゴルフ%' OR ranking_name LIKE '%行動者率%'
       LIMIT 10;
       ```
    3. 存在しないキーの一部をキーワードとしても検索する:
       ```sql
       -- キーの一部で部分一致検索（例: ssdse-d-mc09 → 'golf', 'sports' 等は不要だが、意味的に近いキーを探す）
       SELECT ranking_key, ranking_name FROM ranking_items
       WHERE ranking_key LIKE '%golf%' OR ranking_key LIKE '%sports%participation%'
       LIMIT 10;
       ```
    4. 候補が見つかった場合: 正しいキーへの修正を提案する（`href` の書き換え）
    5. 候補が見つからなかった場合: `/register-ranking` でデータ登録を提案する
  - 登録・修正するかどうかはユーザーの判断に委ねる（自動修正しない）
  - 出力例（類似キーが見つかった場合）:
    ```
    ❌ リンク先未登録: /ranking/ssdse-d-mc09
       → ranking_items に存在しません
       → 🔍 類似キー検索: リンクテキスト「ゴルフ行動者率ランキング」から検索
       → 候補: sports-participation-rate-golf（ゴルフの年間行動者率）
       → 修正案: href="/ranking/ssdse-d-mc09" → href="/ranking/sports-participation-rate-golf"
    ```
  - 出力例（候補なしの場合）:
    ```
    ❌ リンク先未登録: /ranking/total-overnight-guests-ryokan
       → ranking_items に存在しません
       → 🔍 類似キー検索: 該当なし
       → `/register-ranking` で登録できます
    ```

## 手順

1. 引数で指定された記事ファイルを読み込む
2. slug（ディレクトリ名）を特定し、`data/` ディレクトリの内容を確認する
3. 各チェック項目を実行する
4. 結果を下記フォーマットで出力する
5. ❌ 項目がある場合、修正内容を提示してユーザーの承認を得てから article.md を更新する
6. 修正適用後、ローカル D1 の `articles` テーブルに校正日時を記録する:
   ```sql
   UPDATE articles SET proofread_at = datetime('now') WHERE slug = '{slug}';
   ```
   - better-sqlite3 で直接実行する（`node -e "..."` ワンライナー）
   - DB パス: `.local/d1/v3/d1/miniflare-D1DatabaseObject/baffe56c6b0173e34c63a5333065bcdb6642a01b4c2cfecd70ad3607b00c9972.sqlite`

## 未校正記事の一覧取得

校正対象の記事を特定するクエリ:

```sql
-- 未校正の公開記事（古い順）
SELECT slug, title, published_at FROM articles
WHERE published = 1 AND proofread_at IS NULL
ORDER BY published_at ASC;

-- 校正後に更新された記事（再校正が必要）
SELECT slug, title, proofread_at, updated_at FROM articles
WHERE published = 1 AND proofread_at < updated_at
ORDER BY updated_at DESC;
```

## サイト内リンクの追加方針

記事末尾には `### 関連記事` セクションを配置し、**他のブログ記事（`/blog/...`）へのリンクのみ** を掲載する。

```markdown
### 関連記事

- [記事タイトル](/blog/slug)
```

- リンク数は 2〜4 件を目安とする
- リンク先は `/blog/...` のみ。ランキングページ（`/ranking/...`）は本文中に `<source-link>` でインライン配置する
- 記事の `tags`・テーマから関連性の高い既存ブログ記事を選ぶ。該当記事がない場合はセクション自体を省略してよい
- 旧形式チェック: 末尾に `<source-link>` や `/ranking/...` リンクが並んでいたら、ランキングリンクは本文中に移動し、関連ブログ記事のリンクに差し替える
