# Stats47 コーディング規約 Skill

このskillは、stats47プロジェクトのコーディング規約とアーキテクチャ原則に従って、コードの生成・レビュー・リファクタリングを支援します。

## 使用タイミング

- 新規コンポーネントの作成時
- 既存コードのリファクタリング時
- コードレビュー時
- アーキテクチャ原則の確認が必要な時

## 基本原則

このプロジェクトは以下のドキュメントに基づいて設計されています。コード作成時は**必ず以下のドキュメントを読んで理解してから作業を開始してください**：

## 作業フロー

### 新規コンポーネント作成時

1. **事前準備**
   ```
   上記3つの必読ドキュメントをReadツールで読み込む
   ```

2. **配置先の決定**
   - Atom/Molecule: `src/components/atoms/` または `src/components/molecules/`（汎用的に保つ）
   - Organism: `src/components/organisms/[domain]/`（ドメイン別に配置）
   - ビジネスロジック: `src/lib/[domain]/`

3. **パターンの適用**
   - データ取得が必要 → Fetcher/Formatter/Display パターン
   - インタラクション必要 → Client Component (`'use client'`)
   - 静的表示のみ → Server Component

4. **型定義**
   - 型はドメイン直下の `types/` ディレクトリに配置
   - 例: `src/lib/ranking/types/item.ts`

### リファクタリング時

1. **現状分析**
   - 既存コードがどの原則に違反しているか確認
   - ドキュメントと照らし合わせる

2. **改善方針の決定**
   - 単一責任の原則に従っているか
   - 適切なレイヤーに配置されているか
   - ドメイン境界が明確か

3. **段階的リファクタリング**
   - 一度に大きく変更せず、小さな変更を積み重ねる

## コーディング規約チェックリスト

### コンポーネント設計

- [ ] 単一責任の原則を守っているか
- [ ] Organismsはドメイン別ディレクトリに配置されているか
- [ ] Atoms/Moleculesにビジネスロジックが含まれていないか
- [ ] Server/Client Componentが適切に分離されているか
- [ ] PropTypesが明確に定義されているか

### アーキテクチャ

- [ ] Fetcher/Formatter/Displayパターンに従っているか
- [ ] ビジネスロジックがlib/[domain]/に配置されているか
- [ ] 型定義が適切な場所に配置されているか
- [ ] ドメイン間の依存関係が適切か

### スタイル

- [ ] Tailwind CSSのユーティリティクラスを使用しているか
- [ ] ダークモード対応（`dark:`プレフィックス）がされているか
- [ ] レスポンシブ対応がされているか

### アクセシビリティ

- [ ] 適切なARIA属性が設定されているか
- [ ] キーボードナビゲーションが可能か
- [ ] セマンティックHTMLを使用しているか

## ドメイン分類

### コアドメイン
- `analytics/`: ランキング、時系列分析、相関分析、比較分析
- `visualization/`: チャート、地図、ダッシュボード

### 支援ドメイン
- `data-integration/`: e-Stat API連携、キャッシュ管理
- `area/`: 地域管理
- `category/`: カテゴリ管理

### 汎用ドメイン
- `auth/`: 認証・認可
- `layout/`: 共通レイアウト

## 注意事項

- **絶対にドキュメントを読まずにコードを書かない**

- 新しいパターンを導入する場合は、理由を明確に説明する
- 既存のパターンを壊さないよう注意する
