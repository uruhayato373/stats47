コード変更に対応するテスト（ユニット / E2E）を確認・作成・更新する。

## 概要

コードを追加・修正した際に、既存テストの影響確認と不足テストの作成を行う。
`apps/web/tests/README.md` のテスト追加指針に従い、適切な層のテストを選択する。

## 手順

1. **変更範囲の把握**
   - `git diff --name-only` で変更ファイルを確認
   - 変更ファイルに対応する既存テストを検索:
     - ユニットテスト: `src/**/__tests__/` または `*.test.ts(x)`
     - E2E テスト: `tests/e2e/`

2. **既存テストの影響確認**
   - 変更がインターフェース（props, 引数, 戻り値）を変えた場合、関連テストの更新が必要
   - 削除した関数・コンポーネントのテストがあれば削除 or 更新

3. **不足テストの特定と作成**（`apps/web/tests/README.md` の指針に従う）

   | 変更対象 | 必要なテスト |
   |---------|-------------|
   | 純粋関数・ユーティリティ | ユニットテスト（Vitest） |
   | React フック・コンポーネントロジック | ユニットテスト（Vitest） |
   | ページコンポーネント（page.tsx） | E2E テスト（Playwright） |
   | ページ遷移・ナビゲーション | E2E テスト（Playwright） |
   | SEO（メタデータ・構造化データ） | E2E テスト（Playwright） |
   | ミドルウェア・リダイレクト | E2E テスト（Playwright） |
   | Server Action | ユニットテスト（Vitest） |

4. **テストの実装**
   - ユニットテスト: 変更ファイルの `__tests__/` ディレクトリに配置
   - E2E テスト: `tests/e2e/{機能名}/` ディレクトリに配置
   - 既存テストのパターン（import, describe 構造, アサーション）に合わせる

5. **テスト実行**
   - ユニットテスト: `npm run test:run -w apps/web`
   - E2E テスト: `npm run test:e2e -w apps/web`（開発サーバー起動が必要）
   - 型チェック: `NODE_OPTIONS="--max-old-space-size=4096" npx tsc --noEmit -p apps/web/tsconfig.json`

6. **失敗時の原因切り分け**
   - テストが失敗した場合、**まずプロダクションコード側のバグを疑う**
   - 以下の順で原因を確認し、ユーザーに報告してから修正に着手する:
     1. プロダクションコードのバグ（ロジック誤り、型不整合、未処理エッジケース）
     2. プロダクションコードのインターフェース変更にテストが追従できていない
     3. テスト自体の誤り（アサーション間違い、セットアップ不備）
   - **テストを安易に書き換えて通すのではなく、テストが検出した問題を正しく修正する**

7. **結果報告**
   - 作成・更新したテストファイルの一覧
   - テスト実行結果（パス数 / 失敗があれば原因と修正内容）
