長時間セッションでのコンテキスト管理ガイド。タスク境界での戦略的コンパクションを推奨し、自動コンパクションによる文脈喪失を防ぐ。

## 用途

- 200K+ トークンの長時間セッション
- 複数フェーズに跨がるプロジェクト（週次計画、テーマ設計、ブログ記事執筆）
- 無関係なタスクへの切り替え時
- 「コンテキストが長くなってきた」と感じた時

## コンパクション推奨タイミング

### コンパクションすべき時

- リサーチフェーズ完了後（調査結果はプランファイルに保存済み）
- デバッグセッション完了後（修正はコミット済み）
- 失敗したアプローチを放棄する時
- タスクの切り替え時（ブログ執筆 → コード修正など）

### コンパクションしてはいけない時

- 実装の途中（変数名・ファイルパスの文脈が必要）
- テスト実行中（エラーメッセージの文脈が必要）
- レビュー中（指摘事項の文脈が必要）

## コンパクション前チェックリスト

コンパクション実行前に以下を確認:

1. **タスクリスト更新済み?** — 未完了タスクの状態が最新か
2. **重要な決定事項をメモリに保存?** — 非自明な判断は memory ファイルに記録
3. **コード変更コミット済み?** — 未コミットの変更は git diff で確認不可になる
4. **プランファイル更新済み?** — 次のステップが明記されているか

## コンパクション後に生き残る情報

| 生き残る | 失われる |
|---|---|
| CLAUDE.md の全内容 | 会話の中間推論 |
| memory ファイル（MEMORY.md 経由） | ファイル内容のキャッシュ |
| タスクリスト | grep/glob の検索結果 |
| Git の状態（status, log） | 過去のツール実行結果 |
| プランファイル | エラーメッセージの詳細 |

## stats47 固有の推奨ポイント

| タスク種別 | コンパクション推奨タイミング |
|---|---|
| 週次計画（`/weekly-plan`） | 5 並列サブエージェント完了後、計画出力前 |
| テーマ設計（`/design-theme-charts`） | 監査完了後、チャート設計開始前 |
| ブログ記事執筆 | トレンド調査完了後、記事構成開始前 |
| ランキング登録（`/register-ranking`） | e-Stat メタデータ調査完了後 |
| パフォーマンス改善 | Lighthouse 計測完了後、実装開始前 |

## コンパクションの実行

```
/compact [カスタムサマリー]
```

カスタムサマリーに「次に何をすべきか」を明記すると、コンパクション後のコンテキストが最適化される。

例:
```
/compact Phase 1（barrel export 分割）完了。次は Phase 2: correlation DB クエリ最適化。対象ファイル: packages/correlation/src/repositories/
```
