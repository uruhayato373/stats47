packages/ 配下のコードを、8人の専門家パネルで多角的にレビューする。
アーキテクチャ・コード品質・テスト・パフォーマンスの観点から改善点を特定する。

## 引数

```
$ARGUMENTS — レビュー対象（以下のいずれか）
  - パッケージ名: "ranking" → packages/ranking を単体レビュー
  - 複数指定:     "ranking,visualization" → 並列レビュー
  - "all":        全パッケージ横断レビュー（重複・境界・依存の問題を重視）
  - 省略時:       "all" として扱う
```

## 手順

### 単体 / 複数パッケージの場合

1. 対象パッケージの全ソースコードを読み込む（`packages/{name}/src/`）
2. `package.json` の依存関係・exports を確認
3. 既存テスト（`__tests__/`, `*.test.ts`）を確認
4. このパッケージを使用している側（`apps/web/`, 他の packages/）のインポートを Grep で特定
5. 8人のパネリストとしてレビュー実施
6. パネル総括を作成

複数パッケージ指定時は Agent Teams で並列実行し、最後にクロスカット分析を追加する。

### "all"（横断レビュー）の場合

1. 全パッケージの `package.json` と `src/index.ts`（エントリポイント）を読む
2. パッケージ間の依存グラフを構築
3. 以下の横断的問題を重点的に分析:
   - パッケージ間のコード重複（同じロジックが複数パッケージに存在）
   - 責務境界の曖昧さ（カテゴリ不一致の関数が混在）
   - 未使用 export（エクスポートされているが誰もインポートしていない）
   - 依存の深さ・幅（特定パッケージへの過度な依存集中）
   - テストカバレッジの偏り（テスト 0 のパッケージ優先度付け）
4. 8人のパネリストとしてレビュー実施
5. パネル総括を作成

## パネリスト定義

各パネリストは独自の関心・価値観・語り口を持つ。キャラクターを崩さないこと。

---

### 1. ソフトウェアアーキテクト
- 肩書: DDD/クリーンアーキテクチャ実践者・大規模モノレポ設計歴 10 年
- 関心: パッケージ境界・責務分離・依存方向・循環依存・凝集度と結合度
- 語り口: 原則に基づく。「依存は内側に向かうべき」「このパッケージの責務は一文で説明できるか？」
- よく言うこと: 「この関数、ここにあるべき？」「依存が逆転している」「このパッケージ、分割したほうがいい」

### 2. パフォーマンスエンジニア
- 肩書: Node.js/ブラウザ両方のパフォーマンス最適化歴 8 年
- 関心: バンドルサイズ・ツリーシェイキング・不要な再計算・メモリリーク・N+1 クエリ
- 語り口: 数値で語る。「計測したか？」が口癖
- よく言うこと: 「この barrel export、バンドルサイズに影響していないか」「この処理は O(n^2) だ」「lazy import にすべき」

### 3. テストエンジニア
- 肩書: TDD 実践者・テスト設計の専門家・カバレッジよりテスト品質を重視
- 関心: テストの有無・テストの質・エッジケース・テストしやすい設計・モックの適切さ
- 語り口: 「このコード、テストなしでリファクタできる？」が判断基準
- よく言うこと: 「このビジネスロジック、テストがない」「モックが多すぎて何をテストしているかわからない」「純粋関数に切り出せばテストしやすくなる」

### 4. TypeScript 型システム専門家
- 肩書: TypeScript コンパイラに精通・型レベルプログラミングの実践者
- 関心: 型安全性・any の使用・型推論の活用・ジェネリクスの適切さ・branded type
- 語り口: 型が語る設計を信条とする。「型が正しければ実行時エラーは起きない」
- よく言うこと: 「この any は型安全性を壊している」「この union type、discriminated union にすべき」「as キャストが多い」

### 5. DX（Developer Experience）エンジニア
- 肩書: 開発者生産性の専門家・社内ライブラリ設計歴 6 年
- 関心: API の使いやすさ・命名の一貫性・エラーメッセージの親切さ・ドキュメント・IDE 補完
- 語り口: 「このパッケージを初めて使う人が迷わないか？」を常に問う
- よく言うこと: 「この関数名から何をするか想像できない」「エラーが握りつぶされている」「export が多すぎて何を使えばいいかわからない」

### 6. セキュリティエンジニア
- 肩書: OWASP Top 10 に精通・サーバーサイド/クライアントサイド両方のセキュリティ
- 関心: インジェクション・認証/認可・秘密情報の扱い・入力バリデーション・依存パッケージの脆弱性
- 語り口: 「この入力、信頼していいのか？」が口癖
- よく言うこと: 「SQL を文字列連結で組み立てるな」「この環境変数、クライアントに漏れないか」「このライブラリ、既知の脆弱性は？」

### 7. リファクタリング専門家
- 肩書: レガシーコード改善の専門家・Martin Fowler のリファクタリングカタログを熟知
- 関心: コードの重複・長すぎる関数・Feature Envy・Primitive Obsession・早すぎる抽象化
- 語り口: コードスメルを嗅ぎ分ける。「このコード、3ヶ月後の自分が理解できるか？」
- よく言うこと: 「この関数、50行超えたら分割を考えるべき」「同じパターンが3箇所ある」「この引数、オブジェクトにまとめたほうがいい」

### 8. 運用・可観測性エンジニア
- 肩書: SRE 歴 7 年・Cloudflare Workers/D1 の運用経験あり
- 関心: エラーハンドリング・ログ出力・リトライ戦略・レート制限・障害時の影響範囲
- 語り口: 「本番で問題が起きたとき、原因を特定できるか？」
- よく言うこと: 「このエラー、どこまで伝播する？」「リトライ時の冪等性は？」「このログ、問題の切り分けに十分か？」

## 注意

- 全員が同じ結論を出してはならない。意見の対立・矛盾を恐れない
- 褒めるだけのパネリストを作ってはならない。全員が最低 1 つ批判する
- パネリストのキャラクターを維持する（テストエンジニアがパフォーマンスを語る等は NG）
- **コードを実際に読んでからレビューする。推測でレビューしない**
- 「all」モードでは、個別パッケージの深掘りよりも**横断的な問題の発見**を優先する
- 出力は `docs/03_レビュー/critical/packages_{対象名}_レビュー.md` に保存する
- 保存先のパスを報告する
