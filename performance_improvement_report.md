# パフォーマンス改善提案レポート

## 概要

このレポートは、プロジェクト全体のパフォーマンスを向上させるための具体的な改善点をまとめたものです。調査の結果、主に「**データ取得の最適化**」「**フロントエンドのバンドルサイズ削減**」「**静的アセットの最適化**」の3つの領域で大きな改善が見込めることが判明しました。

---

## 1. データ取得の最適化 (効果: 大)

### 現状の問題点

`doc/data-caching-strategy.md`の分析から、e-Stat APIからのデータ取得が毎回行われていることが確認されました。これにより、以下の問題が発生しています。

-   **レスポンスの遅延**: APIからの応答に2〜5秒かかり、ユーザー体験を著しく損なっています。
-   **API制限のリスク**: APIの呼び出し回数には上限がある可能性があり、頻繁なアクセスはサービスの安定性を脅かします。
-   **可用性の低下**: e-Stat APIの障害時に、データが表示できなくなります。

### 提案: データベースキャッシュの導入

`data-caching-strategy.md`で提案されている通り、Cloudflare D1データベースを用いたキャッシュ戦略を導入することを強く推奨します。

#### 補足: Next.js `fetch`キャッシュとの違い

Next.jsには標準で強力な`fetch`キャッシュ機能が備わっていますが、本プロジェクトのコア機能である統計データを扱う上では、データベースキャッシュがより優れています。主な理由は以下の2点です。

1.  **可用性と耐障害性**: `fetch`キャッシュは外部API（e-Stat）が正常に稼働していることが前提です。APIがダウンすると、キャッシュが切れた際にデータ取得に失敗します。一方、データベースキャッシュは一度データを取得すれば、**外部APIから独立してデータを供給し続けられる**ため、サービスの安定性が格段に向上します。

2.  **データ加工とクエリの柔軟性**: `fetch`キャッシュはAPIレスポンスをそのまま保存するだけです。しかし、本プロジェクトでは「特定地域のランキング」や「複数年にまたがるデータの集計」など、複雑なデータ操作が求められます。データベースにデータを格納することで、**SQLを用いた高度で効率的なクエリが可能になり**、アプリケーションサーバー側での重い処理を削減できます。

以上の理由から、サービスの信頼性と拡張性を確保するために、データベースキャッシュの導入が最適なアプローチとなります。

#### 実装手順

1.  **キャッシュ用テーブルの作成**:
    `data-caching-strategy.md`に記載されているSQLスキーマを参考に、`estat_data_cache`テーブルをデータベースに作成します。

    ```sql
    -- 統計データキャッシュテーブル
    CREATE TABLE IF NOT EXISTS estat_data_cache (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      stats_data_id TEXT NOT NULL,           -- 統計表ID
      category_code TEXT NOT NULL,           -- カテゴリコード
      time_code TEXT NOT NULL,               -- 年度コード
      area_code TEXT NOT NULL,               -- 地域コード
      value REAL,                            -- 数値
      unit TEXT,                             -- 単位
      cached_at DATETIME DEFAULT CURRENT_TIMESTAMP,  -- キャッシュ日時
      expires_at DATETIME,                   -- 有効期限
      UNIQUE(stats_data_id, category_code, time_code, area_code)
    );
    ```

2.  **キャッシュ管理サービスの実装**:
    `EstatCacheService.ts`を作成し、以下のロジックを実装します。
    -   まずデータベースに問い合わせる。
    -   キャッシュが存在し、有効期限内であれば、そのデータを返す (キャッシュヒット)。
    -   キャッシュが存在しないか、期限切れであれば、e-Stat APIからデータを取得し、DBに保存してから返す (キャッシュミス)。

3.  **既存ロジックの置き換え**:
    現在APIを直接呼び出している箇所を、新しく作成した`EstatCacheService`経由でデータを取得するように変更します。

#### 期待される効果

-   **応答時間の大幅な短縮**: 2回目以降のアクセスが**10-100ms**に短縮 (95%以上の改善)。
-   **API呼び出し回数の削減**: APIへの負荷を90%以上削減。
-   **サービス可用性の向上**: API障害時でもキャッシュされたデータを提供可能。

## 2. フロントエンドのバンドルサイズ削減 (効果: 中)

### 現状の問題点

アプリケーションの初回読み込み速度は、クライアントに送信されるJavaScriptの量に大きく影響されます。現状では、どのライブラリがバンドルサイズを肥大化させているか不明確です。

### 提案: バンドルサイズの可視化と最適化

`@next/bundle-analyzer`を導入し、不要なライブラリやコードが含まれていないかを確認します。

#### 分析手順

1.  **ビルドコマンドの実行**:
    以下のコマンドを実行して、バンドル分析レポートを生成します。

    ```bash
    ANALYZE=true npm run build
    ```

2.  **レポートの確認**:
    ビルド完了後、`.next/analyze/`ディレクトリに`client.html`と`server.html`が生成されます。これらのファイルをブラウザで開き、特にサイズの大きいチャンクやライブラリを特定します。

#### 最適化のポイント

-   **D3.jsのモジュール単位でのインポート**: `d3`ライブラリは非常に大きいですが、多くの場合は一部のモジュールしか使用しません。必要なモジュールだけを個別にインポートすることで、バンドルサイズを大幅に削減できます。

    ```typescript
    // 悪い例
    import * as d3 from 'd3';

    // 良い例
    import { select } from 'd3-selection';
    import { scaleLinear } from 'd3-scale';
    ```

-   **コンポーネントの動的インポート**: 地図やグラフなど、初期表示に必須でない重いコンポーネントは、`next/dynamic`を使って動的に読み込みます。これにより、初期バンドルサイズが削減され、ページの表示が高速化します。

    ```tsx
    import dynamic from 'next/dynamic'

    const HeavyComponent = dynamic(() => import('../components/HeavyComponent'), {
      ssr: false, // サーバーサイドではレンダリングしない場合
      loading: () => <p>Loading...</p>
    })

    function MyPage() {
      return <HeavyComponent />
    }
    ```

## 3. 静的アセットの最適化 (効果: 中〜大)

### 現状の問題点

日本の地図を表示するために使用されている`public/data/jp_pref.l.topojson`のファイルサイズが**3.4MB**もあります。このサイズのファイルをクライアントがダウンロードするのは、特にモバイル環境において大きなボトルネックとなります。

### 提案: TopoJSONファイルの軽量化

地図の表示に必要な精度を保ちつつ、ファイルサイズを削減するべきです。

#### 実装手順

1.  **TopoJSONの簡素化**:
    [mapshaper.org](https://mapshaper.org/) などのツールを使用して、TopoJSONファイルのポリゴンを簡素化します。
    -   ファイルをアップロードし、画面右上の「Simplify」をクリックします。
    -   視覚的に許容できる範囲で、スライダーを動かして簡素化率を調整します (例: 10%〜30%)。
    -   簡素化したファイルをエクスポートし、既存のファイルと置き換えます。

2.  **サーバーサイドでの事前処理**:
    可能であれば、クライアントで読み込む前に、サーバーサイドで必要な部分だけを抽出・加工することで、転送量を削減することも検討します。

#### 期待される効果

-   **ファイルサイズの削減**: 数百KB程度まで削減できる可能性があります。
-   **地図表示の高速化**: ファイルのダウンロード時間が短縮され、地図の初期表示が速くなります。

## まとめ

以上の3点を実施することで、アプリケーションのパフォーマンスは大幅に向上する見込みです。特に「**1. データ取得の最適化**」はユーザー体験に最も大きな影響を与えるため、優先的に取り組むことを推奨します。
