# ストレージ選択ガイド

## 概要

stats47 プロジェクトにおける D1 Database と R2 Storage の使い分け基準を明確化し、エンティティ vs 値オブジェクトの設計判断と連携させたストレージ選択指針を提供します。

## D1 vs R2 の判断フローチャート

```
1. データは変更されるか？
   YES → 次へ
   NO → R2（値オブジェクト）

2. 複雑なクエリが必要か？
   YES → D1（エンティティ）
   NO → 次へ

3. 全件取得が主な用途か？
   YES → R2（値オブジェクト）
   NO → D1（エンティティ）
```

## エンティティ vs 値オブジェクトとの関係

| 特性               | エンティティ           | 値オブジェクト         |
| ------------------ | ---------------------- | ---------------------- |
| **ストレージ**     | D1 Database            | R2 Storage / 静的 JSON |
| **識別性**         | ID で管理              | 値で識別               |
| **変更**           | 可能（CRUD）           | 不変                   |
| **クエリ**         | SQL                    | JavaScript filter      |
| **ライフサイクル** | 作成 → 更新 → 削除     | 不変                   |
| **関係性**         | 他のエンティティと関連 | 独立                   |

## ドメイン別の適用例

### Area ドメイン ✅ R2 + 値オブジェクト

**理由**:

- 都道府県・市区町村は不変（コードが識別子）
- 更新頻度が低い（年数回の市町村合併のみ）
- 全件取得が主な用途
- JavaScript 配列の `filter` で十分

**実装**:

```typescript
// 値オブジェクトとして設計
export class Prefecture {
  constructor(
    public readonly code: PrefectureCode,
    public readonly name: string
  ) {}

  equals(other: Prefecture): boolean {
    return this.code.equals(other.code);
  }
}

// R2/静的JSON からの読み込み
export class AreaRepository {
  async getPrefectures(): Promise<Prefecture[]> {
    const data = await import("@/config/areas/prefectures.json");
    return data.prefectures.map(Prefecture.fromJson);
  }
}
```

### Ranking ドメイン ✅ D1 + エンティティ

**理由**:

- ランキングデータは定期更新
- 複雑なクエリが必要（TOP N、カテゴリ別など）
- ライフサイクル管理が必要

**実装**:

```typescript
// エンティティとして設計
export class RankingItem {
  constructor(
    private readonly id: RankingItemId,
    private areaCode: AreaCode,
    private rank: number,
    private value: number
  ) {}

  updateRank(newRank: number): void {
    this.rank = newRank;
  }
}

// D1 での永続化
export class RankingRepository {
  async findTopByCategory(
    categoryId: string,
    limit: number
  ): Promise<RankingItem[]> {
    const query = `
      SELECT * FROM ranking_items 
      WHERE category_id = ? 
      ORDER BY rank ASC 
      LIMIT ?
    `;
    return await this.d1.prepare(query).bind(categoryId, limit).all();
  }
}
```

### Geoshape ドメイン ✅ R2 + 値オブジェクト

**理由**:

- TopoJSON は不変
- 大容量データのキャッシュに最適
- 年度・解像度別の取得のみ

**実装**:

```typescript
// 値オブジェクトとして設計
export class GeoshapeData {
  constructor(
    public readonly year: string,
    public readonly level: string,
    public readonly resolution: Resolution,
    public readonly topoJsonData: TopoJSON
  ) {}
}

// R2 でのキャッシュ
export class GeoshapeRepository {
  async getTopoJson(
    year: string,
    level: string,
    resolution: Resolution
  ): Promise<GeoshapeData> {
    const cacheKey = `geoshape:${year}:${level}:${resolution}`;
    const cached = await this.r2.get(cacheKey);
    if (cached) return GeoshapeData.fromJson(cached);

    const data = await this.fetchFromGeoshape(year, level, resolution);
    await this.r2.put(cacheKey, data, { ttl: 30 * 24 * 60 * 60 * 1000 });
    return data;
  }
}
```

### EstatAPI ドメイン ✅ R2 + 値オブジェクト

**理由**:

- API レスポンスは不変
- キャッシュ戦略が重要
- 複雑なクエリは不要

**実装**:

```typescript
// 値オブジェクトとして設計
export class EstatResponse {
  constructor(
    public readonly apiType: EstatApiType,
    public readonly params: EstatParams,
    public readonly data: any,
    public readonly timestamp: Date
  ) {}
}

// R2 でのキャッシュ
export class EstatRepository {
  async getCachedResponse(
    apiType: EstatApiType,
    params: EstatParams
  ): Promise<EstatResponse | null> {
    const cacheKey = this.buildCacheKey(apiType, params);
    const cached = await this.r2.get(cacheKey);
    return cached ? EstatResponse.fromJson(cached) : null;
  }
}
```

### Content ドメイン ✅ R2 + 値オブジェクト

**理由**:

- MDX コンテンツは不変
- 静的ファイルとして配信
- 検索は JavaScript で十分

**実装**:

```typescript
// 値オブジェクトとして設計
export class Content {
  constructor(
    public readonly slug: Slug,
    public readonly title: string,
    public readonly content: string,
    public readonly frontmatter: Frontmatter
  ) {}
}

// 静的ファイルからの読み込み
export class ContentRepository {
  async getContent(slug: string): Promise<Content> {
    const filePath = `content/blog/${slug}.mdx`;
    const fileContent = await fs.readFile(filePath, "utf-8");
    return Content.fromMdx(fileContent);
  }
}
```

## 詳細な判断基準

### D1 Database を選択すべき場合

#### ✅ 適しているケース

1. **動的データ**

   - ランキングデータ（定期更新）
   - ユーザー設定（リアルタイム変更）
   - セッション情報（一時的）

2. **複雑なクエリ**

   - JOIN が必要
   - 集計関数の使用
   - 複雑な WHERE 条件

3. **CRUD 操作**
   - 作成・更新・削除が頻繁
   - トランザクションが必要
   - データ整合性が重要

#### ❌ 不適切なケース

1. **静的データ**

   - 地域マスタ（変更頻度が低い）
   - 設定ファイル（ビルド時固定）

2. **単純なアクセス**
   - 全件取得のみ
   - キーによる単純検索

### R2 Storage を選択すべき場合

#### ✅ 適しているケース

1. **静的・準静的データ**

   - 地域マスタ（年数回の更新）
   - 地理データ（年度別）
   - 設定ファイル

2. **ファイルベースのデータ**

   - JSON ファイル
   - 画像・動画
   - ドキュメント

3. **キャッシュデータ**
   - API レスポンス
   - 計算結果
   - 一時的なデータ

#### ❌ 不適切なケース

1. **頻繁な更新**

   - リアルタイムデータ
   - ユーザー入力データ

2. **複雑なクエリ**
   - JOIN が必要
   - 集計処理

## パフォーマンス比較

### D1 Database

**メリット**:

- SQL による高速検索
- インデックスによる最適化
- トランザクション保証

**デメリット**:

- 読み取り課金
- 接続オーバーヘッド
- 複雑なクエリは遅い

### R2 Storage

**メリット**:

- 読み取り無料
- CDN 配信で高速
- シンプルなアクセス

**デメリット**:

- 複雑な検索ができない
- ファイル単位でのアクセス
- 更新時の整合性管理が困難

## コスト分析

### D1 Database のコスト

```
読み取り: $0.001/1M リクエスト
書き込み: $0.001/1M リクエスト
ストレージ: $0.75/GB/月
```

**適用例**: Ranking ドメイン

- 読み取り: 100 万リクエスト/月 = $0.001
- 書き込み: 10 万リクエスト/月 = $0.0001
- ストレージ: 1GB = $0.75
- **月額合計**: 約 $0.75

### R2 Storage のコスト

```
読み取り: 無料
書き込み: $0.004/1M リクエスト
ストレージ: $0.015/GB/月
```

**適用例**: Area ドメイン

- 読み取り: 無制限 = $0
- 書き込み: 月 1 回 = $0.000004
- ストレージ: 10MB = $0.00015
- **月額合計**: 約 $0.0002

## 移行戦略

### 値オブジェクト → エンティティ

**トリガー**:

- データの更新頻度が増加
- 複雑なクエリが必要
- ライフサイクル管理が必要

**移行手順**:

1. エンティティクラスの作成
2. D1 テーブルの設計
3. データ移行スクリプト
4. リポジトリの更新

### エンティティ → 値オブジェクト

**トリガー**:

- データが不変になった
- シンプルなアクセスパターン
- コスト削減が必要

**移行手順**:

1. 値オブジェクトクラスの作成
2. R2/静的ファイルへの移行
3. キャッシュ戦略の実装
4. リポジトリの更新

## ベストプラクティス

### 設計原則

1. **単一責任**: 各ストレージは明確な責務を持つ
2. **適材適所**: データ特性に応じた選択
3. **コスト意識**: 読み取り無料の R2 を積極活用
4. **パフォーマンス**: CDN 配信による高速化

### 実装ガイドライン

1. **インターフェース統一**: リポジトリパターンで抽象化
2. **キャッシュ戦略**: 3 層キャッシュの活用
3. **エラーハンドリング**: フォールバック戦略の実装
4. **監視**: パフォーマンスとコストの監視
