# Architecture Decision Records (ADR)

## 概要

このディレクトリには、stats47 プロジェクトの重要な技術的決定を記録した ADR を格納します。

ADR は、なぜその技術を選んだのか、なぜその設計にしたのかを記録し、将来の開発者が判断の背景を理解できるようにします。

## ADR 作成ガイドライン

### 作成タイミング

以下の場合に ADR を作成してください：

1. **新しい技術の導入**: フレームワーク、ライブラリ、ツールの選定
2. **アーキテクチャの変更**: システム設計の大幅な変更
3. **設計パターンの採用**: 新しい設計パターンやアプローチの採用
4. **技術的制約の解決**: 技術的課題に対する解決策の決定

### 作成手順

1. **テンプレートを使用**: [template.md](./template.md) をコピーして使用
2. **番号を割り当て**: 次の連番を付与（例: 008\_）
3. **ステータス管理**: draft → in-review → accepted/rejected
4. **レビュー**: 技術リーダーによるレビュー
5. **承認**: 承認後にステータスを更新

### ステータス定義

- **draft**: 作成中・下書き状態
- **in-review**: レビュー中
- **accepted**: 承認済み・採用決定
- **rejected**: 却下・採用見送り
- **superseded**: 別の ADR に置き換えられた

### 品質基準

ADR は以下の基準を満たす必要があります：

1. **明確な問題定義**: 解決すべき問題が明確
2. **選択肢の検討**: 複数の選択肢を検討
3. **根拠の明示**: 決定理由が明確
4. **影響の分析**: 決定による影響を分析
5. **実装方針**: 具体的な実装方針を記載

## ADR 管理

### 定期的な見直し

- **四半期レビュー**: ADR の妥当性を確認
- **技術更新時**: 関連する ADR の見直し
- **プロジェクト終了時**: 全 ADR のアーカイブ
