# 状態管理設計

## 概要

stats47 プロジェクトでは、Zustand を基盤とした状態管理アーキテクチャを採用しています。

## 状態管理アーキテクチャ

### 基本原則

1. **シンプルな状態管理**: 状態とアクションを一箇所で管理
2. **単方向データフロー**: データの流れを明確化
3. **型安全性**: TypeScript による型チェック
4. **パフォーマンス**: 必要な部分のみ再レンダリング

### 状態の分類

```typescript
// 1. 基本状態
interface Store {
  selectedCategory: string | null;
  selectedSubcategory: string | null;
  loading: boolean;
  error: string | null;
}

// 2. 派生状態（getter）
interface Store {
  getSelectedCategoryData: () => CategoryData | null;
  getAvailableSubcategories: () => SubcategoryData[];
}

// 3. アクション
interface Store {
  setCategory: (categoryId: string | null) => void;
  setSubcategory: (subcategoryId: string | null) => void;
  resetSelection: () => void;
}
```

## ベストプラクティス

### 1. ストア設計

```typescript
// ✅ Good: 明確な責務を持つストア
interface UserStore {
  user: User | null;
  setUser: (user: User | null) => void;
  clearUser: () => void;
}

// ❌ Bad: 責務が曖昧なストア
interface GlobalStore {
  user: User | null;
  theme: string;
  data: any[];
  // 複数の責務が混在
}
```

### 2. 状態の更新

```typescript
// ✅ Good: 不変性を保つ
set((state) => ({
  ...state,
  selectedCategory: newCategory,
}));

// ❌ Bad: 直接的な変更
set((state) => {
  state.selectedCategory = newCategory; // 不変性を破る
  return state;
});
```

### 3. 型安全性

```typescript
// ✅ Good: 明確な型定義
interface Store {
  count: number;
  increment: () => void;
}

// ❌ Bad: any型の使用
interface Store {
  data: any; // 型安全性を失う
}
```
