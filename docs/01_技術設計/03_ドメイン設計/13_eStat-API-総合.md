# eStat API（e-Stat API 統合）ドメイン設計

## 概要

eStat API（e-Stat API 統合）ドメインは、stats47 プロジェクトのコアドメインの一つで、e-Stat API との統合を担当します。

e-Stat API（MetaInfo、StatsData、StatsList）との統合、API パラメータの管理とマッピング、データ取得・変換・正規化、e-Stat 特有のエラーハンドリングなど、e-Stat API との連携に関するすべての機能を提供します。

**注意**: estat-apiドメインとランキングドメインは、ともにコアドメインとして扱われます。estat-apiドメインはデータ探索・品質保証を担当し、ランキングドメインはランキング管理・可視化を担当します。

### ビジネス価値

- **e-Stat API 統合**: 政府統計データへの効率的なアクセス
- **データ品質保証**: e-Stat 特有のデータ検証と品質管理
- **パフォーマンス最適化**: API 呼び出しの最適化とキャッシュ連携
- **拡張性**: 新しい e-Stat API エンドポイントの追加が容易
- **統計データの可視化基盤**: ランキング、ダッシュボード、可視化のデータソース

## 責務と主要概念

### 責務

1. **統計データの取得と整形**

   - e-Stat API から統計データを取得
   - 構造化された形式への変換
   - データのフィルタリング（カテゴリ、年度、地域）
   - NULL 値や特殊文字の適切な処理

2. **メタ情報の管理**

   - 統計表のメタ情報（カテゴリ、単位など）の取得・保存・検索
   - データベースへの保存
   - 効率的な一括処理（バッチ処理）

3. **統計リストの取得**

   - 利用可能な統計表の一覧取得
   - キーワード検索
   - フィルタリングとページネーション

4. **API パラメータの管理とマッピング**

   - e-Stat API 呼び出しのパラメータ管理
   - 統計データ ID から API パラメータへの変換

5. **データ取得・変換・正規化**

   - 生データの内部形式への変換
   - データの正規化処理

6. **e-Stat 特有のエラーハンドリングとリトライ**
   - レート制限管理
   - エラー処理とリトライロジック

7. **データ探索・確認機能**
   - stats-list検索による統計表の探索
   - メタ情報取得・確認
   - データプレビューによる品質確認
   - ランキング登録に必要なパラメータ（statsDataId、cdCat01など）の特定支援

8. **ランキングデータの生成とR2保存**
   - `estat_api_metadata`テーブルからマッピング情報を取得
   - `fetchStatsData()`で生データ取得
   - ランキング形式（`RankingDataPoint[]`）に変換（値のみ、順位なし）
   - R2ストレージに保存（パス: `ranking/{rankingKey}/{areaType}/{timeCode}.json`）
   - **注意**: 順位計算は含まない（ランキング機能が担当）

### 主要概念

#### StatsDataId（統計表 ID）

e-Stat の各統計表に付与される一意の識別子を表現する値オブジェクト。

**具体例**: `0000010101`（人口推計）、`0003000001`（人口・世帯）

**制約**: 10 桁の数字、e-Stat で定義された ID のみ有効

**用途**: e-Stat API 呼び出し（`statsDataId`パラメータ）、データベースキー、統計データの一意識別

#### API パラメータ

e-Stat API 呼び出しで使用するパラメータ名。

**分類事項パラメータ**:

- **`cdCat01`〜`cdCat15`**: 分類事項 01-15 の絞り込みパラメータ（任意）
- 具体例: `cdCat01=001,002`（総人口、男性人口）

**地域パラメータ**:

- **`cdArea`**: 地域コード指定パラメータ（任意）
- 具体例: `cdArea=13000,27000`（東京都、大阪府）

**時間軸パラメータ**:

- **`cdTime`**: 時間軸コード指定パラメータ（任意）
- 具体例: `cdTime=2023000000`（2023 年）

**詳細**: 各サブドメインのドキュメントを参照

- [統計データ取得 API パラメータ](./16_eStat-API-統計データ.md#api-パラメータ)
- [メタ情報取得 API パラメータ](./15_eStat-API-メタ情報.md#api-パラメータ)

#### API レスポンス構造

e-Stat API のレスポンス構造。

**CLASS_INF（分類情報）**:

統計データの各次元（軸）の詳細情報を提供するレスポンス構造。

- **構造**: `CLASS_INF.CLASS_OBJ[]`（分類オブジェクトの配列）

**CLASS_OBJ（分類オブジェクト）**:

特定の次元（表章項目、分類、地域、時間等）の定義。

- **`@id`**: 分類 ID（`tab`、`cat01`〜`cat15`、`area`、`time`）
- **`@name`**: 分類名（例: `"表章項目"`、`"地域"`）
- **`CLASS`**: 分類項目の配列

**CLASS（分類項目）**:

各次元の具体的な項目（都道府県、年度、カテゴリ等）。

- **`@code`**: 項目コード（例: `"001"`、`"13000"`、`"A1101"`）
- **`@name`**: 項目名称（例: `"総数"`、`"東京都"`）
- **`@unit`**: 単位（例: `"人"`、`"%"`）
- **`@level`**: 階層レベル（`"1"`: 最上位、`"2"`: 第 2 階層）
- **`@parentCode`**: 親項目コード

**詳細**: 各サブドメインのドキュメントを参照

- [統計データ取得 API レスポンス構造](./16_eStat-API-統計データ.md#api-レスポンス構造)
- [メタ情報取得 API レスポンス構造](./15_eStat-API-メタ情報.md#api-レスポンス構造)

#### ランキングデータ生成フロー

estat-apiドメインがランキングデータを生成し、R2ストレージに保存するフロー（バックグラウンド処理）。

```
ランキングデータ生成リクエスト（バックグラウンド処理）
    │
    ▼
estat-apiドメイン: ランキングデータ生成とR2保存
    │
    ├─► estat_api_metadataテーブルからマッピング情報取得
    │       │
    │       └─► statsDataId, cdCat01, cdCat02, ...
    │
    ├─► fetchStatsData()で生データ取得
    │       │
    │       └─► e-Stat APIから統計データ取得
    │
    ├─► ランキング形式に変換（値のみ、順位なし）
    │       │
    │       └─► RankingDataPoint[]形式
    │           - areaCode, areaName, value のみ
    │           - rank, percentile は含まない
    │
    └─► R2ストレージに保存
            │
            └─► パス: ranking/{rankingKey}/{areaType}/{timeCode}.json
                - values: [{areaCode, areaName, value}]
                - statistics: {min, max, mean, median}
                - metadata: {rankingKey, timeCode, unit}
```

### estat_api_metadataテーブル

estat-apiドメイン固有のパラメータをランキングキーと紐付けて保存するテーブル。テーブルはランキングドメインが管理（書き込み・読み取り）しますが、estat-apiドメインがパラメータ構造を定義します。

**テーブル構造**:
- `ranking_key`: ランキングキー（ranking_itemsの外部キー）
- `area_type`: 地域レベル（prefecture/city/national）
- `calculation_type`: 計算タイプ（direct/ratio/aggregate）
- `metadata`: e-Stat API固有のパラメータ（JSON形式）

**metadata JSON構造例**:
```json
{
  "stats_data_id": "0000010102",
  "cd_cat01": "B1101",
  "cd_area": "00000",
  "cd_time": "2023000000"
}
```

**注意**: このテーブルはe-Stat API専用です。他のデータソース用メタデータは別テーブルで管理されます。

### 設計パターン

#### 1. Fetcher-Formatter パターン

**目的**: データ取得と整形の責務を分離

- **Fetcher**: HTTP 通信と生データ取得
- **Formatter**: データの整形と変換

#### 2. Core Layer パターン

**目的**: 共通インフラ層を提供し、再利用性を確保

- **HTTP Client**: 汎用的な HTTP 通信
- **Constants**: API エンドポイントと設定
- **Types**: 共通型定義
- **Errors**: エラー型定義

#### 3. R2 キャッシュ戦略

**目的**: API 呼び出しの最適化

- 初回: e-Stat API から取得 → R2 に保存（バックグラウンド）
- 2 回目以降: R2 キャッシュから取得
- フォールバック: R2 障害時は e-Stat API に直接アクセス

## データソースとストレージ

### データソース

#### e-Stat API（外部サービス）

**用途**: 統計データ、メタ情報、統計リストの取得

**アクセス方法**: HTTP REST API

**認証**: アプリケーション ID（環境変数 `NEXT_PUBLIC_ESTAT_APP_ID`）

**レート制限**: 1 日あたり 1,000 回、1 時間あたり 100 回（推奨）

### ストレージ

#### D1 Database

各サブドメインで使用するD1データベースの詳細については、それぞれのドキュメントを参照してください。

- **統計データ**: [eStat API（統計データ）サブドメイン設計](./16_eStat-API-統計データ.md#データソースとストレージ)
- **メタ情報**: [eStat API（メタ情報）サブドメイン設計](./15_eStat-API-メタ情報.md#データソースとストレージ)
- **統計リスト**: [eStat API（統計表リスト）サブドメイン設計](./14_eStat-API-統計表リスト.md#データソースとストレージ)
- **メタデータテーブル**: `estat_api_metadata`テーブル（ランキング機能が書き込み、estat-apiドメインが構造定義）

#### R2 Storage

estat-apiドメインは、ランキングデータをR2ストレージに保存します。

**保存パス**: `ranking/{rankingKey}/{areaType}/{timeCode}.json`

**保存データ構造**:

```json
{
  "values": [
    {
      "areaCode": "13000",
      "areaName": "東京都",
      "value": 6439.3
    }
  ],
  "statistics": {
    "min": 64.5,
    "max": 6439.3,
    "mean": 338.7,
    "median": 124.5
  },
  "metadata": {
    "rankingKey": "population-density",
    "timeCode": "2023",
    "unit": "人/km²",
    "dataSourceId": "estat"
  }
}
```

**注意**: 保存されるデータは値のみで、順位（rank）とパーセンタイル（percentile）は含まれません。これらはランキング機能が視覚化設定に基づいて計算します。

## 制約と前提条件

### 制約

1. **レート制限**: e-Stat API の制限を遵守する必要がある
2. **データ取得サイズ**: 1 回の取得で最大 10,000 件まで
3. **統計表 ID の形式**: 10 桁の数字のみ有効

### 前提条件

1. **環境変数**: `NEXT_PUBLIC_ESTAT_APP_ID` が設定されている必要がある
2. **ネットワーク**: e-Stat API へのインターネットアクセスが必要
3. **データの可用性**: 統計データの可用性は e-Stat API に依存
4. **R2 設定**: メタ情報のキャッシュを使用する場合、R2 の設定が必要（オプション）
