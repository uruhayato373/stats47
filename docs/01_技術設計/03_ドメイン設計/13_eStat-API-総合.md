# eStat API（e-Stat API 統合）ドメイン設計

## 概要

eStat API（e-Stat API 統合）ドメインは、stats47 プロジェクトの支援ドメインの一つで、e-Stat API との統合を担当します。e-Stat API（MetaInfo、StatsData、StatsList）との統合、API パラメータの管理とマッピング、データ取得・変換・正規化、e-Stat 特有のエラーハンドリングなど、e-Stat API との連携に関するすべての機能を提供します。

### ビジネス価値

- **e-Stat API 統合**: 政府統計データへの効率的なアクセス
- **データ品質保証**: e-Stat 特有のデータ検証と品質管理
- **パフォーマンス最適化**: API 呼び出しの最適化とキャッシュ連携
- **拡張性**: 新しい e-Stat API エンドポイントの追加が容易
- **統計データの可視化基盤**: ランキング、ダッシュボード、可視化のデータソース

## 責務と主要概念

### 責務

1. **統計データの取得と整形**

   - e-Stat API から統計データを取得
   - 構造化された形式への変換
   - データのフィルタリング（カテゴリ、年度、地域）
   - NULL 値や特殊文字の適切な処理

2. **メタ情報の管理**

   - 統計表のメタ情報（カテゴリ、単位など）の取得・保存・検索
   - データベースへの保存
   - 効率的な一括処理（バッチ処理）

3. **統計リストの取得**

   - 利用可能な統計表の一覧取得
   - キーワード検索
   - フィルタリングとページネーション

4. **API パラメータの管理とマッピング**

   - e-Stat API 呼び出しのパラメータ管理
   - ランキングキーから API パラメータへの変換

5. **データ取得・変換・正規化**

   - 生データの内部形式への変換
   - データの正規化処理

6. **e-Stat 特有のエラーハンドリングとリトライ**
   - レート制限管理
   - エラー処理とリトライロジック

### 主要概念

#### StatsDataId（統計表 ID）

e-Stat の各統計表に付与される一意の識別子を表現する値オブジェクト。

**具体例**: `0000010101`（人口推計）、`0003000001`（人口・世帯）

**制約**: 10 桁の数字、e-Stat で定義された ID のみ有効

**用途**: e-Stat API 呼び出し、データベースキー、統計データの一意識別

#### RankingKey（ランキングキー）

e-Stat のランキングキーを表現する値オブジェクト。

**具体例**: `0003000001-001`（人口・世帯-基本人口）、`0003000002-001`（労働力調査-完全失業率）

**制約**: 統計データ ID + ハイフン + サブ ID、e-Stat で定義された形式

**用途**: 統計データの特定、ランキング計算、API 呼び出しパラメータ

#### CategoryCode（カテゴリコード）

統計データの分類を示すコード。

**具体例**: `A1101`（総人口）、`A1301`（男性人口）、`A1401`（女性人口）

**制約**: 最大 15 種類まで設定可能（cat01-15）、通常は `cat01` が主要カテゴリ

**用途**: 統計データの分類、フィルタリング

#### AreaCode（地域コード）

統計データの対象地域を示すコード。

**具体例**: `00000`（全国）、`13000`（東京都）、`27000`（大阪府）

**制約**: 5 桁のコード体系

**用途**: 統計データの地域フィルタリング、地域別集計

#### TimeCode（時間軸コード）

統計データの対象時期を示すコード。

**具体例**: `2020`（2020 年）、`2020CY`（2020 暦年）、`202001`（2020 年 1 月）

**用途**: 統計データの期間フィルタリング、時系列分析

### 設計パターン

#### 1. Fetcher-Formatter パターン

**目的**: データ取得と整形の責務を分離

- **Fetcher**: HTTP 通信と生データ取得
- **Formatter**: データの整形と変換

#### 2. Core Layer パターン

**目的**: 共通インフラ層を提供し、再利用性を確保

- **HTTP Client**: 汎用的な HTTP 通信
- **Constants**: API エンドポイントと設定
- **Types**: 共通型定義
- **Errors**: エラー型定義

#### 3. R2 キャッシュ戦略

**目的**: API 呼び出しの最適化

- 初回: e-Stat API から取得 → R2 に保存（バックグラウンド）
- 2 回目以降: R2 キャッシュから取得
- フォールバック: R2 障害時は e-Stat API に直接アクセス

## データソースとストレージ

### データソース

#### e-Stat API（外部サービス）

**用途**: 統計データ、メタ情報、統計リストの取得

**アクセス方法**: HTTP REST API

**認証**: アプリケーション ID（環境変数 `NEXT_PUBLIC_ESTAT_APP_ID`）

**レート制限**: 1 日あたり 1,000 回、1 時間あたり 100 回（推奨）

## 制約と前提条件

### 制約

1. **レート制限**: e-Stat API の制限を遵守する必要がある
2. **データ取得サイズ**: 1 回の取得で最大 10,000 件まで
3. **統計表 ID の形式**: 10 桁の数字のみ有効

### 前提条件

1. **環境変数**: `NEXT_PUBLIC_ESTAT_APP_ID` が設定されている必要がある
2. **ネットワーク**: e-Stat API へのインターネットアクセスが必要
3. **データの可用性**: 統計データの可用性は e-Stat API に依存
4. **R2 設定**: メタ情報のキャッシュを使用する場合、R2 の設定が必要（オプション）
