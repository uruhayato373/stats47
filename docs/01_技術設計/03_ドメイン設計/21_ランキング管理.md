# Ranking ドメイン

## 概要

Ranking ドメインは、stats47 プロジェクトのコアドメインの一つで、ビジネスの中核価値を提供する最も重要なドメインです。統計データのランキング計算、地域プロファイル生成、比較分析など、統計サイトの中核機能を担当します。

### ビジネス価値

- **データドリブンな意思決定支援**: 統計データの分析により、ユーザーがデータに基づいた判断を行える
- **地域の特徴把握**: 地域プロファイル機能により、地域の強みや特徴を可視化
- **比較分析**: 複数地域間の比較により、相対的な位置づけを理解

### 最新の改善 (2025-01-27)

- **ランキンググループ機能**: 関連する複数のランキング項目をグループ化して管理
- **改善された UI**: グループ化により、サイドバーでの項目管理がより明確に

## 責務と主要概念

### 責務

1. **ランキング計算とデータ取得**

   - 統計データからランキング計算
   - 順位とパーセンタイルの算出
   - データソースからのデータ取得と変換

2. **ランキング項目の管理**

   - ランキング項目のグループ管理
   - ランキング項目の定義とメタデータ管理
   - 表示順序と可視化設定の管理

3. **比較分析と可視化**

   - 地域間の比較分析
   - 時系列データの傾向分析
   - 統計サマリー生成
   - 地域プロファイル生成

### 主要概念

#### RankingKey（ランキングキー）

ランキング項目の一意識別子を表現する値オブジェクト。

**具体例**:

- `total-population`: 総人口
- `population-density`: 人口密度
- `gdp-per-capita`: 一人当たり GDP
- `unemployment-rate`: 失業率
- `elderly-population-ratio`: 高齢化率

**制約**:

- 空文字列は不可
- ケバブケース形式（小文字とハイフン）
- 一意である必要がある
- 最大 50 文字

**用途**:

- ランキング項目の識別
- URL パラメータとして使用（例: `/population/basic-population/ranking/total-population`）
- データベースのキーとして使用
- API レスポンスでの項目指定

## ドメインモデル

### 主要エンティティ

#### RankingGroup（ランキンググループ）

関連するランキング項目をグループ化して管理するエンティティ。複数の関連項目（例：「製造品出荷額」「製造品出荷額（事業所あたり）」）をまとめて表示する。

**属性**:

- `groupKey`: グループの一意識別子
- `subcategoryId`: サブカテゴリ ID
- `name`: グループ名
- `description`: グループの説明
- `displayOrder`: サブカテゴリ内での表示順
- `isCollapsed`: デフォルトで折り畳み表示するか
- `items`: グループに属するランキング項目の配列

#### RankingItem（ランキング項目）

統計指標の定義とメタデータを管理するエンティティ。

**属性**:

- `rankingKey`: ランキングの一意識別子
- `label`: 表示用ラベル
- `unit`: 単位
- `groupKey`: ランキンググループキー
- `displayOrder`: 表示順序
- `visualization`: 視覚化情報（JSON 形式）
- `isActive`: 有効フラグ

#### RankingValue（ランキング値）

地域ごとの統計値とランキング情報を管理するエンティティ。

**属性**:

- `areaCode`: 地域コード
- `value`: 値
- `rank`: 順位
- `percentile`: パーセンタイル
- `year`: 年度
- `timeCode`: 時間コード

#### RegionProfile（地域プロファイル）

地域の総合的な統計プロファイルを管理するエンティティ。

**属性**:

- `areaCode`: 地域コード
- `basicInfo`: 基本情報
- `keyIndicators`: 主要指標リスト
- `strengths`: 強み（トップ 10 ランキング項目）
- `radarData`: レーダーチャート用データ
- `similarRegions`: 類似地域リスト

### 値オブジェクト

#### Rank（順位）

ランキングの順位を表現する値オブジェクト。

**具体例**:

- `1`: 1 位（東京都の総人口）
- `10`: 10 位（大阪府の総人口）
- `47`: 47 位（鳥取県の総人口）

**制約**:

- 1 以上の整数
- 最大 47（都道府県数）
- 同順位は同じ数値を使用

**用途**:

- 都道府県の順位表示
- ランキング表での位置表示
- 上位・下位の判定（例: 上位 10 位以内）
- 順位変動の計算

#### Percentile（パーセンタイル）

パーセンタイル値を表現する値オブジェクト。

**具体例**:

- `95.7`: 上位 5%（東京都の人口密度）
- `50.0`: 中央値（全国平均レベル）
- `15.3`: 下位 15%（鳥取県の人口密度）
- `99.2`: 上位 1%（非常に高い値）

**制約**:

- 0〜100 の実数
- 小数点以下 2 桁まで
- 0 は最小値、100 は最大値を意味

**用途**:

- 相対的な位置づけの表現
- 全国平均との比較
- 上位・下位の判定
- データの分布理解

## アーキテクチャ設計

### システムアーキテクチャ全体像

```
┌─────────────────────────────────────────┐
│     Presentation Layer                  │
│  (ランキングテーブル、地図、グラフ)      │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│     Service Layer                       │
│  - getRankingData(rankingKey, year)    │
│  - saveRankingData(rankingKey, data)   │
└──────────────────┬──────────────────────┘
                   │
       ┌───────────┼───────────┐
       │           │           │
┌──────▼──────┐ ┌─▼────┐ ┌────▼──────┐
│ eStat-API   │ │ R2   │ │ D1        │
│ (データ取得) │ │保存  │ │(キー管理) │
└─────────────┘ └──────┘ └───────────┘
```

### 3 層アーキテクチャ設計

Ranking ドメインは以下の 3 層アーキテクチャで設計されています：

#### Layer 1: Presentation Layer

**責務**: UI コンポーネント、ユーザーインタラクション

**コンポーネント**:

- コロプレス地図（ChoroplethMap）
- ランキングテーブル（RankingTable）
- 統計グラフ（StatisticsChart）

**ディレクトリ**: `src/components/organisms/visualization/`

#### Layer 2: Service Layer

**責務**: ビジネスロジック、データ取得調整

**主要関数**:

- `getRankingData()`: ランキングデータの取得
- `saveRankingData()`: ランキングデータの保存
- `calculateRanks()`: ランキング計算
- `calculatePercentile()`: パーセンタイル算出

**ディレクトリ**: `src/features/ranking/services/`

#### Layer 3: Infrastructure Layer

**責務**: データベース、ストレージ、外部 API

**コンポーネント**:

- D1 Database クライアント（R2 キー管理、視覚化情報）
- R2 Storage クライアント（ランキング値データ）
- e-Stat API クライアント（データ取得）

**ディレクトリ**: `src/infrastructure/`

### 実装方針

**基本方針**:

1. **eStat-API 固定**: マルチデータソース対応を削除し、eStat-API のみ対応
2. **シンプルな関数**: Adapter Pattern や Repository Pattern を使わず、純粋関数で実装
3. **R2 中心**: データは R2 に保存し、D1 は R2 キーとメタデータのみ管理
4. **視覚化情報の統合**: ranking_items テーブルに視覚化情報を含める

**Server Action による実装**:

- **Server Action を使用**: すべてのデータ取得は Server Action で実装
- **Route Handler は使用しない**: API Route Handler（`/api/...`）は使用せず、Server Action のみを使用
- **useSWR は使用しない**: サーバーサイドでのデータ取得のため、クライアントサイドの useSWR は使用しない
- **純粋関数で実装**: クラスではなく純粋関数を使用し、テストと再利用性を向上
- **キャッシュ**: Next.js 15 の`use cache`ディレクティブと R2 ストレージを併用

**理由**:

- **型安全性**: TypeScript で完全に型付けされた API
- **パフォーマンス**: サーバーサイドでの実行により、クライアントへの負荷を軽減
- **Next.js 15 の推奨パターン**: Next.js 15 では Server Action が推奨される実装方法
- **関数の純粋性**: 副作用を最小化し、テストが容易で、再利用性が高い
- **キャッシュの効率性**: R2 キャッシュにより API 呼び出しを削減
- **シンプルさ**: 複雑な抽象化を避け、保守性を向上

### データフロー

```
ランキングデータ取得リクエスト
    │
    ▼
Server Action (getRankingDataAction)
    │
    ├─► D1: ranking_itemsから視覚化情報取得
    │       │
    │       └─► R2キー生成: ranking/{rankingKey}/{areaLevel}/{timeCode}.json
    │

    │                   ├─► eStat-API呼び出し
    │                   ├─► データ整形・ランキング計算
    │                   └─► R2に保存
    │
    └─► 視覚化情報と統合して返却
```

## データソースとストレージ

### データベース設計

Ranking ドメインは以下のテーブルで構成されています。詳細は [データベース設計ドキュメント](../04_インフラ設計/01_データベース設計.md#31-ランキング管理) を参照してください。

#### 主要テーブル

| テーブル名     | 説明                                 | 保存先 |
| -------------- | ------------------------------------ | ------ |
| ranking_items  | ランキング項目定義（視覚化設定含む） | D1     |
| ranking_groups | ランキング項目のグループ定義         | D1     |

**注意**:

- `data_sources`と`data_source_metadata`テーブルは削除（eStat-API 固定）
- ランキング値データは R2 ストレージに保存（`ranking/{rankingKey}/{areaLevel}/{timeCode}.json`）

#### ranking_items テーブル構造

| カラム名        | 型      | 説明                                                 |
| --------------- | ------- | ---------------------------------------------------- |
| `ranking_key`   | TEXT    | ランキングキー（主キー）（例: `population-density`） |
| `label`         | TEXT    | 表示用ラベル                                         |
| `unit`          | TEXT    | 単位                                                 |
| `group_key`     | TEXT    | ランキンググループキー                               |
| `display_order` | INTEGER | 表示順序                                             |
| `is_active`     | BOOLEAN | 有効フラグ                                           |
| `visualization` | TEXT    | 視覚化情報（JSON 形式）                              |

#### visualization JSON 構造（例）

```json
{
  "colorScale": {
    "type": "sequential",
    "scheme": "Blues",
    "domain": [0, 100]
  },
  "chartType": "choropleth",
  "description": "人口密度の説明"
}
```

#### R2 キー管理の考え方

- R2 ストレージの`ranking/{rankingKey}/{areaLevel}/{timeCode}.json`を「データベース」として扱う
- D1 データベースは R2 キーの存在確認とメタデータ（視覚化情報）のみを管理
- 実際の値データは R2 から直接取得

**設計のポイント**:

1. **ranking_items → ranking_groups**: `group_key`で紐付け
2. **ranking_groups → subcategory**: `subcategory_id` で紐付け
3. **ranking_values**: 大量データのため **R2 に保存**（D1 はメタデータのみ）

### ストレージ分担の原則

#### D1 Database（R2 キー管理・視覚化情報）

- **用途**: ランキング項目定義（R2 キー管理）、視覚化情報設定、グループ定義
- **保存データ**: ranking_items（R2 キーと視覚化情報）, ranking_groups
- **特徴**: 低レイテンシ（10-50ms）、SQL クエリ可能、トランザクション対応
- **容量**: 小規模（数 MB 程度）

#### R2 Storage（値データ専用）

- **用途**: すべてのランキング値データ（都道府県・市区町村レベル）
- **保存データ**: ranking_values（JSON 形式）
- **特徴**: 大容量対応、低コスト、HTTP アクセス、レイテンシ 100-300ms
- **容量**: 大規模（数 GB〜数十 GB）

#### 保存ルール

```typescript
// ルール1: すべてのranking_valuesはR2に保存
await r2.put(
  `ranking/${rankingKey}/${areaLevel}/${timeCode}.json`,
  JSON.stringify(rankingData)
);

// ルール2: D1にはR2キーと視覚化情報のみ
await db.prepare(`
  INSERT INTO ranking_items (ranking_key, label, unit, group_key, visualization, ...)
  VALUES (?, ?, ?, ?, ?, ...)
`).bind(...).run();

// ルール3: R2をデータベースとして管理（D1はキー管理のみ）
```

### キャッシュ戦略

#### 1. R2 ストレージ（永続キャッシュ）

**目的**: 複数リクエスト間でのキャッシュ共有、永続性の確保

**詳細**: R2 ストレージの設計詳細については、[R2 ストレージ設計](../04_インフラ設計/02_R2ストレージ設計.md)を参照してください。

**実装**:

- R2 バケットにランキングデータを JSON 形式で保存
- キー設計: `ranking/{rankingKey}/{areaLevel}/{timeCode}.json`
- キャッシュヒット時は R2 から返却
- キャッシュミス時はデータソースから取得し、バックグラウンドで R2 に保存

**メリット**:

- サーバー再起動後もデータが保持される
- 複数の Worker インスタンス間でキャッシュを共有
- 外部 API 呼び出しを大幅に削減

#### 2. Next.js 15 の`use cache`ディレクティブ

**目的**: 関数レベルのキャッシュによる複数リクエスト間での効率化

**詳細**: R2 ストレージの設計詳細については、[R2 ストレージ設計](../04_インフラ設計/02_R2ストレージ設計.md)を参照してください。

**実装例**:

```typescript
// src/features/ranking/actions.ts
"use server";

import { getRankingData } from "./services/ranking-service";

export async function getRankingDataAction(rankingKey: string, year?: string) {
  "use cache"; // Next.js 15でのキャッシュ（複数リクエスト間で有効）

  return await getRankingData(rankingKey, year);
}
```

**特徴**:

- 関数レベルのキャッシュ
- 入力パラメータに基づく自動キャッシュ
- 複数リクエスト間で有効

## 主要機能

### 1. ランキングデータ取得

**機能**: ランキングキーと年度からランキングデータを取得

**主要関数**:

- `getRankingData()`: ランキングデータ取得（R2 キャッシュ優先）
- `getRankingItemsBySubcategory()`: サブカテゴリに属するランキング項目取得

**特徴**:

- R2 キャッシュによる高速化
- eStat-API 固定（シンプルな実装）
- Server Action による実装

### 2. ランキング計算

**機能**: 統計値からランキングとパーセンタイルを計算

**主要関数**:

- `calculateRanks()`: 統計値のランキング計算と順位付け

**特徴**:

- 同順位の適切な処理
- パーセンタイルの正確な計算
- 純粋関数による実装（同じ入力に対して同じ出力を返す）

### 3. ランキング項目管理

**機能**: ランキング項目の定義とグループ管理

**特徴**:

- ランキンググループによる項目のグループ化
- 表示順序の管理
- 可視化設定の管理

## 制約と前提条件

### 制約

1. **eStat-API 固定**: ランキングデータの取得は eStat-API のみ対応（マルチデータソース対応なし）
2. **ランキングキーの一意性**: ランキングキーは全体で一意である必要がある
3. **データサイズ**: ランキング値データは R2 ストレージを使用（D1 には保存しない）
4. **R2 キー管理**: R2 ストレージのキーをデータベースとして管理（D1 はメタデータのみ）

### 前提条件

1. **データベース**: D1 Database Seed が完了している必要がある
2. **R2 ストレージ**: ランキング値データ用の R2 バケットが設定されている必要がある
3. **eStat-API 設定**: eStat-API キーなどの環境変数設定が必要
