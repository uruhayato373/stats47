# Ranking 機能（ランキング表示・計算）

## 概要

Ranking 機能は、stats47 プロジェクトのアプリケーション層 / Presentation Domain として、ランキングデータの表示とランキング計算を担当します。estat-api ドメインが R2 ストレージに保存したランキングデータ（値のみ）を取得し、視覚化設定に基づいてランキング計算（順位・パーセンタイル）を実行して表示します。

**注意**: estat-api ドメインはコアドメインとして、データ探索・品質保証・データ取得 → 変換 →R2 保存を担当します。Ranking 機能はアプリケーション層として、データ表示とランキング計算を担当します。

### ビジネス価値

- **データドリブンな意思決定支援**: 統計データの分析により、ユーザーがデータに基づいた判断を行える
- **地域の特徴把握**: 地域プロファイル機能により、地域の強みや特徴を可視化
- **比較分析**: 複数地域間の比較により、相対的な位置づけを理解

### 最新の改善 (2025-01-27)

- **ランキンググループ機能**: 関連する複数のランキング項目をグループ化して管理
- **改善された UI**: グループ化により、サイドバーでの項目管理がより明確に

## 責務と主要概念

### 責務

1. **ランキング計算とデータ取得**

   - 各データソースドメイン（estat-api ドメインなど）から生データを取得し、ランキング形式（`RankingDataPoint[]`）に変換
   - 統計データからランキング計算
   - 順位とパーセンタイルの算出
   - **データソース非依存**: データソース固有のパラメータ管理は各データソースドメインが担当
   - ランキングキー（`rankingKey`）とデータソースタイプ識別（`data_source_id`）のみを参照
   - **ワークフロー**: estat-api ドメインでデータを探索・確認し、確認済みデータをランキング情報として登録・管理する流れを実現

2. **ランキング項目の管理**

   - ランキング項目のグループ管理
   - ランキング項目の定義とメタデータ管理
   - 表示順序と可視化設定の管理
   - **メタデータ管理**: `estat_api_metadata`テーブルへの書き込み・読み取り（ランキング項目登録時とランキング表示時）
     - estat-api ドメインで確認済みのパラメータ（`statsDataId`、`cdCat01`など）を`estat_api_metadata`テーブルに保存
     - ランキング表示時に`estat_api_metadata`テーブルからマッピング情報を読み取り、estat-api ドメインの生データ取得関数を呼び出す

3. **比較分析と可視化**

   - 地域間の比較分析
   - 時系列データの傾向分析
   - 統計サマリー生成
   - 地域プロファイル生成

### 主要概念

#### RankingKey（ランキングキー）

ランキング項目の一意識別子を表現する値オブジェクト。

**具体例**:

- `total-population`: 総人口
- `population-density`: 人口密度
- `gdp-per-capita`: 一人当たり GDP
- `unemployment-rate`: 失業率
- `elderly-population-ratio`: 高齢化率

**制約**:

- 空文字列は不可
- ケバブケース形式（小文字とハイフン）
- 一意である必要がある
- 最大 50 文字

**用途**:

- ランキング項目の識別
- URL パラメータとして使用（例: `/population/basic-population/ranking/total-population`）
- データベースのキーとして使用
- API レスポンスでの項目指定

## ドメインモデル

### 主要エンティティ

#### RankingGroup（ランキンググループ）

関連するランキング項目をグループ化して管理するエンティティ。複数の関連項目（例：「製造品出荷額」「製造品出荷額（事業所あたり）」）をまとめて表示する。

| 属性名          | 型               | 説明                                 | 制約     |
| --------------- | ---------------- | ------------------------------------ | -------- |
| `groupKey`      | `string`         | グループの一意識別子                 | 必須     |
| `subcategoryId` | `string`         | サブカテゴリ ID                      | 必須     |
| `name`          | `string`         | グループ名                           | 必須     |
| `description`   | `string \| null` | グループの説明                       | optional |
| `displayOrder`  | `number`         | サブカテゴリ内での表示順             | 必須     |
| `isCollapsed`   | `boolean`        | デフォルトで折り畳み表示するか       | 必須     |
| `items`         | `RankingItem[]`  | グループに属するランキング項目の配列 | optional |

#### RankingItem（ランキング項目）

統計指標の定義とメタデータを管理するエンティティ。

| 属性名          | 型        | 説明                    | 制約     |
| --------------- | --------- | ----------------------- | -------- |
| `rankingKey`    | `string`  | ランキングの一意識別子  | 必須     |
| `label`         | `string`  | 表示用ラベル            | 必須     |
| `unit`          | `string`  | 単位                    | 必須     |
| `groupKey`      | `string`  | ランキンググループキー  | 必須     |
| `displayOrder`  | `number`  | 表示順序                | 必須     |
| `visualization` | `object`  | 視覚化情報（JSON 形式） | optional |
| `isActive`      | `boolean` | 有効フラグ              | 必須     |

#### RankingValue（ランキング値）

地域ごとの統計値とランキング情報を管理するエンティティ。

| 属性名       | 型               | 説明           | 制約     |
| ------------ | ---------------- | -------------- | -------- |
| `areaCode`   | `string`         | 地域コード     | 必須     |
| `value`      | `number`         | 値             | 必須     |
| `rank`       | `number \| null` | 順位           | optional |
| `percentile` | `number \| null` | パーセンタイル | optional |
| `year`       | `string`         | 年度           | 必須     |
| `timeCode`   | `string`         | 時間コード     | 必須     |

#### RegionProfile（地域プロファイル）

地域の総合的な統計プロファイルを管理するエンティティ。

| 属性名           | 型              | 説明                             | 制約     |
| ---------------- | --------------- | -------------------------------- | -------- |
| `areaCode`       | `string`        | 地域コード                       | 必須     |
| `basicInfo`      | `object`        | 基本情報                         | 必須     |
| `keyIndicators`  | `RankingItem[]` | 主要指標リスト                   | optional |
| `strengths`      | `RankingItem[]` | 強み（トップ 10 ランキング項目） | optional |
| `radarData`      | `object`        | レーダーチャート用データ         | optional |
| `similarRegions` | `string[]`      | 類似地域リスト                   | optional |

### 値オブジェクト

#### Rank（順位）

ランキングの順位を表現する値オブジェクト。

**具体例**:

- `1`: 1 位（東京都の総人口）
- `10`: 10 位（大阪府の総人口）
- `47`: 47 位（鳥取県の総人口）

**制約**:

- 1 以上の整数
- 最大 47（都道府県数）
- 同順位は同じ数値を使用

**用途**:

- 都道府県の順位表示
- ランキング表での位置表示
- 上位・下位の判定（例: 上位 10 位以内）
- 順位変動の計算

#### Percentile（パーセンタイル）

パーセンタイル値を表現する値オブジェクト。

**具体例**:

- `95.7`: 上位 5%（東京都の人口密度）
- `50.0`: 中央値（全国平均レベル）
- `15.3`: 下位 15%（鳥取県の人口密度）
- `99.2`: 上位 1%（非常に高い値）

**制約**:

- 0〜100 の実数
- 小数点以下 2 桁まで
- 0 は最小値、100 は最大値を意味

**用途**:

- 相対的な位置づけの表現
- 全国平均との比較
- 上位・下位の判定
- データの分布理解

**コンポーネント**:

#### Layer 2: Service Layer

**責務**: ビジネスロジック、データ取得調整

**主要関数**:

- `getRankingData()`: ランキングデータの取得

**ディレクトリ**: `src/features/ranking/services/`

#### Layer 3: Infrastructure Layer

**責務**: データベース、ストレージ、外部 API

**コンポーネント**:

**ディレクトリ**: `src/infrastructure/`

### 実装方針

**基本方針**:

1. **アプリケーション層としての実装**: Ranking 機能はアプリケーション層 / Presentation Domain として実装。データ表示とランキング計算が主な責務
2. **データソース非依存**: estat-api ドメインが R2 ストレージに保存したデータを取得するだけで、データソース固有の処理は行わない
3. **アダプタ層不要**: estat-api ドメインが直接 R2 に保存するため、アダプタ層は不要
4. **R2 からデータ取得**: ランキングデータ（値のみ、順位なし）は R2 ストレージから取得
5. **D1 で視覚化設定管理**: 視覚化設定（`rankingDirection`、`mapColorScheme`など）は D1 データベースの`ranking_items`テーブルで管理
6. **ランキング計算**: 視覚化設定の`rankingDirection`に基づいてランキング計算（順位・パーセンタイル）を実行

**Server Action による実装**:

- **Server Action を使用**: すべてのデータ取得は Server Action で実装
- **Route Handler は使用しない**: API Route Handler（`/api/...`）は使用せず、Server Action のみを使用
- **useSWR は使用しない**: サーバーサイドでのデータ取得のため、クライアントサイドの useSWR は使用しない
- **純粋関数で実装**: クラスではなく純粋関数を使用し、テストと再利用性を向上
- **キャッシュ**: Next.js 15 の`use cache`ディレクティブと R2 ストレージを併用

**理由**:

- **型安全性**: TypeScript で完全に型付けされた API
- **パフォーマンス**: サーバーサイドでの実行により、クライアントへの負荷を軽減
- **Next.js 15 の推奨パターン**: Next.js 15 では Server Action が推奨される実装方法
- **関数の純粋性**: 副作用を最小化し、テストが容易で、再利用性が高い
- **キャッシュの効率性**: R2 キャッシュにより API 呼び出しを削減
- **シンプルさ**: 複雑な抽象化を避け、保守性を向上

### ワークフロー：データ確認からランキング表示まで

ランキング項目の登録から表示までのワークフローは以下の通りです：

```
1. データ探索・確認フェーズ（estat-apiドメイン）
   │
   ├─► e-Stat APIで統計表を検索（stats-list検索）
   ├─► メタ情報を確認・取得
   ├─► データをプレビューして品質確認
   └─► 適切なパラメータ（statsDataId、cdCat01など）を特定
   │
   ▼
2. ランキング登録フェーズ（ランキング機能）
   │
   ├─► ranking_itemsテーブルにランキング項目を登録
   │   - rankingKey, label, name, unit, data_source_id
   │   - 視覚化設定（rankingDirection, mapColorSchemeなど）
   │
   └─► estat_api_metadataテーブルにマッピング情報を保存
       - rankingKey + areaType → estat-apiパラメータ（JSON形式）
       - 確認済みのstatsDataId、cdCat01などを保存
   │
   ▼
3. データ生成・保存フェーズ（estat-apiドメイン）
   │
   ├─► estat_api_metadataテーブルからマッピング情報を取得
   ├─► fetchStatsData()で生データ取得
   ├─► ランキング形式に変換（値のみ、順位なし）
   └─► R2ストレージに保存
      - パス: ranking/{areaType}/{rankingKey}/{timeCode}.json
       - 構造: { values: [{areaCode, areaName, value}], metadata: {...} }
   │
   ▼
4. ランキング表示フェーズ（ランキング機能）
   │
   ├─► R2ストレージからランキングデータ（値のみ）を取得
   ├─► D1から視覚化設定（rankingDirectionなど）を取得
   ├─► ランキング計算（順位・パーセンタイル）を実行
   └─► UI表示（テーブル、地図、グラフ）
```

このワークフローにより、estat-api ドメインでデータを確認 ⇨estat-api ドメインが R2 に保存 ⇨ ランキング機能が R2 から取得してランキング計算 ⇨ 表示という流れが実現されます。

### データフロー

```
ランキング表示リクエスト (rankingKey, areaLevel, timeCode)
    │
    ▼
Server Action (getRankingDataAction)
    │
    ├─► D1: ranking_itemsから視覚化設定を取得
    │       │
    │       └─► rankingDirection, mapColorScheme, mapDivergingMidpointなど
    │
    ├─► R2ストレージからランキングデータ取得
    │       │
    │       └─► パス: ranking/{areaLevel}/{rankingKey}/{timeCode}.json
    │           - values: [{areaCode, areaName, value}]（値のみ、順位なし）
    │           - statistics: {min, max, mean, median}
    │           - metadata: {rankingKey, timeCode, unit}
    │
    ├─► ランキング計算（ランキング機能の責務）
    │       │
    │       ├─► 視覚化設定のrankingDirectionに基づいてソート
    │       │   - asc: 値の小さい順
    │       │   - desc: 値の大きい順
    │       │
    │       ├─► 順位（rank）の算出
    │       │   - 同順位の適切な処理
    │       │
    │       └─► パーセンタイル（percentile）の算出
    │
    └─► ランキングデータを返却（値 + 順位 + パーセンタイル + 視覚化設定）
```

**注意**: ランキングデータが R2 に存在しない場合は、estat-api ドメインがバックグラウンドでデータ取得 → 変換 →R2 保存を実行します。

#### アダプタ層パターン

ランキングドメインは標準インターフェース（`RankingDataAdapter`）を定義し、各データソースドメインがこのインターフェースを実装したアダプタを提供します。

**標準インターフェース定義**:

```typescript
interface RankingDataAdapter {
  fetchRankingData(
    rankingKey: string,
    options: { areaLevel: string; timeCode: string }
  ): Promise<RankingDataPoint[]>;
}
```

**アダプタ選択ロジック**:

- `ranking_service.ts`が`data_source_id`に基づいて適切なアダプタを選択
- `adapter-registry.ts`でアダプタの登録・取得を管理
- 各データソースアダプタ（`estat-api-adapter.ts`など）は各ドメインから提供

**各アダプタの責務**:

- **estat-api-adapter**:

  - `rankingKey`から estat-api パラメータ（`statsDataId`、`cdCat01`など）への変換（estat-api ドメイン内で管理）
  - e-Stat API 呼び出し
  - ランキング形式（`RankingDataPoint[]`）への変換

- **custom-data-adapter** (将来的に):
  - カスタムデータソース固有のパラメータ変換
  - データ取得・変換
  - ランキング形式への変換

**メリット**:

- ランキングドメインは標準インターフェースのみに依存（データソース固有の詳細を知らない）
- 各データソースドメインは独立性を保ちながらアダプタ実装を提供
- 新規データソース追加時はアダプタ実装を追加するだけで対応可能

#### ranking_items テーブル構造

| カラム名                 | 型      | 説明                                                 |
| ------------------------ | ------- | ---------------------------------------------------- |
| `ranking_key`            | TEXT    | ランキングキー（複合主キーの一部）（例: `population-density`） |
| `area_type`              | TEXT    | 地域タイプ（複合主キーの一部）（'prefecture' \| 'city' \| 'national'） |
| `label`                  | TEXT    | 表示用ラベル                                         |
| `name`                   | TEXT    | 項目名                                               |
| `unit`                   | TEXT    | 単位                                                 |
| `group_key`              | TEXT    | ランキンググループキー                               |
| `display_order_in_group` | INTEGER | グループ内での表示順序                               |
| `is_active`              | BOOLEAN | 有効フラグ                                           |
| **視覚化設定**           |         |                                                      |
| `ranking_direction`      | TEXT    | ランキング方向（"asc" / "desc"）                     |
| `map_color_scheme`       | TEXT    | 地図の色スキーム（例: "interpolateBlues"）           |
| `map_diverging_midpoint` | TEXT    | 色の分岐点（"zero", "mean", "median" または数値）    |
| `conversion_factor`      | REAL    | 表示用の変換係数                                     |
| `decimal_places`         | INTEGER | 小数点桁数                                           |

**注意**: 
- 視覚化設定は D1 データベースで管理し、ランキング計算時に参照されます。
- `ranking_items`テーブルはe-Stat API専用です（`data_source_id`は不要）。
- 複合主キー: `(ranking_key, area_type)`

#### R2 ストレージ構造

- R2 ストレージの`ranking/{rankingKey}/{areaLevel}/{timeCode}.json`にランキングデータ（値のみ、順位なし）を保存
- D1 データベースは視覚化設定（`ranking_items`テーブル）を管理
- ランキング計算（順位・パーセンタイル）はランキング機能で実行

**設計のポイント**:

1. **ranking_items → ranking_groups**: `group_key`で紐付け
2. **ranking_groups → subcategory**: `subcategory_id` で紐付け
3. **ranking_values**: 大量データのため **R2 に保存**（D1 はメタデータのみ）

### ストレージ分担の原則

#### D1 Database（視覚化設定管理）

- **用途**: ランキング項目定義、視覚化設定、グループ定義
- **保存データ**:
  - `ranking_items`: ランキング項目定義と視覚化設定（`rankingDirection`, `mapColorScheme`など）
  - `ranking_groups`: ランキンググループ定義
  - `estat_api_metadata`: estat-api 固有のパラメータ（ランキング機能が書き込み）
- **特徴**: 低レイテンシ（10-50ms）、SQL クエリ可能、トランザクション対応、検索・フィルタリングに適している
- **容量**: 小規模（数 MB 程度）

#### R2 Storage（ランキングデータ保存）

- **用途**: ランキングデータ（値のみ、順位なし）の保存
- **保存データ**: ランキングデータ（JSON 形式）
  - パス: `ranking/{areaLevel}/{rankingKey}/{timeCode}.json`
  - 構造: `{ values: [{areaCode, areaName, value}], statistics: {...}, metadata: {...} }`
- **特徴**: 大容量対応、低コスト、HTTP アクセス、レイテンシ 100-300ms
- **容量**: 大規模（数 GB〜数十 GB）

#### 責務分離

- **D1**: 視覚化設定（`rankingDirection`など）を管理し、ランキング計算時に参照される
- **R2**: estat-api ドメインがランキングデータ（値のみ）を保存、ランキング機能が読み取り
- **ランキング計算**: ランキング機能が R2 からデータ取得 →D1 から設定取得 → 計算 → 表示

### R2 ストレージ構造

#### ディレクトリ構造

```
ranking/
├── prefecture/
│   └── {ranking_key}/
│       ├── 2020.json
│       ├── 2021.json
│       └── 2023.json
└── city/
    └── {ranking_key}/
        ├── 2020.json
        ├── 2021.json
        └── 2023.json
```

#### JSON ファイル構造（値のみ、順位なし）

```json
{
  "values": [
    {
      "areaCode": "13000",
      "areaName": "東京都",
      "value": 6439.3
      // rank, percentile は含まない（ランキング機能で計算）
    },
    {
      "areaCode": "27000",
      "areaName": "大阪府",
      "value": 4646.5
    }
  ],
  "statistics": {
    "min": 64.5,
    "max": 6439.3,
    "mean": 338.7,
    "median": 124.5
  },
  "metadata": {
    "rankingKey": "population_density",
    "timeCode": "2023",
    "timeName": "2023年",
    "unit": "人/km²",
    "dataSourceId": "estat"
  }
}
```

**注意**: R2 に保存されるデータは値のみで、順位（rank）とパーセンタイル（percentile）は含まれません。これらはランキング機能が視覚化設定（`rankingDirection`）に基づいて計算します。

### R2→D1 同期設計（管理者向け）

#### 目的

- R2 内の実データ構造（areaType/top-level）を走査し、D1 の `ranking_items` を冪等的に作成/更新する。

#### スキャン対象とパス規約

- 対象ディレクトリ（新構造）
  - `ranking/prefecture/{rankingKey}/{timeCode}.json`
  - `ranking/city/{rankingKey}/{timeCode}.json`
- 互換対応（旧構造も当面サポート）
  - `ranking/{rankingKey}/{areaType}/{timeCode}.json`（併存時は新構造を優先）

#### 更新ポリシー（上書き: 指定フィールドのみ）

- 次のフィールドのみ更新（その他の表示系は非上書き）
  - `data_source_id` = "estat"
  - `unit` = R2 `metadata.unit`
  - `available_area_types` ∈ {prefecture, city}（存在検知）
  - `latest_time_code`（prefecture/city それぞれ最大）
  - `last_synced_at`（同期実行時刻）

#### 同期フロー（概要）

1. `ranking/prefecture/` と `ranking/city/` を listObjects でページング列挙
2. パスから `{areaType, rankingKey, timeCode}` を抽出
3. 必要最小限の取得で `metadata.unit` を抽出（全体読み込みは回避）
4. `rankingKey` 単位に集約し、上記フィールドを決定
5. D1 `ranking_items` に対して UPSERT（ON CONFLICT → 指定フィールドのみ UPDATE）
6. 統計ログ（新規/更新/スキップ/警告）と dry-run を提供

#### 運用と安全性

- 管理者専用の実行手段（Server Action）
- 既定は dry-run。`apply=true` 指定時のみ書き込み
- レート制御・リトライ・続きトークン対応

### キャッシュ戦略

#### 1. R2 ストレージ（永続キャッシュ）

**目的**: 複数リクエスト間でのキャッシュ共有、永続性の確保

**詳細**: R2 ストレージの設計詳細については、[R2 ストレージ設計](../04_インフラ設計/02_R2ストレージ設計.md)を参照してください。

**実装**:

- R2 バケットにランキングデータを JSON 形式で保存
- キー設計: `ranking/{areaLevel}/{rankingKey}/{timeCode}.json`
- キャッシュヒット時は R2 から返却
- キャッシュミス時はデータソースから取得し、バックグラウンドで R2 に保存

**メリット**:

- サーバー再起動後もデータが保持される
- 複数の Worker インスタンス間でキャッシュを共有
- 外部 API 呼び出しを大幅に削減

#### 2. Next.js 15 の`use cache`ディレクティブ

**目的**: 関数レベルのキャッシュによる複数リクエスト間での効率化

**詳細**: R2 ストレージの設計詳細については、[R2 ストレージ設計](../04_インフラ設計/02_R2ストレージ設計.md)を参照してください。

**実装例**:

```typescript
// src/features/ranking/actions.ts
"use server";

import { getRankingData } from "./services/ranking-service";

export async function getRankingDataAction(rankingKey: string, year?: string) {
  "use cache"; // Next.js 15でのキャッシュ（複数リクエスト間で有効）

  return await getRankingData(rankingKey, year);
}
```

**特徴**:

- 関数レベルのキャッシュ
- 入力パラメータに基づく自動キャッシュ
- 複数リクエスト間で有効

## 主要機能

### 1. ランキングデータ取得と計算

**機能**: ランキングキーと年度からランキングデータを取得し、ランキング計算を実行

**主要関数**:

- `getRankingData()`: R2 からランキングデータ（値のみ）を取得 →D1 から視覚化設定取得 → ランキング計算 → 返却
- `getRankingItemsBySubcategory()`: サブカテゴリに属するランキング項目取得

**特徴**:

- R2 ストレージからデータ取得（estat-api ドメインが保存済み）
- D1 から視覚化設定（`rankingDirection`など）を取得
- ランキング計算（順位・パーセンタイル）を実行
- Server Action による実装
- 純粋関数による実装

### 2. ランキング計算

**機能**: 統計値からランキングとパーセンタイルを計算

**主要関数**:

- `calculateRanks()`: 統計値のランキング計算と順位付け
- `calculatePercentile()`: パーセンタイル値の算出

**特徴**:

- 同順位の適切な処理
- パーセンタイルの正確な計算
- 純粋関数による実装（同じ入力に対して同じ出力を返す）

### 3. ランキング項目管理

**機能**: ランキング項目の定義とグループ管理

**特徴**:

- ランキンググループによる項目のグループ化
- 表示順序の管理
- 可視化設定の管理

## 制約と前提条件

### 制約

1. **R2 ストレージ依存**: ランキングデータは estat-api ドメインが R2 ストレージに保存済みである必要がある
2. **ランキングキーの一意性**: ランキングキーは全体で一意である必要がある
3. **データサイズ**: ランキング値データは R2 ストレージを使用（D1 には保存しない）
4. **視覚化設定**: ランキング計算には D1 の`ranking_items`テーブルの視覚化設定（`rankingDirection`など）が必要

### 前提条件

1. **データベース**: D1 Database Seed が完了している必要がある
2. **R2 ストレージ**: ランキング値データ用の R2 バケットが設定されている必要がある
3. **データソース設定**: 各データソースドメインに必要な環境変数設定（例: estat-api の場合、`NEXT_PUBLIC_ESTAT_APP_ID`）
