# DDD ドメイン分類

## 概要

Domain-Driven Design（DDD）におけるドメイン分類について、stats47 プロジェクトでの実装方針と各ドメインの役割を説明します。

## DDD ドメイン分類の概要

DDD では、システム全体を以下の 3 つのドメインタイプに分類します：

1. **コアドメイン（Core Domain）**: ビジネスの競争優位性の源泉
2. **支援ドメイン（Supporting Domain）**: プロジェクト固有のビジネスロジック
3. **汎用ドメイン（Generic Domain）**: 技術的関心事と汎用機能

## コアドメイン（Core Domain）

### 定義と特徴

**コアドメイン**は、プロジェクトの競争優位性に直結する最も重要なドメインです。

**特徴**:

- **ビジネス価値の源泉**: プロジェクトの存在意義を決定
- **投資優先度最高**: 最も多くのリソースと時間を投入
- **独自性**: 他社との差別化要因
- **複雑性**: 最も複雑なビジネスロジックを含む

### stats47 のコアドメイン

#### Ranking（ランキング）

- **責務**: ランキング生成、ソート、フィルタリング
- **競争優位性**: 多様な統計指標によるランキング機能
- **独自性**: 地域別統計の専門的なランキング表示
- **実装**: `docs/01_技術設計/03_ドメイン設計/01_コアドメイン/01_Ranking.md`

#### TimeSeries（時系列分析）

- **責務**: 時系列データの分析、トレンド抽出、予測
- **競争優位性**: 統計データの時間軸分析機能
- **独自性**: 年度別・月別の変化分析に特化
- **実装**: `docs/01_技術設計/03_ドメイン設計/01_コアドメイン/02_TimeSeries.md`

#### Comparison（比較分析）

- **責務**: 地域間比較、相関分析、散布図生成
- **競争優位性**: 複数地域の統計データ比較機能
- **独自性**: 統計的有意性を考慮した比較分析
- **実装**: `docs/01_技術設計/03_ドメイン設計/01_コアドメイン/03_Comparison.md`

### 設計方針

- **品質重視**: 最高品質のコードと設計
- **テスト充実**: 包括的なテストカバレッジ
- **ドキュメント詳細**: 詳細な設計ドキュメント
- **継続的改善**: 定期的なリファクタリング

## 支援ドメイン（Supporting Domain）

### 定義と特徴

**支援ドメイン**は、プロジェクト固有のビジネスロジックを含み、コアドメインの実現を支援するドメインです。

**特徴**:

- **プロジェクト固有**: stats47 特有の要件に基づく設計
- **ドメイン知識必要**: ビジネスコンテキストの理解が必要
- **カスタマイズ性**: プロジェクト要件に最適化
- **コアドメイン支援**: コアドメインの機能実現に不可欠

### stats47 の支援ドメイン

#### Area（地域管理）

- **責務**: 地域コード管理、地域情報の提供
- **プロジェクト固有性**: 日本の行政区域コード体系の理解が必要
- **ビジネスロジック**: 都道府県・市区町村の階層関係管理
- **実装**: `docs/01_技術設計/03_ドメイン設計/02_支援ドメイン/01_Area.md`

#### Taxonomy（分類管理）

- **責務**: カテゴリ・タグ・フィルタリング管理
- **プロジェクト固有性**: stats47 固有の統計分類体系
- **ビジネスロジック**: 統計データの分類と検索ロジック
- **実装**: `docs/01_技術設計/03_ドメイン設計/02_支援ドメイン/02_Taxonomy.md`

#### EstatAPI（e-Stat API 統合）

- **責務**: e-Stat API との統合、データ変換
- **プロジェクト固有性**: 統計データ取得の固有ロジック
- **ビジネスロジック**: API レスポンスの正規化と変換
- **実装**: `docs/01_技術設計/03_ドメイン設計/02_支援ドメイン/03_EstatAPI.md`

#### Geoshape（地理データ管理）

- **責務**: 歴史的行政区域データの管理
- **プロジェクト固有性**: 日本の行政区域の歴史的変遷の知識が必要
- **ビジネスロジック**: TopoJSON データの解像度管理と年度別データ処理
- **実装**: `docs/01_技術設計/03_ドメイン設計/02_支援ドメイン/04_Geoshape.md`

#### Search（検索）

- **責務**: 統計データの検索、フィルタリング、ソート
- **プロジェクト固有性**: stats47 固有の統計データ検索ロジック
- **ビジネスロジック**: 複合条件検索、検索結果の最適化
- **実装**: `docs/01_技術設計/03_ドメイン設計/02_支援ドメイン/05_Search.md`

### 設計方針

- **プロジェクト最適化**: stats47 の要件に特化した設計
- **ビジネスロジック明示**: ドメイン知識をコードに反映
- **適度な品質**: コアドメインよりは簡潔だが、保守性を重視
- **要件変更対応**: ビジネス要件の変更に柔軟に対応

## 汎用ドメイン（Generic Domain）

### 定義と特徴

**汎用ドメイン**は、技術的関心事を扱い、他プロジェクトでも再利用可能なドメインです。

**特徴**:

- **汎用的**: どんなプロジェクトでも使える技術機能
- **ドメイン知識不要**: 技術的な知識のみで理解可能
- **高い再利用性**: そのまま他プロジェクトに移植可能
- **インフラストラクチャ**: 技術的な基盤機能

### stats47 の汎用ドメイン

#### Authentication（認証・認可）

- **責務**: ユーザー認証、権限管理
- **汎用性**: 標準的な認証・認可パターン
- **技術的関心事**: JWT、セッション管理、権限チェック
- **実装**: `docs/01_技術設計/03_ドメイン設計/03_汎用ドメイン/01_Authentication.md`

#### Cache（キャッシュ管理）

- **責務**: R2/D1 連携によるキャッシュ管理
- **汎用性**: どんなデータでも同じキャッシュロジック
- **技術的関心事**: TTL 管理、キャッシュ無効化、統計収集
- **実装**: `docs/01_技術設計/03_ドメイン設計/03_汎用ドメイン/04_Cache.md`

#### Content（コンテンツ管理）

- **責務**: MDX/Markdown コンテンツの管理とレンダリング
- **汎用性**: どんな React コンポーネントでも表示可能
- **技術的関心事**: MDX 処理、構文解析、検索機能
- **実装**: `docs/01_技術設計/03_ドメイン設計/03_汎用ドメイン/05_Content.md`

### 設計方針

- **汎用性重視**: 標準パターンの採用
- **外部ライブラリ活用**: 既存の優れたライブラリを積極活用
- **シンプル設計**: 過度な抽象化を避ける
- **ドキュメント簡潔**: 技術仕様中心の簡潔なドキュメント

## ドメイン分類の判断基準

### フローチャート

```
1. プロジェクトの競争優位性に直結するか？
   YES → コアドメイン
   NO → 次へ

2. プロジェクト固有のビジネスロジックを含むか？
   YES → 支援ドメイン
   NO → 次へ

3. 技術的関心事で汎用的に使えるか？
   YES → 汎用ドメイン
```

### 具体例による判断

#### コアドメインの例

- **ランキング分析**: 都道府県別ランキング生成、stats47 の核心機能
- **時系列可視化**: 年度別トレンド分析、変化率計算
- **比較分析**: 地域間比較、相関分析、散布図生成

#### 支援ドメインの例

- **e-Stat API 統合**: プロジェクト固有（統計データ特有の処理）
- **地域コード管理**: 日本の行政区域コード体系の理解が必要
- **歴史的行政区域**: 日本の行政区域の歴史的変遷の知識が必要
- **検索機能**: stats47 固有の統計データ検索ロジック

#### 汎用ドメインの例

- **キャッシュ管理**: 汎用的（どんなデータでも同じロジック）
- **認証機能**: 標準的な認証・認可パターン
- **コンテンツ管理**: MDX/Markdown の汎用的な処理

### ドメイン間の依存関係

```
コアドメイン
    ↓ (依存)
支援ドメイン
    ↓ (依存)
汎用ドメイン
```

**原則**: 上位ドメインから下位ドメインへの一方向依存のみ

## ベストプラクティス

### 1. ドメイン間依存の管理

- **一方向依存**: 上位 → 下位のみ許可
- **循環依存禁止**: ドメイン間の循環参照は避ける
- **インターフェース分離**: 必要最小限のインターフェースのみ公開

### 2. 各ドメインの責務明確化

- **単一責務**: 各ドメインは明確な単一の責務を持つ
- **境界明確**: ドメイン間の境界を明確に定義
- **変更の局所化**: 変更の影響範囲をドメイン内に限定

### 3. 汎用ドメインの活用

- **外部ライブラリ検討**: 既存の優れたライブラリを積極活用
- **標準パターン採用**: 業界標準のパターンとアーキテクチャを採用
- **過度な抽象化回避**: 必要以上に複雑にしない

### 4. 支援ドメインの最適化

- **プロジェクト要件特化**: stats47 の要件に最適化
- **ビジネスロジック明示**: ドメイン知識をコードに反映
- **柔軟性確保**: ビジネス要件の変更に柔軟に対応

### 5. コアドメインの品質管理

- **最高品質**: 最高品質のコードと設計を維持
- **包括的テスト**: 高いテストカバレッジを維持
- **継続的改善**: 定期的なリファクタリングと改善

## 開発順序の推奨

### 基本原則: ボトムアップアプローチ

DDD では依存関係の方向性に従い、**下位ドメインから上位ドメインへ**開発を進めることを推奨します。

```
1. 汎用ドメイン（Generic Domain）
   ↓
2. 支援ドメイン（Supporting Domain）
   ↓
3. コアドメイン（Core Domain）
```

### 理由

1. **依存関係の方向性**: 上位ドメインは下位ドメインに依存する
2. **リスクの低減**: 汎用ドメインは既存ライブラリ活用でリスク低
3. **投資配分の最適化**: 汎用ドメイン(10%)→ 支援ドメイン(30%)→ コアドメイン(60%)

### stats47 での具体的な開発順序

#### Phase 1: 汎用ドメイン（基盤構築）

1. **Authentication（認証）** - NextAuth.js 統合
2. **Cache（キャッシュ管理）** - R2/D1 連携
3. **Content（コンテンツ管理）** - MDX/Markdown 処理

#### Phase 2: 支援ドメイン（ビジネス基盤）

1. **Area（地域管理）** - 地域コード、都道府県・市区町村
2. **Taxonomy（分類管理）** - カテゴリ、タグ
3. **EstatAPI（e-Stat 統合）** - API 統合、データ変換
4. **Geoshape（地理データ）** - TopoJSON、歴史的行政区域
5. **Search（検索）** - 統計データ検索機能

#### Phase 3: コアドメイン（競争優位性）

1. **Ranking（ランキング）** - ランキング生成、ソート
2. **TimeSeries（時系列分析）** - トレンド分析、変化率計算
3. **Comparison（比較分析）** - 地域間比較、相関分析

### MVP 開発の場合

**垂直スライス**アプローチも有効：

1. 汎用ドメイン（最小限）- Authentication, Cache
2. 支援ドメイン（最小限）- Area, EstatAPI
3. コアドメイン（1 機能のみ）- Ranking

## 投資配分の指針

### リソース配分比率

```
コアドメイン: 60%
支援ドメイン: 30%
汎用ドメイン: 10%
```

### 各ドメインの投資方針

- **コアドメイン**: 最高品質、継続的改善、包括的テスト
- **支援ドメイン**: プロジェクト最適化、ビジネスロジック明示
- **汎用ドメイン**: 標準化、外部ライブラリ活用、シンプル設計
