# プロジェクト構造

## 概要

stats47 プロジェクトのディレクトリ構造と設計思想について説明します。

## プロジェクトルート構造

```
stats47/
├── src/                    # ソースコード
├── docs/                   # ドキュメント
├── data/                   # データファイル
```

## ルートグループ設計

### ルートグループの利点

- **レイアウトの分離**: 各機能ごとに異なるレイアウトを適用
- **メタデータの個別化**: 機能別の SEO 設定
- **エラーハンドリングの分離**: 機能別のエラー表示
- **アクセス制御の分離**: 管理画面と一般ページの認証制御を分離
- **セキュリティの向上**: e-Stat API 関連ページを管理画面配下に配置

### 各ルートグループの役割

#### (public) - 公開ページグループ

- 一般ユーザー向けの公開ページ
- SEO 最適化されたレイアウト
- 認証不要

#### (stats) - 統計データグループ

- 統計データ表示ページ
- データ可視化に特化したレイアウト
- パフォーマンス最適化

#### admin - 管理画面

- 管理者向けの管理機能
- 認証必須
- 開発ツールを含む

## URL 構造設計

### 統計データページ

- `/{category}/{subcategory}` - サブカテゴリトップ（統合ビュー）
- `/{category}/{subcategory}/ranking/{rankingKey}` - ランキング詳細
- `/{category}/{subcategory}/area/{areaCode}` - 地域別ダッシュボード

### 管理画面

- `/admin` - 管理画面トップ
- `/admin/geoshape-cache` - 地図データキャッシュ管理
- `/admin/visualization-settings` - 可視化設定管理
- `/admin/dev-tools/estat-api/*` - e-Stat API 関連ツール（認証必須）

## アクセス制御設計

### 認証戦略

#### 管理画面の認証

```typescript
// admin/middleware.ts
export async function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl;

  // 管理画面全体の認証チェック
  if (pathname.startsWith("/admin")) {
    const token = await getToken({ req: request });

    if (!token) {
      return NextResponse.redirect(new URL("/login", request.url));
    }

    // 開発ツールは開発環境のみアクセス可能
    if (pathname.startsWith("/admin/dev-tools/")) {
      const env = process.env.NODE_ENV;
      if (env === "production" && !token.isDeveloper) {
        return NextResponse.redirect(new URL("/admin", request.url));
      }
    }
  }

  return NextResponse.next();
}
```

#### 環境別アクセス制御

| 環境            | 一般ページ | 管理画面 | 開発ツール |
| --------------- | ---------- | -------- | ---------- |
| **development** | 自由       | 認証必須 | 認証必須   |
| **staging**     | 自由       | 認証必須 | 認証必須   |
| **production**  | 自由       | 認証必須 | 開発者のみ |

## 設計原則

### 1. 関心の分離

- 各ディレクトリは明確な責務を持つ
- ビジネスロジックと UI ロジックを分離

### 2. スケーラビリティ

- 新しい機能の追加が容易な構造
- ドメイン駆動設計に基づく拡張性

### 3. 保守性

- 一貫した命名規則
- 明確な依存関係

### 4. パフォーマンス

- 必要な部分のみをロード
- 適切なキャッシング戦略
