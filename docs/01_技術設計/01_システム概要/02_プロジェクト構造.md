## アトミックデザインの基本概念

### 5 層のコンポーネント階層

アトミックデザインでは、UI コンポーネントを 5 つの階層に分類して管理します。

#### Atoms（原子）

- **責務**: 最小単位の UI 部品
- **特徴**: 単一の責務を持ち、他のコンポーネントに依存しない
- **具体例**: Button, Input, Label, Icon, Badge

#### Molecules（分子）

- **責務**: 2-3 個の Atoms を組み合わせた小さな機能単位
- **特徴**: 特定の機能を提供するが、まだビジネスロジックは持たない
- **具体例**: SearchBox（Input + Button）、FormField（Label + Input）、NavigationItem

#### Organisms（有機体）

- **責務**: 複雑な機能を提供する大きなコンポーネント
- **特徴**: ビジネスロジックを含み、特定のドメイン機能を実現
- **具体例**: RankingTable, ChoroplethMap, DataDashboard, CategoryNavigation

#### Templates（テンプレート）

- **責務**: ページのレイアウト構造を定義
- **特徴**: コンテンツの配置とレイアウトのみを管理
- **具体例**: DashboardLayout, ArticleLayout, AdminLayout

#### Pages（ページ）

- **責務**: 実際のページコンポーネント
- **特徴**: Templates に具体的なコンテンツを配置
- **具体例**: RankingPage, AreaComparisonPage, AdminDashboardPage

## ビジネスロジック層のドメイン別整理

### レイヤー別の責務

**domain/** - ドメイン層（ビジネスロジック）

- 各ドメインのビジネスルールとロジックを実装
- エンティティ、値オブジェクト、ドメインサービスを含む
- 外部への依存を持たない純粋なビジネスロジック

**infrastructure/** - インフラストラクチャ層

- データベース、外部 API、キャッシュ等の技術的詳細を実装
- ドメイン層のインターフェースを実装
- 外部システムとの連携を担当

**shared/** - 共有カーネル

- 複数のドメインで共通して使用される機能
- ユーティリティ関数、定数、型定義
- ドメイン非依存の汎用機能

**application/** - アプリケーション層

- ユースケースの実装
- ドメイン層とインフラ層を組み合わせてアプリケーション機能を実現
- トランザクション管理、認証等の横断的関心事を処理

### ドメイン別の責務

#### コアドメイン（Core Domain）

**analytics/** - 統計分析ドメイン

- ランキング計算、時系列分析、相関分析、比較分析
- ビジネスの中核価値を提供する最も重要なドメイン

**visualization/** - 可視化ドメイン

- チャート生成、地図処理、ダッシュボード構築
- 統計データの視覚表現を担当

#### 支援ドメイン（Supporting Domain）

**data-integration/** - データ統合ドメイン

- e-Stat API 連携、キャッシュ管理、データアダプター
- 外部データソースとの統合を担当

**area/** - 地域管理ドメイン

- 都道府県・市区町村管理、階層構造管理
- 日本の行政区画データを管理

**category/** - カテゴリ管理ドメイン

- カテゴリナビゲーション、階層構造管理
- 統計データの分類体系を管理

#### 汎用ドメイン（Generic Domain）

**auth/** - 認証ドメイン

- セッション管理、権限管理
- ユーザー認証とアクセス制御を担当

## ルートグループ設計

### ルートグループの利点

- **レイアウトの分離**: 各機能ごとに異なるレイアウトを適用
- **メタデータの個別化**: 機能別の SEO 設定
- **エラーハンドリングの分離**: 機能別のエラー表示
- **アクセス制御の分離**: 管理画面と一般ページの認証制御を分離
- **セキュリティの向上**: e-Stat API 関連ページを管理画面配下に配置

### 各ルートグループの役割

#### (public) - 公開ページグループ

- 一般ユーザー向けの公開ページ
- SEO 最適化されたレイアウト
- 認証不要

#### (stats) - 統計データグループ

- 統計データ表示ページ
- データ可視化に特化したレイアウト
- パフォーマンス最適化

#### admin - 管理画面

- 管理者向けの管理機能
- 認証必須
- 開発ツールを含む

## URL 構造設計

### 統計データページ

- `/{category}/{subcategory}` - サブカテゴリトップ（統合ビュー）
- `/{category}/{subcategory}/ranking/{rankingKey}` - ランキング詳細
- `/{category}/{subcategory}/area/{areaCode}` - 地域別ダッシュボード

### 管理画面

- `/admin` - 管理画面トップ
- `/admin/geoshape-cache` - 地図データキャッシュ管理
- `/admin/visualization-settings` - 可視化設定管理
- `/admin/dev-tools/estat-api/*` - e-Stat API 関連ツール（認証必須）

## アクセス制御設計

### 認証戦略

#### 管理画面の認証

```typescript
// admin/middleware.ts
export async function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl;

  // 管理画面全体の認証チェック
  if (pathname.startsWith("/admin")) {
    const token = await getToken({ req: request });

    if (!token) {
      return NextResponse.redirect(new URL("/login", request.url));
    }

    // 開発ツールは開発環境のみアクセス可能
    if (pathname.startsWith("/admin/dev-tools/")) {
      const env = process.env.NODE_ENV;
      if (env === "production" && !token.isDeveloper) {
        return NextResponse.redirect(new URL("/admin", request.url));
      }
    }
  }

  return NextResponse.next();
}
```

#### 環境別アクセス制御

| 環境            | 一般ページ | 管理画面 | 開発ツール |
| --------------- | ---------- | -------- | ---------- |
| **development** | 自由       | 認証必須 | 認証必須   |
| **staging**     | 自由       | 認証必須 | 認証必須   |
| **production**  | 自由       | 認証必須 | 開発者のみ |

## DDD とアトミックデザインの統合

### なぜ Organisms だけをドメイン別に整理するのか

**Organisms がビジネスロジックを持つため**

- Organisms は特定のドメイン機能を実現する複雑なコンポーネント
- ビジネスルールやドメイン知識が含まれる
- ドメイン境界を明確にすることで保守性が向上

**ドメイン境界が明確になるため**

- 各ドメインの責務が視覚的に分かりやすくなる
- 変更時の影響範囲が予測しやすくなる
- チーム開発時に担当範囲が明確になる

### Atoms/Molecules は汎用的に保つ理由

**再利用性を最大化**

- 複数のドメインで共通して使用される UI 部品
- ドメイン固有のロジックを含まない純粋な UI コンポーネント
- 一箇所で管理することで一貫性を保つ

**ドメイン非依存の UI 部品として機能**

- ビジネスロジックを持たないため、どのドメインからも利用可能
- テストが容易で、独立性が高い
- UI ライブラリとしての役割を果たす

### Templates/Pages の役割

**Templates は汎用的なレイアウト**

- ページ構造の定義のみを担当
- ドメイン固有のロジックは含まない
- 複数のページで再利用可能

**Pages はドメイン固有の組み立て**

- 特定のページの実装を担当
- ドメイン固有のコンポーネントを組み合わせる
- ルーティングと密接に関連

## Organisms のドメイン別整理

### ドメイン駆動設計に基づく分類

organisms コンポーネントは、DDD（ドメイン駆動設計）の原則に基づいてドメイン別に整理されています。

#### コアドメイン（Core Domain）

**analytics/** - 統計分析ドメイン

- ビジネスの中核価値を提供する最も重要なドメイン
- ランキング、時系列分析、相関分析、比較分析を担当

**visualization/** - 可視化ドメイン

- 統計データの視覚表現を担当
- チャート、地図、可視化制御を管理

#### 支援ドメイン（Supporting Domain）

**data-integration/** - データ統合ドメイン

- 外部データソース（e-Stat API 等）との統合を担当
- データ取得、変換、キャッシュ管理を管理

**area/** - 地域管理ドメイン

- 日本の行政区画の階層構造と地理データを管理
- 都道府県・市区町村の管理を担当

**category/** - カテゴリ管理ドメイン

- 統計データの分類体系を管理
- カテゴリナビゲーションを担当

#### 汎用ドメイン（Generic Domain）

**auth/** - 認証ドメイン

- ユーザー認証とアクセス制御を担当
- セッション管理、権限制御を管理

**layout/** - レイアウト（汎用）

- 全体的なレイアウトコンポーネント
- ヘッダー、サイドバー等の共通 UI

### ドメイン別整理の利点

1. **責務の明確化**: 各ドメインの役割が明確
2. **保守性の向上**: 変更が局所化される
3. **拡張性**: 新しいドメインの追加が容易
4. **チーム開発**: ドメインごとに担当を分けやすい
5. **テスタビリティ**: ドメイン単位でのテストが可能

## コンポーネント配置の指針

### 配置基準

**Atoms: 単一の責務を持つ UI 部品**

- 最小単位の UI 要素
- プロパティのみで動作を制御
- ビジネスロジックを含まない

**Molecules: 2-3 個の Atoms の組み合わせ**

- 小さな機能単位を提供
- 状態管理は最小限
- 再利用可能な機能ブロック

**Organisms: 特定のドメイン機能を提供**

- 複雑なビジネスロジックを含む
- ドメイン別にディレクトリを整理
- 特定のユースケースを実現

**Templates: ページ構造の定義**

- レイアウトのみを管理
- コンテンツの配置を定義
- ドメイン非依存の構造

**Pages: 具体的なページ実装**

- 特定のページの完全な実装
- Templates と Organisms を組み合わせ
- ルーティングと密接に関連

### ドメイン横断的なコンポーネントの扱い

**layout ディレクトリに配置**

- 複数のドメインで使用される共通レイアウト
- ヘッダー、サイドバー、フッター等
- ドメイン固有のロジックは含まない

**複数ドメインで使用される場合は共通化を検討**

- 類似機能が複数ドメインに存在する場合
- 共通インターフェースを定義
- ドメイン固有の実装は分離

## 設計原則

### 1. 関心の分離

- 各ディレクトリは明確な責務を持つ
- ビジネスロジックと UI ロジックを分離
- ドメイン境界を明確に保つ
- レイヤー間の依存関係を適切に管理

### 2. スケーラビリティ

- 新しい機能の追加が容易な構造
- ドメイン駆動設計に基づく拡張性
- ドメイン別の独立した開発が可能

### 3. 保守性

- 一貫した命名規則
- 明確な依存関係
- ドメイン間の依存関係を最小化
- 変更の影響範囲が局所化される

### 4. パフォーマンス

- 必要な部分のみをロード
- 適切なキャッシング戦略
- ドメイン別の最適化が可能

### 5. テスタビリティ

- ドメイン単位での独立したテスト
- モックやスタブの作成が容易
- ビジネスロジックの単体テストが可能
