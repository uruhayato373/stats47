# 11. データ基盤設計

stats47 のデータ基盤の **唯一の真実源**。ローカル D1 (source of truth) → R2 snapshot → 本番配信のフローを定義する。

> **重要**: 2026-04-29 に Phase 10 で **リモート D1 を完全撤廃** し、本ドキュメントの設計に切り替わっている。古い設計（リモート D1 + R2 ハイブリッド）の記述が他ドキュメントに残っていれば、**本ドキュメントが優先**。

## 0. 設計原則

| 原則 | 内容 |
|---|---|
| **D1 はローカルのみ** | リモート D1 は存在しない。復活させない |
| **本番配信は R2 のみ** | `apps/web` (Cloudflare Workers) は D1 binding を持たず、R2 snapshot のみを fetch |
| **同期は単方向** | ローカル D1 → exporter → R2 snapshot の 1 方向のみ |
| **source of truth はローカル D1 (8.4 GB)** | バックアップは git (ignore) と R2 snapshot で間接担保 |

## 2. データの流れ

### 2.1 取り込み (Ingest)

外部 API → ローカル D1 への投入。

| 入力 | 出力テーブル | スキル |
|---|---|---|
| e-Stat メタ情報 | `ranking_items`, `surveys`, `categories` | `/register-ranking` |
| e-Stat 統計値 (47 都道府県) | `ranking_data` ← 旧テーブル, `observations` (Phase D 進行中) | `/populate-all-rankings` |
| e-Stat 統計値 (市区町村) | `ranking_data` ← city 限定 | `/populate-city-rankings` |
| MLIT 国土数値情報 | `.local/r2/gis/` 直接 | `/fetch-mlit-ksj` |
| ブログ MD 記事 | `articles`, `tags`, `article_tags` | `/import-blog-articles` |

### 2.2 加工 (Transform)

ローカル D1 内での集計・AI 生成。

| 入力 | 出力テーブル | スキル |
|---|---|---|
| `ranking_items` + `ranking_data` | `correlation_analysis` (一時) | `/compute-correlation` |
| `ranking_items` + `ranking_data` | `ranking_ai_content` | `/generate-ai-content` |

### 2.3 出力 (Export)

ローカル D1 → R2 snapshot への変換。

| ローカル D1 テーブル | R2 path | exporter |
|---|---|---|
| `articles` + `article_tags` + `tags` | `snapshots/blog/all.json` | `apps/web/scripts/export-blog-snapshot.ts` |
| `ports` | `snapshots/ports/all.json` | `apps/web/scripts/export-port-statistics-snapshot.ts` |
| `port_statistics` | `snapshots/port-statistics/{years.json, by-year/, by-port/}` | 同上 |
| `fishing_ports` | `snapshots/fishing-ports/all.json` | `apps/web/scripts/export-fishing-ports-snapshot.ts` |
| `page_components` | `snapshots/page-components/all.json` | `apps/web/scripts/export-page-components-snapshot.ts` |
| `affiliate_ads` | `snapshots/affiliate-ads/all.json` | `apps/web/scripts/export-affiliate-ads-snapshot.ts` |
| `ranking_page_cards` | `snapshots/ranking-page-cards/all.json` | `apps/web/scripts/export-ranking-page-cards-snapshot.ts` |
| `ranking_items` + `surveys` + `categories` (3 keys) | `snapshots/{ranking-items,surveys,categories}/all.json` | `packages/ranking/src/scripts/export-master-snapshots.ts` |
| `observations` (旧 `ranking_data`) | `snapshots/ranking-values/<rk>/<areaType>/<year>.json` (29,642 partitions) | `packages/ranking/src/scripts/export-ranking-values-snapshots.ts` |
| `ranking_ai_content` | `snapshots/ai-content/all.json` | `packages/ai-content/src/scripts/export-snapshot.ts` |
| `area_profile_rankings` | `snapshots/area-profile/all.json` | `packages/area-profile/src/scripts/export-snapshot.ts` |
| `correlation_analysis` | `snapshots/correlation/{top-pairs.json, stats.json, by-ranking-key/}` | `packages/correlation/src/scripts/export-snapshot.ts` |

**`/export-snapshots`** で全 11 exporter を順次実行する。

### 2.4 配信 (Serve)

R2 → page.tsx (in-memory cache 経由)。

| R2 path | reader | 利用 page |
|---|---|---|
| `snapshots/blog/all.json` | `packages/blog/repositories/blog-snapshot-reader.ts` | blog list, tag |
| `snapshots/ranking-items/all.json` 他 master 系 | `packages/ranking/src/repositories/master/*` | 全 ranking ページ |
| `snapshots/ranking-values/<rk>/<areaType>/<year>.json` | `packages/ranking/src/repositories/ranking-value/read-ranking-values-from-r2.ts` | ranking 詳細 |
| `snapshots/correlation/by-ranking-key/<rk>.json` | `packages/correlation/src/repositories/read-correlation-by-key.ts` | correlation widget |
| `snapshots/ai-content/all.json` | `packages/ai-content/src/repositories/read-ranking-ai-content-snapshot.ts` | ranking 詳細 (AI 解説) |
| `snapshots/area-profile/all.json` | `packages/area-profile/src/repositories/*` | area profile |
| `snapshots/page-components/all.json` | `packages/stat-charts/services/page-components-snapshot.ts` | dashboard |
| その他 | 各 package の reader | — |

**配信時の cache 戦略**: 各 reader が module-scope に `let cached: Snapshot | null = null` を持ち、初回 fetch で温める。30 日 stale check（blog のみ実装、要拡張）。

## 3. 各層の責任

### 3.1 ローカル D1 (8.4 GB)

- **保存場所**: `.local/d1/v3/d1/miniflare-D1DatabaseObject/baffe56c6b0173e34c63a5333065bcdb6642a01b4c2cfecd70ad3607b00c9972.sqlite`
- **アクセス**: `better-sqlite3` (Node.js)、ハードコードされたパスを `__dirname` 相対で解決
- **特徴**:
  - source of truth（外部 API から取り込んだデータ + 加工結果が全部入っている）
  - git 管理されていない（巨大すぎるため）
  - 単一 PC 運用前提（複数 PC で同期する仕組みは無い）

### 3.2 exporter

- **役割**: ローカル D1 → R2 snapshot 変換
- **配置**: `packages/<domain>/src/scripts/export-*.ts` または `apps/web/scripts/export-*-snapshot.ts`
- **実行**: 個別 `npx tsx -r ./packages/ranking/src/scripts/setup-cli.js <script>` または一括 `/export-snapshots`
- **共通課題**:
  - boilerplate（.env 読込、D1 接続、R2 client init）が exporter ごとに重複（後の PR で `packages/exporter-framework/` に統合予定）
  - partition 戦略が exporter ごとに異なる（all.json / by-year / triple-partition / per-key）

### 3.3 R2 snapshot (`stats47` bucket)

- **公開 URL**: `https://storage.stats47.jp`
- **総サイズ**: 約 900 MB（94% は `ranking-values/` の 29,642 partitions）
- **更新方針**: exporter で生成 → ローカル R2 (`.local/r2/`) → `/push-r2` で本番 R2 に同期
- **ロールバック**: R2 自身に version は無い。緊急時はローカル R2 から `/push-r2 --force` で過去版を巻き戻すか、再 export
- **災害復旧**: ローカル D1 (8.4 GB) が source of truth なので、R2 が消えても再 export 可

### 3.4 R2 reader

- **配置**: `packages/<domain>/repositories/*` または `packages/<domain>/services/*`
- **共通実装**: `packages/r2-storage/src/lib/operations/*.ts` の `fetchFromR2AsJson()` を呼ぶ
- **cache**: module-scope の `let cached: T | null = null` パターン（一度温まると Workers のライフタイム内で再利用）
- **build 時**: `NEXT_PHASE=phase-production-build` で fetch を skip し、空配列フォールバック（ISR で初回フェッチ）

## 6. 運用フロー（ユースケース別）

### 6.1 新規ランキング登録

```bash
/register-ranking            # ローカル D1 に ranking_items + surveys 追加
/populate-all-rankings       # ranking_data に全年度の値を投入
/generate-ai-content         # AI 解説をローカル D1 に保存（オプション）
/compute-correlation       # 相関分析を再計算（オプション）
/export-snapshots            # ローカル R2 に全 snapshot 出力
/push-r2                     # 本番 R2 に push
/deploy                      # Cloudflare Workers デプロイ
```

### 6.2 既存ランキングの値更新

```bash
/populate-all-rankings --key <ranking-key>   # 部分更新
/export-snapshots --only ranking-values
/push-r2 --prefix snapshots/ranking-values/<key>/
/deploy
```

### 6.3 ブログ記事追加

```bash
# 記事 MD を .local/r2/blog/articles/<slug>.md に置く
/import-blog-articles               # ローカル D1 に articles 追加
/export-snapshots --only blog
/push-r2 --prefix snapshots/blog/
/deploy
```

### 6.4 緊急ロールバック

R2 snapshot は version 管理されていないため、以下のいずれか:

1. **ローカル R2 から戻す**: `.local/r2/` に過去 export がある場合 `/push-r2 --force` で巻き戻し
2. **再 export**: ローカル D1 から再生成 (`/export-snapshots`) → `/push-r2`
3. **Cloudflare Workers ロールバック**: 直前 deploy へ戻す（`wrangler deployments rollback`）

## 8. トラブルシューティング

| 症状 | 原因候補 | 対処 |
|---|---|---|
| `Error: Cannot find database` | ローカル D1 のパスが間違い | `.local/d1/v3/d1/miniflare-D1DatabaseObject/<hash>.sqlite` を確認、CLAUDE.md の絶対パスを参照 |
| 本番に古いデータが出ている | R2 snapshot が更新されていない | `/export-snapshots` → `/push-r2` → `/deploy` の流れを再確認 |
| R2 fetch が build 時にハング | NEXT_PHASE skip が漏れている | 該当 reader に `if (process.env.NEXT_PHASE === 'phase-production-build') return null;` を追加 |
| exporter が OOM で落ちる | 全件メモリ展開 | partition 単位（key / year / area）で chunk 処理に書き換え |
| R2 push が 429/504 | rate limit / timeout | `parallelism=1` まで下げて retry、または S3 API ↔ wrangler CLI フォールバック |
| `wrangler d1 execute --remote` が `Couldn't find DB` | リモート D1 を叩こうとしている（撤廃済） | コードを書き換え、ローカル D1 + R2 export に変更 |
