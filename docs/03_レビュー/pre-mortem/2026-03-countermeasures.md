# Pre-Mortem 対策: 失敗を防ぐ / 失敗しても大丈夫にする

**作成日**: 2026-03-10
**対象**: `docs/03_レビュー/pre-mortem/pre-mortem-2026-03.md` の各シナリオ

---

## 方針

対策は2層で設計する:

- **予防策（Prevention）**: そもそも起きないようにする
- **耐性策（Resilience）**: 起きても致命傷にならないようにする

---

## 1. 「配信恐怖症」が治らない → 人間を投稿ループから外す

### 予防策: 自動投稿パイプラインを作る

心理的ブロックの正体は「投稿ボタンを押す瞬間の判断」。判断を排除すれば恐怖もなくなる。

**具体的な仕組み**:
- `.local/r2/sns/` に生成済みコンテンツがある → GitHub Actions の scheduled workflow で X API v2 に自動投稿
- 人間がやることは「スキルでコンテンツを生成する」まで。投稿は機械がやる
- 投稿前レビューが不安なら、「生成後24時間バッファ → 自動投稿」にする。24時間以内に削除すれば投稿されない

**最小実装**:
1. X API v2 の POST endpoint を叩くスクリプト（Bearer Token は取得済み）
2. GitHub Actions cron（毎日9時）で `.local/r2/sns/x/` の未投稿ファイルを1件投稿
3. 投稿後に `/mark-sns-posted` 相当の処理を自動実行

### 耐性策: SNS が全滅しても死なない構造にする

- SNS はあくまで「SEO の加速装置」と位置づける
- SEO 単体で月3,000PV を達成できるコンテンツ量（69記事 + 1,188ランキングページ）は既にある
- SNS ゼロでも、ブログ記事の検索流入だけで最低限の PV は確保できる
- **判断基準**: 3ヶ月後に SEO のみで月1,000PV 未満なら、SNS 自動投稿を最優先で実装する

## 2. SEO が崩壊する → 既に対策済み。監視を追加する

### 現状確認（調査で判明）

- **301リダイレクトは実装済み**（`apps/web/src/middleware.ts`）
  - 旧URL → 新URL のマッピングが middleware で処理されている
  - 削除済みランキングキーには 410 Gone を返している
- **サイトマップは動的生成**（`apps/web/src/app/sitemap.ts`）
  - ランキングページ・ブログ記事が自動的にサイトマップに含まれる

### 予防策: 月次 SEO ヘルスチェック

- 毎月1日に `/fetch-gsc-data` を実行し、以下を確認:
  - インデックス済みページ数が増加しているか
  - 旧URLパターンのインプレッションが減少（→新URLに移行）しているか
  - CTR が 1% 以上を維持しているか
- 異常があれば `docs/90_課題管理/` に課題として記録

### 耐性策: Google 依存を下げる

- YouTube からの流入経路を維持する（動画説明欄にサイトリンク）
- note.com からの被リンクを獲得する（無料記事にサイト内リンクを埋め込む）
- Google AI Overview に引用されるための構造化データ（JSON-LD）は既に実装済み

---

## 3. 収益が1年間ゼロ円 → 最小収益を早期に確認する

### 予防策: 今週中に「収益が発生するか」を確認する

AdSense は既に設置済み、GA4 も稼働中。やることは:

1. **AdSense 管理画面を開いて、現在の収益を確認する**（5分）
   - ゼロでなければ「収益は発生する」と証明される
   - ゼロなら広告表示を確認（ブロックされていないか）
2. **GA4 でPV数を確認する**（5分）
   - `/fetch-ga4-data` スキルで直近30日のPVを取得
   - 現在のベースラインを知る

### 予防策: アフィリエイトの外部依存を排除する

- A8.net の承認を待つのではなく、**Amazon Associates を先に設定する**（承認が早い）
- 記事内に Amazon リンクを1本入れるだけで「アフィリエイト収益が発生するか」のテストになる
- ふるさと納税系は後回し。まず「仕組みが動くか」の検証を最短で行う

### 耐性策: 収益ゼロでも続けられる構造にする

- 月額固定費は約1,000円。年間12,000円。これは「趣味の予算」として許容する
- 収益目標を「月1,000円」に下げる。AdSense だけなら月3,000-10,000PV で達成可能
- **撤退ライン**: 12ヶ月後に月PV 500未満 かつ 収益ゼロ → ドメイン更新しない
- **継続ライン**: 月PV 1,000以上 or 月収益100円以上 → 趣味として継続

## 4. 技術的に楽しいことだけやる → 時間配分ルールを設ける

### 予防策: 「配信デー」を固定する

- **毎週月曜日は「配信・計測の日」**。コードを1行も書かない
  - GA4/GSC のデータを見る
  - SNS投稿キューを確認する
  - AdSense 収益を確認する
  - ブログ記事を1本公開する
- 残りの曜日は好きなだけ技術をやってよい

### 予防策: 週次レビューに「配信実績」を必須項目にする

`/weekly-review` の出力に以下を含める:
- 今週の SNS 投稿数（0 なら赤字で警告）
- 今週のブログ公開数
- GA4 PV 数（前週比）
- AdSense 収益（前週比）

### 耐性策: 技術投資がムダにならないようにする

- 作った自動化スキルは「将来の時間を節約する投資」として正当化できる
- ただし **新しいスキルを作る前に、既存スキルの利用回数を確認する**
- 利用回数0回のスキルが10個以上あるなら、新規スキル開発を凍結する

## 5. PC とインフラのトラブル → 定期メンテナンスと分散

### 予防策: 月次メンテナンスルーティン

- **毎月1日**:
  - `.git` サイズ確認。1GB 超えたら `/reset-git-history` 実行
  - `.local/r2/` サイズ確認。不要な生成済み動画を削除
  - `npm audit` で脆弱性チェック
  - Node.js バージョン確認（EOL まで6ヶ月以内ならアップグレード計画）

### 予防策: 複数 PC 対応を活かす

- CLAUDE.md にテーブルオーナーシップルールが既にある
- **自宅 PC が壊れても、もう1台で最低限の運用（記事公開・デプロイ）ができる状態を維持**
- リモート D1 + リモート R2 が single source of truth なので、ローカル環境は再構築可能

### 耐性策: Cloudflare の課金リスクに備える

- D1 の行数制限に近づいたら、古い年度の ranking_data をアーカイブ（最新3年分のみ保持）
- 無料枠の上限を把握し、超過する3ヶ月前にアラートを設定

## 6. 一人プロジェクトの限界 → 外部の目を入れる

### 予防策: 最小限の外部コミットメント

- **月1回、note.com に進捗記事を書く**（無料記事でよい）
  - 「stats47 開発日記」として、やったこと・PV数・収益を公開する
  - 読者がゼロでも、「公開する」行為自体が外部コミットメントになる
  - 3ヶ月後に読者がつけば、サボれないプレッシャーが生まれる

### 予防策: 「最低限これだけやれば OK」を定義する

週あたりの最低ラインを決め、それ以上は自分を褒める:
- ブログ記事 1本公開 or SNS 3投稿 → 最低ライン達成
- GA4 を1回確認 → 最低ライン達成
- **2つとも達成すれば、その週は合格**。コードを書かなくてもよい

### 耐性策: 1ヶ月休んでもサイトが死なないようにする

- 静的サイト（Cloudflare Pages）なので、サーバーダウンのリスクはほぼゼロ
- e-Stat データは年次更新なので、1ヶ月データが古くても致命的ではない
- **ブログ記事に「最終更新日」を表示し、古い記事でも価値がある形にする**

## 7. モチベーション崩壊 → 小さな成功体験を設計する

### 予防策: マイルストーンを細かく刻む

「月30,000PV」のような遠い目標ではなく:
- 月 500PV → 「認知フェーズ突破」
- 月 1,000PV → 「定着フェーズ開始」
- 月 3,000PV → 「収益化ライン到達」
- 初の AdSense 収益 → 「1円でも発生したら祝う」
- 初の SNS リアクション → 「いいね1つで祝う」

### 予防策: 「やめるライン」を事前に決めておく

モチベーションが尽きてからダラダラ続けるのが最悪。事前に撤退条件を決める:
- **6ヶ月後（2026年9月）に月PV 300未満** → SEO戦略を根本的に見直す
- **12ヶ月後（2027年3月）に月PV 500未満 かつ 収益ゼロ** → 凍結（サイトは残すが開発停止）
- **撤退 ≠ 失敗**。「検証が終わった」と定義する

### 耐性策: 「とどラン」や AI Overview に負けても大丈夫なポジションを取る

- stats47 の独自価値は「元県庁職員の解説 × モダンUI × e-Stat 直接参照」
- 競合がUI を真似ても、ブログ記事のドメイン知識は真似できない
- AI Overview に引用される側になれば、トラフィック減少を相殺できる（JSON-LD は実装済み）

## 8. 外部環境の変化 → 依存先を分散する

### 予防策: e-Stat API 変更への備え

- `estat_metainfo` テーブルにメタ情報をキャッシュ済み（62件）
- API が変わっても、既存データは DB に保持されているので即死はしない
- **年1回、e-Stat API のバージョン情報を確認する習慣をつける**

### 予防策: Google 依存を下げるトラフィック源

| チャネル | 現状 | 目標 |
|---|---|---|
| Google 検索 | 主力（未計測） | 全体の50%以下に |
| YouTube | 55本、628平均再生 | 説明欄リンクからの流入を計測 |
| note.com | 0本 | 月1本の無料記事で被リンク獲得 |
| X (Twitter) | 0投稿 | 自動投稿で週3件、プロフィールにサイトリンク |
| 直接流入 | 不明 | ブックマーク・リピーター層を育てる |

### 耐性策: 最悪 Cloudflare を離れても動くようにする

- Next.js ベースなので Vercel / Netlify への移行は技術的に可能
- D1（SQLite）は標準SQLなので、他のホスティングでも動く
- R2 のデータはローカル（`.local/r2/`）に全てコピーがある

## 9. データ品質の問題 → レビュープロセスを入れる

### 予防策: AI 生成コンテンツのサンプルチェック

- 全件チェックは非現実的。**10件に1件をランダムに目視確認する**
- チェック項目:
  - 数値がランキングデータと一致しているか
  - 因果推論をしていないか（プロンプトで禁止済み）
  - 括弧付き数値列挙をしていないか（今回のプロンプト改善で対策済み）
- `/expert-review` スキルが既にあるので、気になる記事に対して実行する

### 予防策: データ鮮度の自動管理

- ブログ記事のフロントマターに `yearCode` を記録し、2年以上古い記事にバナーを表示:
  「この記事のデータは YYYY 年度のものです。最新データはランキングページをご覧ください。」
- ランキングページのデータは e-Stat 更新時にバッチ更新（`/populate-all-rankings`）

### 耐性策: 指摘されたら即修正できる体制

- ブログ記事はローカル R2 → デプロイで即更新可能（ビルド不要のCMS的構造）
- AI コンテンツは DB の該当行を更新するだけで反映される
- 「ご指摘ありがとうございます、修正しました」を24時間以内に返せれば信頼は毀損しない

## 10. 完璧主義とスコープクリープ → 「やらないことリスト」を作る

### 予防策: 今後6ヶ月間やらないこと

| やらないこと | 理由 |
|---|---|
| 市区町村ランキングの拡充 | 16件で十分。データ構造が違いすぎてROIが悪い |
| Google Earth Studio 動画 | 個人では非現実的。YouTube Shorts で十分 |
| note 有料記事 | 無料記事を5本公開してから検討する |
| 新しい Remotion テンプレート | 既存5種で十分。配信が先 |
| 新しい CLI スキル開発 | 既存58個の利用率を上げるのが先 |
| packages/ の新規追加 | 17個で十分。既存パッケージ内で対応する |

### 予防策: 「やること」は3つに絞る

**2026年3月〜6月の最優先事項（これだけやればよい）**:

1. **配信**: ブログ記事を週1本公開 + SNS 自動投稿を設定
2. **計測**: GA4 の PV を毎週確認し、記録する
3. **収益**: AdSense 収益を確認し、月次で記録する

この3つが回っていれば、技術開発は好きなだけやってよい。

## まとめ: 最悪のシナリオでも「大丈夫」な理由

| 最悪のケース | なぜ大丈夫か |
|---|---|
| SNS を1件も投稿できない | SEO 単体でも月1,000PV は到達可能なコンテンツ量がある |
| PV が月500以下のまま | 固定費は月1,000円。趣味として1年は持つ |
| 収益がゼロのまま | 元県庁職員のドメイン知識はポートフォリオとしての価値がある |
| モチベーションが消えて2ヶ月放置 | 静的サイトなのでサーバーは落ちない。再開はいつでもできる |
| 競合に追い抜かれる | ブログ69本 + ランキング1,188ページのコンテンツ資産は残る |
| PC が壊れる | リモート D1 + R2 にデータがあり、新PCで再構築可能 |

**このプロジェクトが本当に死ぬ唯一のパターン**: ドメインを更新せずに放置すること。それ以外はすべて回復可能。
