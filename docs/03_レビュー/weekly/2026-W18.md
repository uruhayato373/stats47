## 批判的レビュー

> **「技術的に楽しいだけでは？」**
> Must 3 件は全て戦略直結:
> - LCP 改善は PV・SEO・CVR すべてに波及する基盤問題（1 ヶ月放置したが、PSI 計測復活で初めて可視化）
> - EXP-001 measure は Sprint 2 完了判定の必須手続
> - X 週 3 投稿は Sprint 2 完了条件そのもの
>
> 技術的に楽しいだけの要素は Must に入っていない。Should-2（browser-use 自動化）は「楽しいだけ」寄りの部分があるが、手動依存タスクの 3 週連続停滞を断つ構造対策なので Should 妥当。
>
> **「先週と同じ失敗を繰り返してないか？」**
> - `/update-sns-metrics` X 分 → W16-W17 で未達継続 → W18 で「browser-use 自動化」として Should に入れた（Must から外してでも構造解決を優先）
> - `/discover-trends-all` → W16-W17 で未達継続 → W18 で Could に維持。Must/Should に入れない理由: X 10 件予約済・年収トレンド消化済で鮮度勝負ではない。ただし 3 週連続未達なら W19 で Must 昇格
> - 記事 -39 本影響調査 → W15-W17 の 3 週繰り越しが W17 で部分着地（GONE_BLOG_SLUGS 機構）。W18 では「網羅的突合」を明示的に Must に入れるべきか検討したが、GONE 機構の稼働を 1 週間観察してから判断するため敢えて入れない
>
> **「今週やらないと機会損失になるものは？」**
> - EXP-001 FINAL 観測は 2026-04-28 火曜が期限 → Must-2 で確保
> - LCP 改善は今週から着手しないと、Sprint 2 完了時点で「計測はしたが何もしていない」になる → Must-1 で確保
> - GW 連動は 4/29 からの連休直前が勝負 → Could-2 に配置（余力があれば実行、なければ Sprint 3 でゆとりを持って設計）
> - Sprint 2 完了レビュー下準備は W18 中にやらないと W19 冒頭が詰まる → Should-3 で金曜期限
>
> **工数妥当性**: Must は S + S + L = L 相当、Should は M + M + S + S = L 相当、Could は M + S = M 相当。合計で週前半集中 + 後半の余裕ありの構成。LCP 改善（Must-1 の L）が重いが、EXP-002 として Phase 分割すれば 1 週間で 1 Phase は妥当。
