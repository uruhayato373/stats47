# ドキュメント管理ルール

## 概要

stats47 プロジェクトのドキュメント管理に関する統一的なルールとガイドラインを定義します。

プロジェクトの品質向上、保守性の確保、チーム内での知識共有を目的とします。

## ドキュメント構造

### ディレクトリ命名規則

#### 1. トップレベルディレクトリ

```
docs/
├── 00_プロジェクト管理/     # プロジェクト概要・要件
├── 01_開発ガイド/          # 開発ガイド・規約
├── 02_デザインシステム/     # デザインシステム
├── 03_技術設計/            # 技術設計・アーキテクチャ
├── 04_ドメイン設計/        # ドメイン別ドキュメント
├── 05_テスト戦略/          # テスト戦略
├── 06_運用保守/            # 運用・保守
├── 07_コンテンツ計画/       # コンテンツ計画
├── 08_リソース/            # リソース・素材
├── 99_課題管理/            # 既知の課題・調査
└── INDEX.md               # ドキュメントインデックス
```

### ディレクトリの役割と使い分け

#### 全体構成の原則

- **プロジェクト管理** → **開発実践** → **運用** → **リソース** の流れ
- **HOW（実装）** と **WHAT/WHY（設計）** の明確な分離
- 開発者が日常的に参照する順序でディレクトリを配置

#### 00\_プロジェクト管理

**目的**: プロジェクト全体の方向性、要件、戦略を管理

**対象読者**: プロジェクトマネージャー、ステークホルダー、アーキテクト

**内容の特徴**:

- プロジェクト憲章・概要・収益化戦略
- 機能要件・非機能要件・MVP 定義
- 実装計画・ロードマップ・統合計画
- 技術選定評価・戦略決定

**配置するドキュメント**:

- `プロジェクト憲章.md`、`プロジェクト概要.md`
- `機能要件.md`、`非機能要件.md`、`MVP定義.md`
- `実装計画/`、`技術選定評価_2025.md`

**更新頻度**: 中程度（プロジェクトフェーズ変更時）

**他ディレクトリとの関係**: 全ディレクトリの基盤となる情報を提供

#### 01\_開発ガイド（実装 HOW）

**目的**: 開発者が**どうやって実装するか**を指南

**対象読者**: コードを実装する開発者、日常的な開発作業を行うエンジニア

**内容の特徴**:

- コード行レベルの具体例と実装パターン
- コーディング規約（TypeScript、React、Tailwind CSS）
- 開発ワークフローとツールの使い方
- トラブルシューティングと解決策

**配置するドキュメント**:

- `コーディング規約.md` - 日々のコーディングで参照する規約
- `大規模実装ガイド.md` - 実装手順とパターン
- `パフォーマンス最適化.md`、`データフェッチ.md`
- `SEOガイド.md`、`開発環境チューニング.md`

**更新頻度**: 高い（プロジェクト進行中に随時更新）

**他ディレクトリとの関係**:

- `02_デザインシステム/` の UI 原則を実装レベルで具体化
- `03_技術設計/` の設計決定を実装方法に変換
- `04_ドメイン設計/` の実装ガイドと連携

**実装例**:

```typescript
// ✅ 推奨
interface UserProps {
  id: string;
  name: string;
}

// ❌ 避ける
const user: any = { ... };
```

#### 02\_デザインシステム（UI/UX 標準）

**目的**: UI/UX の原則、スタイル、コンポーネント設計を基準化

**対象読者**: フロントエンド開発者、デザイナー、UI 実装者

**内容の特徴**:

- デザイン原則とスタイルガイド
- コンポーネント設計原則と Atomic Design
- レスポンシブデザインガイド
- アクセシビリティ基準

**配置するドキュメント**:

- `スタイルガイド.md`、`アイコンシステム.md`
- `タイポグラフィシステム.md`、`レスポンシブデザインガイド.md`
- `Atomic Design実装.md`、`shadcn-ui統合ガイド.md`
- `Storybook運用ガイド.md`

**更新頻度**: 中程度（デザインシステム更新時）

**他ディレクトリとの関係**:

- `01_開発ガイド/` で実装方法を具体化
- `04_ドメイン設計/` の各ドメインで UI 実装パターンを適用

#### 03\_技術設計（設計 WHAT/WHY）

**目的**: システム全体の**設計決定と理由**を記録

**対象読者**: アーキテクト、技術選定を行うリードエンジニア、プロジェクト全体像を理解したい人

**内容の特徴**:

- システム全体のアーキテクチャ
- 技術選定の理由と背景（ADR）
- データベース設計と環境戦略
- API 設計の全体像と型定義の設計思想

**配置するドキュメント**:

- `システムアーキテクチャ.md` - プロジェクト全体の構造
- `設計仕様書.md` - 詳細な設計とデータフロー
- `001_Cloudflare採用理由.md` など ADR 群
- `DDDドメイン分類設計書.md` - ドメイン駆動設計の方針
- `TypeScript型ガイド.md`、`データフェッチ設計.md`

**更新頻度**: 低い（設計フェーズで確定、大きな変更時のみ更新）

**他ディレクトリとの関係**:

- `01_開発ガイド/` で実装方法に変換
- `04_ドメイン設計/` の各ドメインに設計方針を適用

**設計例**:

```
データベースアーキテクチャ
├── 環境別データ取得戦略
│   ├── mock: JSON ファイル
│   ├── development: ローカル D1
│   ├── staging: リモート D1
│   └── production: リモート D1
```

#### 04\_ドメイン設計（DDD 実践）

**目的**: 各ドメインの責務・仕様・実装・テスト・リファクタリングを一元管理

**対象読者**: ドメインエキスパート、機能開発者、テストエンジニア

**内容の特徴**:

- ドメイン駆動設計（DDD）に基づく構造
- 各ドメインの 4 層構造（仕様/実装/リファクタリング/テスト）
- ドメイン間の境界と依存関係の管理
- ビジネスロジックの詳細設計

**配置するドキュメント**:

- `e-Stat API/`、`ダッシュボード/`、`可視化/` など各ドメイン
- 各ドメイン内に `仕様/`、`実装/`、`リファクタリング/`、`テスト/` の 4 層
- `共有/` - 複数ドメインで使用する共通要素

**更新頻度**: 中程度（機能開発・リファクタリング時）

**他ディレクトリとの関係**:

- `03_技術設計/` の設計方針を各ドメインに適用
- `01_開発ガイド/` の実装パターンをドメイン固有で具体化
- `02_デザインシステム/` の UI 原則をドメイン別に実装

#### 05\_テスト戦略

**目的**: 単体/統合/E2E/ビジュアル等のテスト方針・運用を定義

**対象読者**: テストエンジニア、開発者、QA 担当者

**内容の特徴**:

- テスト戦略の全体像と各層の役割
- テストケース設計とテストデータ管理
- テスト自動化と CI/CD 連携
- 品質メトリクスと分析

**配置するドキュメント**:

- `ユニットテストガイド.md` - 単体テストの実装方法
- 各ドメインの `テスト/` ディレクトリと連携

**更新頻度**: 中程度（テスト戦略変更・新機能追加時）

**他ディレクトリとの関係**:

- `04_ドメイン設計/` の各ドメインでテストを実装
- `01_開発ガイド/` の実装パターンにテストを組み込み

#### 06\_運用保守

**対象読者**: インフラエンジニア、DevOps 担当者、運用担当者

**内容の特徴**:

- デプロイ手順と環境管理

- 監視・アラート設定

**配置するドキュメント**:

- `デプロイ手順.md`、`バックアップリストア.md`
- `データベーストラブルシューティング.md`

**更新頻度**: 低い（運用手順変更時）

**他ディレクトリとの関係**:

- `03_技術設計/` のインフラ設計を運用レベルで具体化

#### 07\_コンテンツ計画

**目的**: 記事やダッシュボードのコンテンツ企画・編集計画

**対象読者**: コンテンツ企画者、編集者、マーケター

**内容の特徴**:

- ランキング記事の企画・構成
- ダッシュボード記事の設計
- ブログ記事の計画・編集方針
- SEO・アクセス解析を考慮したコンテンツ戦略

**配置するドキュメント**:

- `01_ランキング記事/`、`02_ダッシュボード/`、`03_ブログ記事/`

**更新頻度**: 高い（コンテンツ企画・更新時）

**他ディレクトリとの関係**:

- `04_ドメイン設計/` の各機能をコンテンツ化

#### 08\_リソース

**目的**: 画像・アイコンなどの素材、外部リンク、参考資料の集約

**対象読者**: デザイナー、開発者、全チームメンバー

#### ディレクトリ間の相互関係

各ディレクトリは以下の依存関係と参照パターンを持ちます：

```
00_プロジェクト管理 (基盤)
    ↓ 参照・影響
03_技術設計 (設計決定) ←→ 01_開発ガイド (実装方法)
    ↓ 適用              ↓ 実装
04_ドメイン設計 ←→ 02_デザインシステム
    ↓ テスト
05_テスト戦略

06_運用保守 ← 03_技術設計 (インフラ設計)
07_コンテンツ計画 ← 04_ドメイン設計 (機能のコンテンツ化)
08_リソース ← 02_デザインシステム (素材管理)
99_課題管理 ← 全ディレクトリ (横断的課題管理)
```

**主要な参照パターン**:

- **01\_開発ガイド** → **02\_デザインシステム**: UI 原則を実装レベルで具体化
- **01\_開発ガイド** → **03\_技術設計**: 設計決定を実装方法に変換
- **01\_開発ガイド** → **04\_ドメイン設計**: 実装パターンをドメイン固有で適用
- **02\_デザインシステム** → **04\_ドメイン設計**: UI 原則をドメイン別に実装
- **03\_技術設計** → **04\_ドメイン設計**: 設計方針を各ドメインに適用
- **04\_ドメイン設計** → **05\_テスト戦略**: ドメイン固有のテストを実装
- **99\_課題管理** ← **全ディレクトリ**: 横断的な課題を一元管理

**相互参照の例**:

```markdown

各ドメインでの具体的な実装は [ドメイン設計](../04_ドメイン設計/) を参照してください。
```

### ファイル命名規則

#### 1. 基本ルール

- **日本語ファイル名を許可**（本リポジトリは日本語ドキュメントを第一読者と想定）
- **英語ファイル名も可**（外部公開やツール連携が必要な場合に適用）
- **命名は一貫性重視**: 同カテゴリ内で日本語/英語を混在させない
- **番号プレフィックス**: 順序が重要な場合に使用（例: `01_`, `02_`）
- **長さの目安**: 50 文字以内を推奨

補足指針:

- 検索性を高めるため、主要キーワードを先頭に置く
- OS/ツール互換性の観点から、半角英数字・アンダースコア・ハイフン中心を推奨（日本語も可）

## ドキュメント作成・更新ルール

### 1. 新規作成時の手順

#### 1.1 ドキュメントの種類を決定

- **仕様書**: 機能の詳細な仕様を定義
- **実装ガイド**: 開発者向けの実装手順
- **リファクタリング**: 既存コードの改善計画
- **テスト**: テスト戦略・手順

#### 1.2 適切なディレクトリに配置

```bash
# ドメイン別ドキュメントの場合
docs/05_ドメイン設計/{ドメイン名}/仕様/          # 仕様書
docs/05_ドメイン設計/{ドメイン名}/実装/          # 実装ガイド
docs/05_ドメイン設計/{ドメイン名}/リファクタリング/ # リファクタリング
docs/05_ドメイン設計/{ドメイン名}/テスト/        # テスト関連

# 開発ガイドの場合
docs/01_開発ガイド/

# 技術設計の場合
docs/03_技術設計/
```

#### 1.3 テンプレートを使用

各ドキュメントタイプに応じたテンプレートを使用：

```markdown
# {ドキュメントタイトル}

## 概要

{ドキュメントの目的と概要}

## 前提条件

{必要な前提知識や環境}

## 詳細内容

{メインの内容}

## 使用例

{具体的な使用例}

## 参考資料

{関連するドキュメントやリンク}
```

#### 1.4 メタデータの追加（フロントマター）

**重要**: 全ての Markdown ドキュメントには、Obsidian 用の YAML フロントマターを必ず記載すること

##### 必須フィールド

```yaml
---
title: ドキュメントタイトル
created: 2024-01-01
updated: 2024-01-01
status: draft
tags:
  - stats47
  - domain/category
  - type/specification
author: 作成者名
---
```

##### フィールド定義

| フィールド | 必須 | 説明                          | 例                                            |
| ---------- | ---- | ----------------------------- | --------------------------------------------- |
| `title`    | ✅   | ドキュメントのタイトル        | `"カテゴリ管理システム仕様書"`                |
| `created`  | ✅   | 作成日（YYYY-MM-DD 形式）     | `2024-01-15`                                  |
| `updated`  | ✅   | 最終更新日（YYYY-MM-DD 形式） | `2024-01-20`                                  |
| `status`   | ✅   | ドキュメントの状態            | `draft`, `in-review`, `published`, `archived` |
| `tags`     | ✅   | タグのリスト                  | 下記タグ規則参照                              |
| `author`   | ✅   | 作成者名                      | `"開発チーム"`                                |

##### ステータスの定義

- **draft**: 作成中・下書き状態
- **in-review**: レビュー中
- **published**: 公開済み・使用可能
- **archived**: アーカイブ済み・非推奨

##### タグ規則

**階層構造を使用**: Obsidian のタグ階層機能を活用

```yaml
tags:
  - stats47 # プロジェクト名（必須）
  - domain/category # ドメイン名
  - type/specification # ドキュメントタイプ
  - tech/typescript # 技術スタック
  - priority/high # 優先度
```

**ドメインタグ**:

- `domain/auth` - 認証ドメイン
- `domain/blog` - ブログドメイン
- `domain/category` - カテゴリ管理ドメイン
- `domain/dashboard` - ダッシュボードドメイン
- `domain/database` - データベースドメイン
- `domain/estat-api` - e-Stat API ドメイン
- `domain/ranking` - ランキングドメイン
- `domain/visualization` - 可視化ドメイン

**技術スタックタグ**:

- `tech/typescript` - TypeScript
- `tech/react` - React
- `tech/nextjs` - Next.js
- `tech/d3js` - D3.js
- `tech/cloudflare` - Cloudflare

**優先度タグ**:

- `priority/high` - 高優先度
- `priority/medium` - 中優先度
- `priority/low` - 低優先度

### 2. 更新時の手順

#### 2.1 フロントマターの更新

ドキュメントを更新する際は、必ずフロントマターも更新すること：

```yaml
---
title: ドキュメントタイトル
created: 2024-01-01
updated: 2024-01-20 # ← 更新日を変更
```

## 品質管理

- **明確性**: 読み手が理解しやすい文章

- **コード例**: 動作するコード例の提供

## 共有ドメイン（Shared Kernel）の運用

**目的**: 真に横断的な契約・語彙・型を集中管理し、重複と循環依存を防ぐ。

- **配置先**: `docs/04_ドメイン設計/共有/`
- **含めてよい**: 共通 DTO/イベントスキーマ/値オブジェクト/語彙（2 ドメイン以上で実利用）
- **含めてはいけない**: 特定ドメインに依存する実装や詳細
- **依存ポリシー**: 各ドメイン → 共有（可）／共有 → 各ドメイン（不可）
- **昇格要件**:
  - 2 ドメイン以上での実利用
  - 仕様レビュー（影響分析/代替案/破壊的変更の有無）
  - 破壊的変更時は移行ガイドを付与

詳細: `docs/04_ドメイン設計/共有/README.md` を参照

## Markdown スタイルガイド

### 見出しの直後の空行

- すべての見出し（#〜######）の直後に 1 行の空行を入れること。
- 例外は箇条書き直後の見出し遷移などでも認めない（自動整形に任せる）。
- 自動検出は markdownlint の MD022 で担保する。

#### 例

```markdown
### セクション名

本文をここに書く。
```

### 自動チェック（Lint）

- ルートの `.markdownlint.json` に以下設定を追加（MD022 を有効化）
- 可能であれば CI で markdownlint を実行

```json
{
  "MD022": { "lines_above": 1, "lines_below": 1 },
  "MD041": false
}
```

### 整形ツール

- Prettier は補助的に使用（Markdown の文章整形）
- 将来的に remark-lint を導入する場合は同等ルールを適用
