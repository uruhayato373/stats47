# データフェッチングと useSWR ガイド（実務最小）

> 設計総論（WHAT/WHY）は `docs/03_技術設計/01_アーキテクチャ/データフェッチ設計.md` に移設。本書は useSWR 実装（HOW）に特化します。

## 1. 目的と原則

- 目的: フロントのデータ取得を最小ルールで統一（再現性・保守性・性能）
- 原則: 責務分離（キー/フェッチャー/フック/表示）、型安全、キャッシュ前提、失敗に強い

## 2. データ取得層の構成

- API クライアント: 外部/内部 API 呼び出し（副作用層）
- アダプター/フォーマッター: レスポンス正規化・型付与
- 型: `Params`/`Response`/`Error` を明示し入出力を固定

推奨配置（抜粋）:

```
src/
├── lib/[domain]/{cache-key.ts, fetcher.ts, formatter.ts}
├── hooks/[domain]/use[Domain]Data.ts
└── components/[domain]/...
```

## 3. useSWR ベストプラクティス

- キー設計: 無効条件は `null`、順序は正規化、クエリは `URLSearchParams`
- 設定: `revalidateOnFocus`/`dedupingInterval` をデータ特性で調整
- エラー: フェッチャーでマッピングし、UI では文言統一
- 型安全: 判別ユニオン・型ガードでハンドリング簡素化

例（最小パターン）:

```typescript
import useSWR from "swr";
import { fetcher } from "@/lib/swr/fetcher";

export function useDomainData(params: DomainParams | null) {
  const key = params
    ? `/api/domain?${new URLSearchParams(params as any)}`
    : null;
  const { data, error, isLoading, mutate } = useSWR<DomainResponse>(
    key,
    fetcher,
    {
      revalidateOnFocus: false,
      revalidateOnReconnect: true,
      dedupingInterval: 300000,
      errorRetryCount: 3,
      errorRetryInterval: 5000,
    }
  );
  return { data: data ?? null, error, isLoading, refetch: mutate };
}
```

## 4. SSR/CSR/ISR の使い分け（要点）

- SSR/ISR: 初期表示の体験が重要・共有可能なデータ（設計詳細はシステムアーキテクチャ参照）
- CSR+SWR: 対話性・頻更新・ユーザー依存データ
- ルール: 「CPU/IO コスト高 or 初速重要 → SSR/ISR」「ユーザー固有/頻更新 → CSR+SWR」

## 5. 標準パターン（抜粋）

- 条件付きフェッチ: `key = condition ? url : null`
- 並列/バッチ: まとめ API か複数キーの SWR 併用
- ページネーション: `?offset/limit` or カーソル、キーに反映
- 楽観更新: `mutate(updater, { revalidate: false })` → 成功後 revalidate

## 6. チェックリスト

- キーが一意・正規化されている
- 無効条件で `null` を返している
- 例外はフェッチャーで握り UI は文言統一
- `dedupingInterval` がデータ特性と一致
- 型（Params/Response/Error）が明示
