## 🌍 環境別データ取得戦略（概要）

### アーキテクチャ概要

```
┌─────────────────────────────────────────────────────────────┐
│                    Server Component (page.tsx)              │
│  - 環境判定はしない（サーバー側で自動判定）                 │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│              Data Access Layer (lib/db/index.ts)            │
│  - 環境変数を読み取り、自動的にデータソースを選択           │
│  - NEXT_PUBLIC_ENV: mock / development / staging / prod     │
└──────────────────────────┬──────────────────────────────────┘
                           │
          ┌────────────────┼────────────────┬─────────────────┐
          ▼                ▼                ▼                 ▼
    ┌─────────┐      ┌──────────┐    ┌──────────┐     ┌──────────┐
    │  Mock   │      │  Local   │    │  Remote  │     │  Remote  │
    │  JSON   │      │   D1     │    │   D1     │     │   D1     │
    │  Files  │      │ (SQLite) │    │(Staging) │     │  (Prod)  │
    └─────────┘      └──────────┘    └──────────┘     └──────────┘
       mock         development        staging        production
```

### データソースの自動選択（概要）

環境変数 `NEXT_PUBLIC_ENV` に基づいて、以下のようにデータソースが自動選択されます：

- **mock**: `data/mock/database/*.json` から直接読み込み
- **development**: ローカル D1（SQLite）に接続
- **staging**: リモート D1（stats47_staging）に接続
- **production**: リモート D1（stats47）に接続

### 実装の利点

1. **関心の分離**: Server Component は UI ロジックに集中、データアクセス層で環境判定
2. **テスト容易性**: Mock 環境で完全にオフライン開発可能
3. **保守性**: 環境追加時は環境変数と provider の追加のみ
4. **型安全性**: Environment 型で環境を厳密に管理

```typescript
  },

  mock: {
    dataPath: process.env.MOCK_DATA_PATH || "data/mock",
  },
} as const;

```

### `/src/lib/api/estat-api.ts`

```typescript
import { config } from "@/lib/config";
import mockData from "@/data/mock/prefecture-stats.json";

export async function getPrefectureStats() {
  if (config.useMock) {
    console.log("Using mock data for development");
    return mockData;
  }

  const res = await fetch(
    `${config.estat.baseUrl}/getStatsData?appId=${config.estat.apiKey}&...`
  );

  if (!res.ok) {
    throw new Error("Failed to fetch e-Stat data");
  }

  return res.json();
}
```

### DB 接続例

```typescript
export const runtime = "edge";

export async function getDB() {
  if (process.env.NEXT_PUBLIC_ENV === "local") {
    return (await import("@cloudflare/workers-types")).env.STATS47_DB;
  } else {
    const response = await fetch(
      `https://api.cloudflare.com/client/v4/accounts/${process.env.CLOUDFLARE_ACCOUNT_ID}/d1/database/${process.env.CLOUDFLARE_D1_DATABASE_ID}/query`,
      {
        headers: {
          Authorization: `Bearer ${process.env.CLOUDFLARE_API_TOKEN}`,
        },
      }
    );
    return response;
  }
}
```

## 🔄 データベース同期戦略（概要）

### スキーマの同期（マイグレーション）

#### ローカル → ステージング → 本番

```bash
# 1. ローカルでマイグレーションファイルを作成
npx wrangler d1 migrations create stats47 "add_new_column"

# 2. マイグレーションファイルを編集
# database/migrations/0001_add_new_column.sql

# 3. Development環境（ローカルD1）で適用・テスト
npx wrangler d1 migrations apply stats47 --local

# 4. Staging環境（リモートD1）に適用 ⚠️ --remote必須
npx wrangler d1 migrations apply stats47_staging --env staging --remote

# 5. Production環境（リモートD1）に適用（慎重に！） ⚠️ --remote必須
npx wrangler d1 migrations apply stats47 --remote
```

### データの同期（エクスポート/インポート）

#### 本番 → ステージングへのデータコピー（推奨方法）

本番データをステージング環境にコピーする場合：

```bash
# 1. 本番DBからデータをエクスポート（リモート）
npx wrangler d1 execute stats47 --remote --command "SELECT * FROM ranking_items;" --json > database/backups/$(date +%Y%m%d)/ranking_items.json

# 2. スクリプトでSQLに変換
node scripts/import-to-staging.js

# 3. Staging環境（リモートD1）にインポート ⚠️ --remote必須
npx wrangler d1 execute stats47_staging --env staging --remote --file=database/backups/$(date +%Y%m%d)/import_to_staging.sql
```

#### 方法 1: 特定テーブルのみエクスポート

```bash
# Development環境（ローカルD1）からエクスポート
npx wrangler d1 execute stats47 --local --command "SELECT * FROM ranking_items;" --json > database/backups/ranking_items.json

# Production環境（リモートD1）からエクスポート ⚠️ --remote必須
npx wrangler d1 execute stats47 --remote --command "SELECT * FROM ranking_items;" --json > database/backups/ranking_items_prod.json
```

⚠️ **注意**: `.dump`コマンドは Cloudflare D1 のリモート接続ではサポートされていません。テーブルごとにエクスポートする必要があります。

### データ同期の推奨フロー

#### 1. **Development 環境（ローカル D1）**

```
e-Stat API → ローカルD1 → 開発・テスト
```

- **データベース**: `.wrangler/state/v3/d1`に SQLite ファイル
- **接続方法**: `--local`フラグ（デフォルト）
- **用途**:
  - e-Stat API からデータ取得
  - ローカル D1 に保存
  - 機能開発・テスト実施

#### 2. **Staging 環境（リモート Cloudflare D1）**

```
Production DB → Staging DB (リモートコピー) → 統合テスト
```

- **データベース**: Cloudflare D1 `stats47_staging` (ID: `39f18714-83bf-423c-9956-dcc9e466affb`)
- **接続方法**: `--env staging --remote`フラグ **必須**
- **用途**:
  - 本番データのコピーで統合テスト
  - マイグレーションの検証
  - 本番環境と同じ構成でテスト

⚠️ **重要**: Staging 環境は本番データをコピーして使用します。ローカルデータは同期しません。

#### 3. **Production 環境（リモート Cloudflare D1）**

```
e-Stat API → Cloudflare D1 (production) → 本番運用
```

- **データベース**: Cloudflare D1 `stats47` (ID: `e6533698-d05a-475b-9f39-5558703feef7`)
- **接続方法**: `--remote`フラグ **必須**
- **用途**:
  - e-Stat API から直接データ取得
  - 本番サービス提供
  - 読み取りが中心（書き込みは慎重に）

### データベース管理スクリプト

```bash
# 本番DBからステージングDBへのデータコピー
node scripts/import-to-staging.js

# Development環境（ローカルD1）のマイグレーション
npx wrangler d1 migrations apply stats47 --local

# Staging環境（リモートD1）のマイグレーション ⚠️ --remote必須
npx wrangler d1 migrations apply stats47_staging --env staging --remote

# Production環境（リモートD1）のマイグレーション ⚠️ --remote必須
npx wrangler d1 migrations apply stats47 --remote
```

### 重要な注意事項

⚠️ **環境とデータベースの対応関係**

- **Development**: **ローカル D1 のみ**使用（`--local`フラグ）
- **Staging**: **リモート D1 のみ**使用（`--env staging --remote`フラグ **必須**）
- **Production**: **リモート D1 のみ**使用（`--remote`フラグ **必須**）

⚠️ **データ同期の原則**

- **スキーマ（マイグレーション）**: `Development → Staging → Production` の順で同期
- **データ**:
  - Development: e-Stat API から取得してローカル D1 に保存
  - Staging: 本番 DB からコピー（統合テスト用）
  - Production: e-Stat API から直接取得
- **ローカルデータを本番/ステージングに同期しない**: 各環境でデータソースが異なる

✅ **推奨される方法**

- Development 環境では`--local`フラグを使用
- Staging/Production 環境では**必ず**`--remote`フラグを使用
- Staging 環境のテストには本番データのコピーを使用
- マイグレーション（スキーマ変更）は段階的に適用
- データのバックアップは定期的に実施

❌ **避けるべき方法**

- Staging/Production 環境で`--remote`フラグを忘れる（ローカル DB に誤って適用される）
- ローカルの開発データを本番/ステージングにコピー
- 本番データをローカルで直接編集
- マイグレーションなしでのスキーマ変更
- 環境変数の誤設定

`NEXT_PUBLIC_ENV=mock` の場合のみ利用。

> **用語の統一**: 本ドキュメントでは「モックデータ」と「スタブデータ」は同じ意味で使用しています。

### モックデータの特徴

- **e-Stat API の実際のレスポンス形式**を再現
- **都道府県・市区町村別**の階層構造
- **メタ情報と統計データ**を分離管理
- **データベース接続不要**（ローカル JSON ファイルのみ）
- **Git 管理対象**（複数端末での共有）
- **デプロイ時は除外**（Cloudflare Pages）

### データフロー比較

| 環境            | データソース  | データベース  | 用途         |
| --------------- | ------------- | ------------- | ------------ |

| **mock**        | ローカル JSON | なし          | デザイン検証 |
| **staging**     | e-Stat API    | Cloudflare D1 | 統合テスト   |
| **production**  | e-Stat API    | Cloudflare D1 | 本番運用     |

## 🧭 CI/CD・環境切替例（概要）

### cross-env を使用した環境変数管理

本プロジェクトでは`cross-env`パッケージを使用して、クロスプラットフォーム対応の環境変数設定を実現しています。

#### cross-env を使用する理由

- **クロスプラットフォーム対応**: Windows、macOS、Linux で同じコマンドが動作
- **環境変数の統一**: OS ごとの環境変数設定の違いを吸収
- **Next.js 対応**: Next.js の`--env-file`オプション非対応を解決
- **シンプルな記述**: コマンドライン上で直接環境変数を設定

## 🔄 環境切り替え方法（概要）

### .env ファイルの切り替え

各環境に対応する `.env` ファイルを作成し、適切に切り替えます：

```bash
# モック環境
cp env.mock.example .env.mock

# 開発環境
cp env.development.example .env.development

# ステージング環境
cp env.staging.example .env.staging

# 本番環境
cp env.production.example .env.production
```

### npm スクリプトの追加（推奨）

`package.json` に環境別のスクリプトを追加：

```json
{
  "scripts": {
    "dev:mock": "cross-env NEXT_PUBLIC_ENV=mock npm run dev",
    "dev:local": "cross-env NEXT_PUBLIC_ENV=development npm run dev",
    "dev:staging": "cross-env NEXT_PUBLIC_ENV=staging npm run dev",
    "build:staging": "cross-env NEXT_PUBLIC_ENV=staging npm run build",
    "build:production": "cross-env NEXT_PUBLIC_ENV=production npm run build"
  }
}
```

### 環境切り替えの実行

```bash
# モック環境で開発
npm run dev:mock

# ローカルD1環境で開発
npm run dev:local

# ステージング環境でビルド
npm run build:staging

# 本番環境でビルド
npm run build:production
```

### トラブルシューティング

#### 1. 環境変数が正しく読み込まれない場合

```bash
# 現在の環境変数を確認
echo $NEXT_PUBLIC_ENV

# .env ファイルの内容を確認
cat .env.development
```

#### 2. データが表示されない場合

```bash
# コンソールログで環境を確認
# 期待される出力例：
# [mock] Fetching estat metainfo from mock data...
# [development] Fetching estat metainfo from database...
# [staging] Fetching estat metainfo from database...
```

#### 3. データベース接続エラーの場合

```bash
# ローカルD1の状態確認
npx wrangler d1 list

# リモートD1の状態確認
npx wrangler d1 list --remote
```
