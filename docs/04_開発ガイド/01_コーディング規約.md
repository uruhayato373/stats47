# コーディング規約

## 概要

このドキュメントは、stats47 プロジェクト固有のコーディング規約を定義します。プロジェクトで採用している設計パターン（Fetcher/Formatter/Display、Atomic Design、ドメイン駆動設計）に基づいた実践的な規約を提供します。

## 1. プロジェクト概要

### 1.1 採用技術スタック

- **フレームワーク**: Next.js 15 (App Router)
- **UI ライブラリ**: React 19
- **言語**: TypeScript 5.x
- **スタイリング**: Tailwind CSS 4
- **状態管理**: Jotai
- **データフェッチ**: SWR
- **可視化**: Recharts, D3.js

### 1.2 設計パターン

- **Fetcher/Formatter/Display**: データ取得・整形・表示の責務分離
- **Atomic Design**: コンポーネントの階層化設計
- **ドメイン駆動設計**: ビジネスロジックのドメイン別整理
- **Server/Client Components**: レンダリング戦略の使い分け

## 2. ディレクトリ構造規約

### 2.1 Atomic Design の階層

```
src/components/
├── atoms/           # 最小単位のコンポーネント
├── molecules/       # atoms の組み合わせ
├── organisms/       # molecules の組み合わせ
├── templates/       # ページレイアウト
└── pages/          # ページ固有のコンポーネント
```

### 2.2 ドメイン別構成

```
src/infrastructure/
├── domain/         # ドメイン別ビジネスロジック
│   ├── estat-api/  # e-Stat API ドメイン
│   ├── ranking/    # ランキングドメイン
│   └── visualization/ # 可視化ドメイン
└── shared/         # 共通ユーティリティ
```

### 2.3 コンポーネント配置ルール

- **atoms**: 再利用可能な最小単位（Button, Input, Modal など）
- **molecules**: 特定の機能を持つ組み合わせ（SearchForm, DataCard など）
- **organisms**: 複雑な機能を持つ大きなコンポーネント（DataTable, Chart など）
- **templates**: ページレイアウト（DashboardLayout, AuthLayout など）
- **pages**: ページ固有のコンポーネント（HomePage, RankingPage など）

### 3.5 関数名の動詞選択

関数名には、その関数が**何をするか**を明確に示す動詞を使用します。

参考: [Get で始まる名前を関数につける前に読む記事](https://zenn.dev/blue_jam/articles/2a347b36b43d59)

#### 動詞選択ガイドライン

| 関数の性質                     | 推奨動詞                            | 例                                        |
| ------------------------------ | ----------------------------------- | ----------------------------------------- |
| 外部サービス呼び出し           | `fetch`, `retrieve`                 | `fetchUserData()`, `retrieveStatistics()` |
| dictionary/map から読み出し    | `lookup`                            | `lookupCategoryName()`                    |
| データベース検索               | `find`, `search`                    | `findSubcategoryById()`, `searchItems()`  |
| 配列・リスト全体を返す         | `list`, `getAll`                    | `listCategories()`, `getAllUsers()`       |
| 簡単な計算                     | `calculate`                         | `calculateAverage()`                      |
| 複雑な計算                     | `compute`                           | `computeRankings()`                       |
| 新しいオブジェクト生成         | `build`, `create`, `compose`        | `buildUrl()`, `createReport()`            |
| データ変換                     | `transform`, `convert`, `normalize` | `transformData()`, `normalizeCategory()`  |
| 検証                           | `validate`, `check`                 | `validateInput()`, `checkPermission()`    |
| private フィールドそのまま返す | `get`                               | `getName()`, `getId()`                    |

#### ❌ 避けるべきパターン

```typescript
// ❌ 悪い例: getは何をするか不明確
function getItemName(id: string): Promise<string> {
  return fetch(`/api/items/${id}`).then((r) => r.json());
}

// ✅ 良い例: fetchで外部呼び出しを明示
function fetchItemName(id: string): Promise<string> {
  return fetch(`/api/items/${id}`).then((r) => r.json());
}
```

#### ✅ 推奨パターン

```typescript

// ✅ リスト取得

```

#### パフォーマンス推測の重要性

適切な動詞選択により、コードレビュアーが関数のパフォーマンス特性を推測できます：

```typescript
// fetchを見れば「ネットワーク通信がある」と推測できる
for (let i = 0; i < 100; i++) {
  await fetchItemName(i); // ⚠️ 100回のAPI呼び出し！
}

// lookupを見れば「辞書検索なので高速」と推測できる
for (let i = 0; i < 100; i++) {
  lookupItemName(i); // ✅ 問題なし
}
```

#### 命名のガイドライン

1. **具体的な動詞を使用**: `get`の代わりに処理内容を表す動詞を選択
2. **一貫性を保つ**: プロジェクト全体で同じ種類の処理には同じ動詞を使用
3. **簡潔さと明確さのバランス**: 長すぎず、意図が明確な名前
4. **型情報を活用**: 戻り値の型で意図を補完（例: `findCategory` は `Category | undefined` を返す）

### 4.4 純粋な関数の実装

```typescript
// ✅ 推奨（純粋な関数）
export function calculateRanking(data: RankingData[]): RankedItem[] {
  // 副作用なし、同じ入力に対して常に同じ出力
  return data
    .sort((a, b) => b.value - a.value)
    .map((item, index) => ({ ...item, rank: index + 1 }));
}

// ✅ 推奨（複数の純粋な関数を組み合わせ）
export function processData(raw: RawData): ProcessedData {
  const validated = validateData(raw);
  const filtered = filterByStatus(validated, "active");
  return transformToOutput(filtered);
}

// ✅ 推奨（lookup関数 - dictionary/mapからの読み出し）
const categoryMap = new Map<string, Category>();

export function lookupCategory(id: string): Category | null {
  return categoryMap.get(id) ?? null;
}

// ✅ 推奨（compute関数 - 複雑な計算）
export function computeStatistics(data: number[]): Statistics {
  const sum = data.reduce((a, b) => a + b, 0);
  const mean = sum / data.length;
  const variance =
    data.map((x) => Math.pow(x - mean, 2)).reduce((a, b) => a + b, 0) /
    data.length;

  return {
    count: data.length,
    sum,
    mean,
    variance,
    stdDev: Math.sqrt(variance),
  };
}

// ❌ 避ける（副作用を持つ関数）
let cache: Data[] = [];
export function getCachedData(): Data[] {
  if (cache.length === 0) {
    cache = fetchData(); // 副作用: モジュールレベルの状態変更
  }
  return cache;
}

// ✅ 推奨（副作用が必要な場合は明示的に）
export async function fetchAndCacheData(
  cache: Map<string, Data>,
  key: string
): Promise<Data> {
  // 引数として渡されたcacheを更新（明示的な副作用）
  const data = await fetchData(key);
  cache.set(key, data);
  return data;
}
```

**純粋関数のガイドライン**:

1. **純粋関数を優先**: 同じ入力に対して常に同じ出力を返す
2. **副作用の分離**: データフェッチやキャッシュは別の関数に分離し、明示的に
3. **適切な動詞を使用**: 処理の性質に応じた動詞を選択（fetch, lookup, calculate, compute 等）
4. **テスト可能性**: 純粋関数は単体テストが容易
5. **不変性**: 引数を変更せず、新しいオブジェクトを返す
6. **型安全性**: 戻り値の型で処理結果を明確に表現

## 12. コード品質管理

### 12.1 未使用コードの管理

#### YAGNI 原則の適用

「必要になるまで実装しない (You Aren't Gonna Need It)」の原則に従います。

**理由**:

- コードベースの肥大化を防ぐ
- 保守コストを削減
- テストの対象範囲を最小化
- 実際の要件に基づいた設計が可能

#### 実装タイミング

1. **実装する**: 現在のユースケースで確実に使用される機能
2. **実装しない**: 将来使うかもしれない機能、推測に基づく機能
3. **リファクタリング時に追加**: 同様のパターンが 3 回以上出現した時

#### 未使用コードの検出と削除

定期的に以下を実行します:

```bash
# 未使用エクスポートの検出
npx ts-prune

# 未使用インポートの削除
npx eslint --fix
```

**削除対象**:

- 一度も参照されていない関数・クラス
- テストコードからのみ参照される実装コード
- コメントアウトされたコード

### 12.2 コードレビュー指針

- **YAGNI 原則の遵守**: 必要最小限の実装を確認
- **未使用コードの検出**: 不要な機能の実装を指摘
- **責務の明確化**: 単一責任の原則の遵守
- **テストカバレッジ**: 実装した機能のテスト確認
