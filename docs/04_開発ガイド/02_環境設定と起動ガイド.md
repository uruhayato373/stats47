# 環境設定と起動ガイド

## 概要

このドキュメントは、stats47 プロジェクトの環境設定から起動までを一元的に説明します。各環境の特徴、セットアップ手順、起動方法を実践的に提供します。

## 1. 環境一覧

| 環境            | コマンド                   | データソース  | 用途         |
| --------------- | -------------------------- | ------------- | ------------ |
| **development** | `npm run dev`              | ローカル D1   | ローカル開発 |
| **staging**     | `npm run dev:staging`      | リモート D1   | 統合テスト   |
| **production**  | `npm run build:production` | リモート D1   | 本番運用     |

詳細は[環境設定ガイド](../../01_技術設計/04_インフラ設計/04_環境設定ガイド.md)を参照してください。

## 2. 初回セットアップ

### 2.1 クイックスタート（5 分セットアップ）

最速で開発を開始したい場合:

```bash
# リポジトリのクローン
git clone https://github.com/your-org/stats47.git
cd stats47

# 依存関係のインストール
npm install

# 開発環境のセットアップ
cp env.development.example .env.development
npx wrangler d1 migrations apply stats47 --local
npm run dev
```

### 2.2 前提条件

```bash
# Node.js (18.x以上)
node --version

# npm
npm --version

# Git
git --version
```

### 2.3 プロジェクトのセットアップ

```bash
# リポジトリのクローン
git clone https://github.com/your-org/stats47.git
cd stats47

# 依存関係のインストール
npm install
```

### 2.4 環境変数ファイルの作成

#### 開発環境用（共通設定）

```bash
# .env.developmentファイルを作成
cp env.development.example .env.development
```

**.env.development の設定:**

```bash
# NextAuth v5設定（Auth.js）
AUTH_SECRET=<生成されたシークレット>
AUTH_URL=http://localhost:3000

# e-Stat API設定
ESTAT_API_BASE_URL=https://api.e-stat.go.jp/rest/3.0/app
NEXT_PUBLIC_ESTAT_APP_ID=your_estat_app_id

# モックデータパス（テストやStorybook用）
MOCK_DATA_PATH=data/mock
```

環境変数の詳細な説明については、[環境設定ガイド](../../01_技術設計/04_インフラ設計/04_環境設定ガイド.md#環境変数設計)を参照してください。

#### AUTH_SECRET の生成方法

**Node.js で生成（推奨）:**

```bash
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
```

**OpenSSL で生成:**

```bash
openssl rand -base64 32
```

**セキュリティ注意:**

- 各環境で異なるシークレットを使用
- シークレットは最低 32 文字
- `.env.*` ファイルは Git にコミットしない

## 3. 環境別詳細

### 3.1 開発環境

#### 起動方法

```bash
npm run dev
```

#### 特徴

- **データソース**: ローカル D1 Database または e-Stat API
- **データベース**: ローカル D1（`.wrangler/state/v3/d1`）
- **API 接続**: e-Stat API（オプション）
- **認証**: NextAuth（Auth.js）による認証管理
- **用途**: フル機能の開発とテスト、オフライン開発、デザイン検証、Storybook

各環境の詳細な特徴については、[環境設定ガイド](../../01_技術設計/04_インフラ設計/04_環境設定ガイド.md#各環境の目的と特徴)を参照してください。

**注意**: モックデータ（`data/mock/`）はテストやStorybookなどで個別に使用できますが、環境としてのモックモードは提供していません。

#### 前提条件

- ローカル D1 データベースのセットアップ（`npx wrangler d1 migrations apply stats47 --local`）
- `.env.development` に `AUTH_SECRET` の設定
- e-Stat API キーの取得と設定（オプション）

#### アクセス先

- アプリケーション: `http://localhost:3000`
- 管理画面: `http://localhost:3000/admin` （認証必須）
- 開発ツール: `http://localhost:3000/admin/dev-tools/estat-api` （認証必須）

#### モックデータの活用

モックデータはテストやStorybookで使用できます：

```
data/mock/database/
├── ranking_items.json
├── ranking_values.json
└── estat_metainfo.json
```

ローカル D1 からモックデータを生成する場合:

```bash
# ranking_itemsデータをエクスポート
npx wrangler d1 execute stats47 --local --command "SELECT * FROM ranking_items LIMIT 10;" --json > data/mock/database/ranking_items.json

# ranking_valuesデータをエクスポート
npx wrangler d1 execute stats47 --local --command "SELECT * FROM ranking_values LIMIT 50;" --json > data/mock/database/ranking_values.json

# estat_metainfoデータをエクスポート
npx wrangler d1 execute stats47 --local --command "SELECT * FROM estat_metainfo LIMIT 50;" --json > data/mock/database/estat_metainfo.json
```

### 3.2 ステージング環境（統合テスト用）

#### 起動方法

```bash
npm run dev:staging
```

#### 特徴

- **データソース**: リモート D1（ステージング）
- **API 接続**: e-Stat API に接続
- **用途**: 統合テスト

各環境の詳細な特徴については、[環境設定ガイド](../../01_技術設計/04_インフラ設計/04_環境設定ガイド.md#各環境の目的と特徴)を参照してください。

#### 前提条件

- ステージング用 D1 データベースの作成
- Cloudflare 認証設定

### 3.3 本番環境

#### ビルド方法

```bash
npm run build:production
```

#### 特徴

- **データソース**: リモート D1（本番）
- **認証**: **必須** - セキュリティを最優先
- **用途**: 本番運用

各環境の詳細な特徴については、[環境設定ガイド](../../01_技術設計/04_インフラ設計/04_環境設定ガイド.md#各環境の目的と特徴)を参照してください。

#### 開発ワークフロー

**通常の開発**:

1. `npm run dev` で起動
2. ローカル D1 データベースを使用
3. 認証機能を含む全機能を開発・テスト
4. 実データでの動作確認

**本番デプロイ前**:

1. `npm run dev` で最終確認
2. 認証機能の動作テスト
3. 実データでの統合テスト
4. パフォーマンス確認
5. ステージング環境で最終検証

## 8. トラブルシューティング

### 8.1 AUTH_SECRET エラー

```bash
# エラー: [auth][error] MissingSecret

# 原因: .env.developmentファイルが作成されていない、または読み込まれていない

# 解決方法:
# 1. 環境変数ファイルが存在するか確認
ls -la .env.*

# 2. .env.developmentを作成
cp env.development.example .env.development

# 3. AUTH_SECRETが設定されているか確認
cat .env.development | grep AUTH_SECRET

# 4. サーバーを再起動
npm run dev
```

### 8.2 環境変数が読み込まれない

```bash
# .envファイルの内容を確認
cat .env.development

# 環境変数が正しく設定されているか確認
cat .env.development | grep AUTH_SECRET
cat .env.development | grep NEXT_PUBLIC_ESTAT_APP_ID
```

### 8.3 データベース接続エラー

```bash
# ローカルD1の状態確認
npx wrangler d1 list

# リモートD1の状態確認
npx wrangler d1 list --remote

# データベースIDの確認
npx wrangler d1 list --json
```

### 8.4 データが表示されない

コンソールログで環境を確認してください：

```bash
# 期待される出力例：
# [development] Fetching estat metainfo from database...
# [staging] Fetching estat metainfo from database...
```

### 8.5 認証エラー

```bash
# AUTH_SECRETが設定されているか確認
cat .env.development | grep AUTH_SECRET

# AUTH_SECRETが未設定の場合は生成
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"

# .env.developmentに追加
echo "AUTH_SECRET=<生成されたシークレット>" >> .env.development

# サーバーを再起動
npm run dev
```

### 8.6 本番環境でのセキュリティ確認

```bash
# NODE_ENVを確認
echo $NODE_ENV
# → "production" であることを確認

# 本番ビルドを再実行
npm run build:production
npm start

# 本番環境変数ファイルを確認
cat .env.production
# → 必要な環境変数が正しく設定されていることを確認
```

## 9. 技術的な補足

### 9.1 環境変数の読み込み仕組み

プロジェクトでは Next.js の標準的な環境変数読み込み機能を使用しています。

**環境ファイルの読み込み順序:**

1. `.env.development.local` (最優先)
2. `.env.local` (test を除く)
3. `.env.development`
4. `.env`

### 9.2 環境判定の方法

コード内で環境変数を使用して設定を管理：

```typescript
// 環境判定の例
const env = process.env.NEXT_PUBLIC_ENV || "development";

if (env === "development") {
  // ローカルD1またはe-Stat APIを使用
  return fetchFromDatabase();
} else if (env === "staging" || env === "production") {
  // リモートD1またはe-Stat APIを使用
  return fetchFromDatabase();
}
```

### 9.3 設定コード例

環境設定を一元管理する場合:

```typescript
// /src/infrastructure/config.ts
export const config = {
  env: process.env.NEXT_PUBLIC_ENV || "development",
  estat: {
    baseUrl:
      process.env.ESTAT_API_BASE_URL || "https://api.e-stat.go.jp/rest/3.0/app",
    apiKey: process.env.ESTAT_API_KEY,
  },
  cloudflare: {
    d1: process.env.CLOUDFLARE_D1_DATABASE_ID,
    r2: process.env.CLOUDFLARE_R2_BUCKET,
    accountId: process.env.CLOUDFLARE_ACCOUNT_ID,
    apiToken: process.env.CLOUDFLARE_API_TOKEN,
  },
  mock: {
    dataPath: process.env.MOCK_DATA_PATH || "data/mock", // テストやStorybook用
  },
} as const;

export type Config = typeof config;
```

## 11. セキュリティ注意事項

### 11.1 環境変数の管理

- `.env.*`ファイルは Git にコミットしない
- 機密情報は Cloudflare Secrets で管理
- 本番環境の API キーは慎重に管理
- 各環境で異なる`AUTH_SECRET`を使用

### 11.2 データベースアクセス

- 本番環境のデータベースは読み取り中心
- 書き込み操作は慎重に実行
- マイグレーションは段階的に適用

### 11.3 認証のセキュリティ

**開発環境の制限**:

- **本番環境では厳格な認証**: 本番環境では常に認証が必須
- **開発環境でのテスト**: ローカルでの開発とテストが可能
- **環境別の設定**: 各環境で適切なセキュリティ設定を適用

**チェックリスト**:

- [ ] `NODE_ENV=production`が正しく設定されていること
- [ ] 本番ビルド時に認証が正常に動作することを確認
- [ ] 各環境で異なる`AUTH_SECRET`を使用していること
- [ ] 本番環境変数ファイルに必要な設定が含まれていること

                  CLASS: [
                    {
                      "@code": "2020",
                      "@name": "2020年",
                    },
                    {
                      "@code": "2021",
                      "@name": "2021年",
                    },
                  ],
                },
              ],
            },
          },
        },
      };

      const result = parseCompleteMetaInfo(mockResponse);

      expect(result).toHaveProperty("tableInfo");
      expect(result).toHaveProperty("categories");
      expect(result).toHaveProperty("timeAxis");
      expect(result.categories).toHaveLength(2);
    });

    it("空のデータを正しく処理できる", () => {
      const mockResponse: EstatMetaInfoResponse = {
        GET_META_INFO: {
          RESULT: {
            STATUS: 0,
            ERROR_MSG: null,
            DATE: "2024-01-01T00:00:00+09:00",
          },
          METADATA_INF: { CLASS_INF: { CLASS_OBJ: [] } },
        },
      };

      const result = parseCompleteMetaInfo(mockResponse);

      expect(result.categories).toEqual([]);
      expect(result.timeAxis.years).toEqual([]);
    });
  });

  describe("extractCategories", () => {
    it("分類情報を正しく抽出できる", () => {
      const mockClassObj = [
        {
          "@id": "cat01",
          "@name": "分類",
          CLASS: [
            { "@code": "A1101", "@name": "総人口", "@level": "1" },
            { "@code": "A1102", "@name": "男性人口", "@level": "1" },
          ],
        },
      ];

      const result = extractCategories(mockClassObj);

      expect(result).toHaveLength(2);
      expect(result[0]).toMatchObject({
        code: "A1101",
        name: "総人口",
        categoryId: "cat01",
      });
    });
  });

  describe("extractTimeAxis", () => {
    it("時間軸情報を正しく抽出できる", () => {
      const mockClassObj = [
        {
          "@id": "time",
          "@name": "時間軸",
          CLASS: [
            { "@code": "2020", "@name": "2020年" },
            { "@code": "2021", "@name": "2021年" },
            { "@code": "2022", "@name": "2022年" },
          ],
        },
      ];

      const result = extractTimeAxis(mockClassObj);

      expect(result.years).toEqual(["2020", "2021", "2022"]);
      expect(result.timeAxis).toHaveLength(3);
      expect(result.timeAxis[0]).toMatchObject({
        code: "2020",
        name: "2020年",
      });
    });
  });
});
