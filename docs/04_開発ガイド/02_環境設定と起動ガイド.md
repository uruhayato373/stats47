# 環境設定と起動ガイド

## 概要

このドキュメントは、stats47 プロジェクトの環境設定から起動までを一元的に説明します。各環境の特徴、セットアップ手順、起動方法を実践的に提供します。

## 1. 環境一覧

| 環境            | コマンド                   | データソース  | 用途         |
| --------------- | -------------------------- | ------------- | ------------ |
| **development** | `npm run dev:mock`         | JSON ファイル | デザイン検証 |
| **development** | `npm run dev:api`          | e-Stat API    | API 開発     |
| **staging**     | `npm run dev:staging`      | リモート D1   | 統合テスト   |
| **production**  | `npm run build:production` | リモート D1   | 本番運用     |

### モードの切り替え

development 環境では`NEXT_PUBLIC_USE_MOCK`フラグでモードを切り替えます：

- **`npm run dev:mock`**: `NEXT_PUBLIC_USE_MOCK=true` - モックデータを使用
- **`npm run dev:api`**: `NEXT_PUBLIC_USE_MOCK=false` - e-Stat API を使用

## 2. 初回セットアップ

### 2.1 クイックスタート（5分セットアップ）

最速で開発を開始したい場合:

```bash
# リポジトリのクローン
git clone https://github.com/your-org/stats47.git
cd stats47

# 依存関係のインストール
npm install

# パターン1: モック環境（最速・DB不要）
cp env.development.example .env.development
npm run dev:mock

# パターン2: API環境（実データ・ローカルD1使用）
cp env.development.example .env.development
npx wrangler d1 migrations apply stats47 --local
npm run dev:api
```

### 2.2 前提条件

```bash
# Node.js (18.x以上)
node --version

# npm
npm --version

# Git
git --version
```

### 2.3 プロジェクトのセットアップ

```bash
# リポジトリのクローン
git clone https://github.com/your-org/stats47.git
cd stats47

# 依存関係のインストール
npm install
```

### 2.4 環境変数ファイルの作成

#### 開発環境用（共通設定）

```bash
# .env.developmentファイルを作成
cp env.development.example .env.development
```

**.env.development の設定:**

```bash
# NextAuth v5設定（Auth.js）
AUTH_SECRET=<生成されたシークレット>
AUTH_URL=http://localhost:3000

# e-Stat API設定
ESTAT_API_BASE_URL=https://api.e-stat.go.jp/rest/3.0/app
NEXT_PUBLIC_ESTAT_APP_ID=your_estat_app_id

# モックデータパス
MOCK_DATA_PATH=data/mock

# 注意: NEXT_PUBLIC_USE_MOCKはpackage.jsonのスクリプトで指定します
# dev:mock → NEXT_PUBLIC_USE_MOCK=true
# dev:api → NEXT_PUBLIC_USE_MOCK=false
```

#### AUTH_SECRET の生成方法

**Node.js で生成（推奨）:**

```bash
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
```

**OpenSSL で生成:**

```bash
openssl rand -base64 32
```

**セキュリティ注意:**

- 各環境で異なるシークレットを使用
- シークレットは最低 32 文字
- `.env.*` ファイルは Git にコミットしない

## 3. 環境別詳細

### 3.1 開発環境（モックモード）

#### 起動方法

```bash
npm run dev:mock
```

#### 特徴

- **データソース**: JSON ファイル（`data/mock/`）
- **データベース**: 接続不要
- **API 接続**: 不要
- **認証**: **バイパス有効** - 自動的に管理者としてログイン
- **モード**: `NEXT_PUBLIC_USE_MOCK=true`

#### 認証動作

**Mock環境の認証バイパス**:
- 認証プロセスを自動的にスキップ
- モック管理者として自動ログイン
- `/admin` に直接アクセス可能

**実装**:
```typescript
// src/features/auth/lib/auth.ts
export const auth = async () => {
  const isMockEnv = process.env.NEXT_PUBLIC_USE_MOCK === "true";

  if (isMockEnv && process.env.NODE_ENV !== "production") {
    return {
      user: {
        id: "mock-admin",
        name: "モック管理者",
        email: "admin@example.com",
        username: "admin",
        role: "admin" as const,
      },
    };
  }

  return null;
};
```

**セキュリティ**:
- 本番環境では自動的に無効化（`NODE_ENV !== "production"`条件）
- 開発環境のみで有効

#### 適用場面

- デザインシステムの検証
- コンポーネントの開発
- オフライン開発
- 管理画面のデザイン確認
- 素早いUI動作確認

#### アクセス先

- アプリケーション: `http://localhost:3000`
- 管理画面: `http://localhost:3000/admin` （認証不要）

#### モックデータの構造

```
data/mock/database/
├── ranking_items.json
├── ranking_values.json
└── estat_metainfo.json
```

#### モックデータの更新

ローカルD1からモックデータを生成する場合:

```bash
# ranking_itemsデータをエクスポート
npx wrangler d1 execute stats47 --local --command "SELECT * FROM ranking_items LIMIT 10;" --json > data/mock/database/ranking_items.json

# ranking_valuesデータをエクスポート
npx wrangler d1 execute stats47 --local --command "SELECT * FROM ranking_values LIMIT 50;" --json > data/mock/database/ranking_values.json

# estat_metainfoデータをエクスポート
npx wrangler d1 execute stats47 --local --command "SELECT * FROM estat_metainfo LIMIT 50;" --json > data/mock/database/estat_metainfo.json
```

### 3.2 開発環境（API モード）

#### 起動方法

```bash
npm run dev:api
```

#### 特徴

- **データソース**: e-Stat API
- **API 接続**: 必要（e-Stat API キー）
- **認証**: **必須** - 完全なログインフローをテスト可能
- **モード**: `NEXT_PUBLIC_USE_MOCK=false`

#### 認証動作

**API環境の認証**:
- 認証バイパスは無効
- 実際の認証フローが動作
- 未認証時は自動的にログインページへリダイレクト
- NextAuth（Auth.js）による認証管理

**実装**:
```typescript
// API環境では認証チェックを有効化
export const auth = async () => {
  // NextAuth実装（TODO: 有効化予定）
  // 現在は一時的に無効化
  return null;
};
```

**用途**:
- 認証機能の開発とテスト
- ログインフローの確認
- セッション管理の検証
- 本番に近い環境での統合テスト

#### 前提条件

- e-Stat API キーの取得と設定
- `.env.development` に `AUTH_SECRET` の設定

#### アクセス先

- アプリケーション: `http://localhost:3000`
- 管理画面: `http://localhost:3000/admin` （認証必須）
- 開発ツール: `http://localhost:3000/admin/dev-tools/estat-api` （認証必須）

#### 起動方法

```bash
npm run dev:staging
```

#### 特徴

- **データソース**: リモート D1（ステージング）
- **API 接続**: e-Stat API に接続
- **用途**: 統合テスト

#### 前提条件

- ステージング用 D1 データベースの作成
- Cloudflare 認証設定

### 3.4 本番環境

#### ビルド方法

```bash
npm run build:production
```

#### 特徴

- **データソース**: リモート D1（本番）
- **認証**: **必須** - セキュリティを最優先
- **用途**: 本番運用

#### 認証動作

**本番環境のセキュリティ**:
- 認証バイパスは完全に無効化
- `NODE_ENV=production` で認証チェックを強制
- すべての管理画面・開発ツールで認証必須
- セッション管理は本番用設定

**実装**:
```typescript
// 本番環境では必ず認証チェック
export const auth = async () => {
  if (process.env.NODE_ENV === "production") {
    // 本番用NextAuth実装
    return await nextAuth();
  }

  return null;
};
```

**セキュリティ対策**:
- Mock環境の認証バイパスは自動的に無効
- すべてのAPIエンドポイントで認証確認
- 環境変数による厳密な制御

### 3.5 認証システムの詳細

#### 認証バイパスの条件

Mock環境でのみ認証バイパスが有効になるための条件:

```typescript
// セキュリティチェック
if (isMockEnv && process.env.NODE_ENV !== "production") {
  // Mock環境かつ本番環境でない場合のみバイパス
  return mockSession;
}
```

**条件**:
1. `NEXT_PUBLIC_USE_MOCK=true`
2. `NODE_ENV !== "production"`

両方の条件を満たす場合のみ認証バイパスが動作します。

#### 開発ワークフロー

**UI開発時（Mock環境）**:
1. `npm run dev:mock` で起動
2. `/admin` に直接アクセス
3. 認証なしで管理画面を開発・テスト
4. モックデータでUI動作確認

**認証機能開発時（API環境）**:
1. `npm run dev:api` で起動
2. ログイン機能を実装・テスト
3. 認証フローを確認
4. 実データでの動作確認

**本番デプロイ前（API環境）**:
1. `npm run dev:api` で最終確認
2. 認証機能の動作テスト
3. 実データでの統合テスト
4. パフォーマンス確認

#### 今後の拡張予定

**予定されている機能**:
- **ステージング環境**: 本番に近い環境でのテスト
- **認証プロバイダー**: Google、GitHub等のOAuth連携
- **多要素認証**: セキュリティ強化
- **セッション管理**: より高度なセッション制御

**設定の拡張例**:
```typescript
// 将来的な設定例
const authConfig = {
  mock: {
    enabled: process.env.NEXT_PUBLIC_USE_MOCK === "true",
    bypassAuth: true,
    mockUser: { /* ... */ },
  },
  api: {
    enabled: process.env.NEXT_PUBLIC_USE_MOCK === "false",
    providers: ["credentials", "google"],
    session: { strategy: "jwt" },
  },
  production: {
    enabled: process.env.NODE_ENV === "production",
    providers: ["credentials", "google"],
    session: { strategy: "database" },
  },
};
```

## 8. トラブルシューティング

### 8.1 AUTH_SECRET エラー

```bash
# エラー: [auth][error] MissingSecret

# 原因: .env.developmentファイルが作成されていない、または読み込まれていない

# 解決方法:
# 1. 環境変数ファイルが存在するか確認
ls -la .env.*

# 2. .env.developmentを作成
cp env.development.example .env.development

# 3. AUTH_SECRETが設定されているか確認
cat .env.development | grep AUTH_SECRET

# 4. サーバーを再起動
npm run dev:mock
# または
npm run dev:api
```

### 8.2 環境変数が読み込まれない

```bash
# 現在の環境変数を確認
echo $NEXT_PUBLIC_USE_MOCK

# .envファイルの内容を確認
cat .env.development
```

### 8.3 データベース接続エラー

```bash
# ローカルD1の状態確認
npx wrangler d1 list

# リモートD1の状態確認
npx wrangler d1 list --remote

# データベースIDの確認
npx wrangler d1 list --json
```

### 8.4 データが表示されない

コンソールログで環境を確認してください：

```bash
# 期待される出力例：
# [development] NEXT_PUBLIC_USE_MOCK=true - Fetching estat metainfo from mock data...
# [development] NEXT_PUBLIC_USE_MOCK=false - Fetching estat metainfo from API...
# [staging] Fetching estat metainfo from database...
```

### 8.5 Mock環境で認証が効かない

```bash
# 環境変数を確認
echo $NEXT_PUBLIC_USE_MOCK
# → "true" であることを確認

# 再起動
npm run dev:mock

# それでも解決しない場合はNODE_ENVを確認
echo $NODE_ENV
# → "production" でないことを確認（development または未設定）
```

### 8.6 API環境で認証エラー

```bash
# 環境変数を確認
echo $NEXT_PUBLIC_USE_MOCK
# → "false" であることを確認

# AUTH_SECRETが設定されているか確認
cat .env.development | grep AUTH_SECRET

# AUTH_SECRETが未設定の場合は生成
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"

# .env.developmentに追加
echo "AUTH_SECRET=<生成されたシークレット>" >> .env.development

# サーバーを再起動
npm run dev:api
```

### 8.7 本番環境で認証バイパスが有効になっている

これは重大なセキュリティ問題です：

```bash
# NODE_ENVを確認
echo $NODE_ENV
# → "production" であることを確認

# 本番ビルドを再実行
npm run build:production
npm start

# NEXT_PUBLIC_USE_MOCKが設定されていないことを確認
echo $NEXT_PUBLIC_USE_MOCK
# → 空または"false"であること

# 本番環境変数ファイルを確認
cat .env.production
# → NEXT_PUBLIC_USE_MOCKが存在しないか"false"であること
```

**重要**: 本番環境で認証バイパスが有効な場合、すぐに以下を実行:
1. アプリケーションを停止
2. 環境変数を修正
3. 再ビルドとデプロイ
4. 認証が正常に動作することを確認

## 9. 技術的な補足

### 9.1 環境変数の読み込み仕組み

プロジェクトでは Next.js の標準的な環境変数読み込み機能を使用しています。

**環境ファイルの読み込み順序:**

1. `.env.development.local` (最優先)
2. `.env.local` (test を除く)
3. `.env.development`
4. `.env`

### 9.2 環境判定の方法

コード内で`process.env.NEXT_PUBLIC_USE_MOCK`を使用してモードを判定：

```typescript
// モックモードの判定例
const useMock = process.env.NEXT_PUBLIC_USE_MOCK === "true";

if (useMock) {
  // モックデータを使用
  return fetchFromJSON();
} else {
  // e-Stat APIを使用
  return fetchFromAPI();
}
```

### 9.3 設定コード例

環境設定を一元管理する場合:

```typescript
// /src/infrastructure/config.ts
export const config = {
  env: process.env.NEXT_PUBLIC_ENV || "development",
  estat: {
    baseUrl:
      process.env.ESTAT_API_BASE_URL || "https://api.e-stat.go.jp/rest/3.0/app",
    apiKey: process.env.ESTAT_API_KEY,
  },
  cloudflare: {
    d1: process.env.CLOUDFLARE_D1_DATABASE_ID,
    r2: process.env.CLOUDFLARE_R2_BUCKET,
    accountId: process.env.CLOUDFLARE_ACCOUNT_ID,
    apiToken: process.env.CLOUDFLARE_API_TOKEN,
  },
  mock: {
    dataPath: process.env.MOCK_DATA_PATH || "data/mock",
  },
} as const;

export type Config = typeof config;
```

## 11. セキュリティ注意事項

### 11.1 環境変数の管理

- `.env.*`ファイルは Git にコミットしない
- 機密情報は Cloudflare Secrets で管理
- 本番環境の API キーは慎重に管理
- 各環境で異なる`AUTH_SECRET`を使用

### 11.2 データベースアクセス

- 本番環境のデータベースは読み取り中心
- 書き込み操作は慎重に実行
- マイグレーションは段階的に適用

### 11.3 認証バイパスのセキュリティ

**Mock環境の制限**:
- **本番環境では無効**: `NODE_ENV=production` では認証バイパス無効
- **開発環境のみ**: 本番データへのアクセスは不可
- **一時的な機能**: 本番デプロイ時は自動的に無効化

**認証バイパスの条件**:
```typescript
// セキュリティチェック
if (isMockEnv && process.env.NODE_ENV !== "production") {
  // Mock環境かつ本番環境でない場合のみバイパス
  return mockSession;
}
```

**チェックリスト**:
- [ ] 本番環境で`NEXT_PUBLIC_USE_MOCK=true`が設定されていないこと
- [ ] `NODE_ENV=production`が正しく設定されていること
- [ ] 本番ビルド時に認証が正常に動作することを確認
- [ ] 各環境で異なる`AUTH_SECRET`を使用していること
- [ ] `.env.production`に認証バイパス設定が含まれていないこと
