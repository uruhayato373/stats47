# ドキュメント管理ルール

## 概要

stats47 プロジェクトのドキュメント管理に関する統一的なルールとガイドラインを定義します。

プロジェクトの品質向上、保守性の確保、チーム内での知識共有を目的とします。

---

## 設計ドキュメントと開発ガイドの分類基準

### 基本原則

**設計ドキュメント（Technical Design）**: **「What（何を作るか）」と「Why（なぜそう設計したか）」**を記録

**開発ガイド（Development Guide）**: **「How（どう実装するか）」**を記録

### 判断フローチャート

「この内容はどちらに書くべき？」と迷ったら:

```
質問1: これは設計思想や判断の根拠を説明しているか？
  YES → 設計ドキュメント（docs/01_技術設計/）
  NO  → 質問2へ

質問2: これは具体的な実装手順やコード例か？
  YES → 開発ガイド（docs/04_開発ガイド/）
  NO  → 質問3へ

質問3: これは「なぜ」を説明しているか、「どうやって」を説明しているか？
  「なぜ」 → 設計ドキュメント（docs/01_技術設計/）
  「どうやって」 → 開発ガイド（docs/04_開発ガイド/）
```

### 設計ドキュメント（docs/01_技術設計/）に記載すべき内容

#### ✅ 記載すべき内容

| カテゴリ | 具体例 | 理由 |
|---------|--------|------|
| **アーキテクチャ決定** | 5層アーキテクチャ、DDD、Adapter Pattern | 設計思想の根幹 |
| **ドメインモデル** | エンティティ、値オブジェクト、ドメインサービス | ビジネスロジックの概念定義 |
| **データモデル** | ER図、テーブル構造、リレーションシップ | データ構造の設計 |
| **システム構成図** | レイヤー図、データフロー図（シーケンス図） | システム全体像の可視化 |
| **設計判断の根拠** | 「なぜR2を使うか」「なぜAdapter Patternか」 | 将来の変更時の判断材料 |
| **制約と前提条件** | Cloudflare D1の制限、パフォーマンス要件 | 設計の制約を明確化 |
| **インターフェース定義** | API仕様、型定義、統一データモデル | 契約の定義 |

**例**:

```markdown
## アーキテクチャ設計

### 5層アーキテクチャ

Ranking ドメインは以下の 5 層で設計されています：

1. Presentation Layer: UIコンポーネント
2. Service Layer: ビジネスロジック
3. Repository Layer: データアクセス抽象化
4. Adapter Layer: 外部データソース統合
5. Infrastructure Layer: DB/ストレージ

**設計判断**:
- なぜ5層か？ → 関心の分離、テスタビリティ向上
- なぜAdapter Patternか？ → マルチデータソース対応の柔軟性

### データフロー設計

[シーケンス図]

### データモデル

[簡易ER図 + データベース設計.mdへの参照]
```

#### ❌ 記載すべきでない内容

- 具体的な実装手順（「まず〇〇を作り、次に△△を...」）
- コード例（最小限の型定義例は可）
- トラブルシューティング手順
- 環境構築手順

### 開発ガイド（docs/04_開発ガイド/）に記載すべき内容

#### ✅ 記載すべき内容

| カテゴリ | 具体例 | 理由 |
|---------|--------|------|
| **実装手順** | 「新しいランキング項目を追加する手順」 | 開発者の作業ガイド |
| **コード例** | RankingServiceの使い方、アダプター実装例 | 実装の具体的な参考 |
| **ベストプラクティス** | エラーハンドリングパターン、ログ出力 | 実装の質向上 |
| **トラブルシューティング** | よくあるエラーと対処法 | 問題解決の効率化 |
| **環境構築** | D1セットアップ、R2バケット作成 | 開発開始の手順 |
| **テスト方法** | 単体テスト、統合テストの書き方 | 品質保証 |
| **デバッグ方法** | D1クエリ確認、R2確認コマンド | 開発効率化 |

**例**:

```markdown
## 新しいランキング項目の追加手順

### 手順1: D1にランキング項目を登録

```sql
INSERT INTO ranking_items (ranking_key, label, unit, ...)
VALUES ('gdp-per-capita', '一人当たりGDP', '万円', ...);
```

### 手順2: データソースメタデータを登録

```sql
INSERT INTO data_source_metadata (ranking_key, metadata, ...)
VALUES ('gdp-per-capita', '{"stats_data_id": "..."}', ...);
```

### 手順3: データ取得をテスト

```bash
curl http://localhost:3000/api/rankings/gdp-per-capita?year=2023
```

### よくあるエラー

- "Ranking item not found" → 手順1の登録確認
- "API request failed: 403" → 環境変数の確認
```

#### ❌ 記載すべきでない内容

- アーキテクチャ全体図（設計ドキュメントへの参照でOK）
- ドメインモデルの詳細定義
- 設計思想の説明

### 両者の関係性

```
┌─────────────────────────────────────┐
│   設計ドキュメント (What & Why)      │
│                                     │
│  - アーキテクチャ決定                │
│  - ドメインモデル                    │
│  - データモデル                      │
│  - 設計判断の根拠                    │
└──────────────┬──────────────────────┘
               │
               │ 参照
               │
┌──────────────▼──────────────────────┐
│   開発ガイド (How)                   │
│                                     │
│  - 実装手順                          │
│  - コード例                          │
│  - トラブルシューティング            │
│  - 環境構築                          │
└─────────────────────────────────────┘
```

**重要原則**:
- 開発ガイドは設計ドキュメントを参照すべき
- 設計ドキュメントから開発ガイドへの参照は可（実装手順へのリンク）
- **重複を避ける**: 同じ内容を2箇所に書かない

### 特性の比較

| 観点 | 設計ドキュメント | 開発ガイド |
|------|----------------|-----------|
| **目的** | What & Why | How |
| **内容** | アーキテクチャ、ドメインモデル、設計判断 | 実装手順、コード例、トラブルシューティング |
| **読者** | アーキテクト、新規参加者、将来の自分 | 実装者、トラブル対応者 |
| **更新頻度** | 低（アーキテクチャ変更時） | 高（機能追加、エラー事例追加） |
| **安定性** | 高（設計は変わりにくい） | 低（実装パターンは進化） |
| **参照方向** | 開発ガイドから参照される | 設計ドキュメントを参照する |

### 相互参照のベストプラクティス

#### 1. 開発ガイドのREADME.mdで橋渡し

開発ガイドのREADME.mdで、設計ドキュメントへの強い参照を作る:

```markdown

実装前に以下の設計ドキュメントを読んでください：

- [Rankingドメイン設計](../../../01_技術設計/03_ドメイン設計/10_ranking.md)
- [データベース設計](../../../01_技術設計/04_インフラ設計/01_データベース設計.md)

```

➡️ [ランキングドメイン 開発ガイド](../../04_開発ガイド/01_ドメイン/ranking/README.md)
```

## ドキュメント構造

### ディレクトリ命名規則

#### 1. トップレベルディレクトリ

```
docs/
├── 00_プロジェクト管理/     # プロジェクト概要・要件
├── 01_技術設計/            # 技術設計・アーキテクチャ
├── 02_ドメイン設計/        # ドメイン別ドキュメント
├── 03_デザインシステム/     # デザインシステム
├── 04_開発ガイド/          # 開発ガイド・規約
├── 05_テスト戦略/          # テスト戦略
├── 06_運用保守/            # 運用・保守
├── 07_コンテンツ計画/       # コンテンツ計画・リソース
│   └── リソース/           # リソース・素材（07配下に格納）
├── 99_課題管理/            # 既知の課題・調査
└── INDEX.md               # ドキュメントインデックス
```

### ディレクトリの役割と使い分け

#### 全体構成の原則

- **設計** → **実装** → **運用** の論理的な順序
- **WHAT/WHY（設計）** と **HOW（実装）** の明確な分離
- プロジェクト参加時の学習順序と一致した配置

#### 00\_プロジェクト管理

**目的**: プロジェクト全体の方向性、要件、戦略を管理

**対象読者**: プロジェクトマネージャー、ステークホルダー、アーキテクト

**内容の特徴**:

- プロジェクト憲章・概要・収益化戦略
- 機能要件・非機能要件・MVP 定義
- 実装計画・ロードマップ・統合計画
- 技術選定評価・戦略決定

**配置するドキュメント**:

- `プロジェクト憲章.md`、`プロジェクト概要.md`
- `機能要件.md`、`非機能要件.md`、`MVP定義.md`
- `実装計画/`、`技術選定評価_2025.md`

**更新頻度**: 中程度（プロジェクトフェーズ変更時）

**他ディレクトリとの関係**: 全ディレクトリの基盤となる情報を提供

#### 01\_技術設計（設計 WHAT/WHY）

**目的**: システム全体の**設計決定と理由**を記録

**対象読者**: アーキテクト、技術選定を行うリードエンジニア、プロジェクト全体像を理解したい人

**内容の特徴**:

- システム全体のアーキテクチャ
- 技術選定の理由と背景（ADR）
- データベース設計と環境戦略
- API 設計の全体像と型定義の設計思想

**配置するドキュメント**:

- `01_システム概要/` - 技術スタック、システムアーキテクチャ、プロジェクト構造
- `02_ADR/` - Architecture Decision Records
- `03_ドメイン設計/` - DDD ドメイン分類、ドメイン間依存関係
- `04_データ設計/` - データベース設計、データフェッチ設計、型定義管理
- `05_フロントエンド設計/` - ルーティング、コンポーネント、状態管理、レンダリング戦略
- `06_設計パターン/` - 実装パターン、エラーハンドリング、パフォーマンス設計

**更新頻度**: 低い（設計フェーズで確定、大きな変更時のみ更新）

**他ディレクトリとの関係**:

- `04_開発ガイド/` で実装方法に変換
- `02_ドメイン設計/` の各ドメインに設計方針を適用

**設計例**:

```
データベースアーキテクチャ
├── 環境別データ取得戦略
│   ├── mock: JSON ファイル
│   ├── development: ローカル D1
│   ├── staging: リモート D1
│   └── production: リモート D1
```

#### 02\_ドメイン設計（DDD 実践）

**目的**: 各ドメインの責務・仕様・実装・テスト・リファクタリングを一元管理

**対象読者**: ドメインエキスパート、機能開発者、テストエンジニア

**内容の特徴**:

- ドメイン駆動設計（DDD）に基づく構造
- 各ドメインの 4 層構造（仕様/実装/リファクタリング/テスト）
- ドメイン間の境界と依存関係の管理
- ビジネスロジックの詳細設計

**配置するドキュメント**:

- `e-Stat API/`、`ダッシュボード/`、`可視化/` など各ドメイン
- 各ドメイン内に `仕様/`、`実装/`、`リファクタリング/`、`テスト/` の 4 層
- `共有/` - 複数ドメインで使用する共通要素

**更新頻度**: 中程度（機能開発・リファクタリング時）

**他ディレクトリとの関係**:

- `01_技術設計/` の設計方針を各ドメインに適用
- `04_開発ガイド/` の実装パターンをドメイン固有で具体化
- `03_デザインシステム/` の UI 原則をドメイン別に実装

#### 03\_デザインシステム（UI/UX 標準）

**目的**: UI/UX の原則、スタイル、コンポーネント設計を基準化

**対象読者**: フロントエンド開発者、デザイナー、UI 実装者

**内容の特徴**:

- デザイン原則とスタイルガイド
- コンポーネント設計原則と Atomic Design
- レスポンシブデザインガイド
- アクセシビリティ基準

**配置するドキュメント**:

- `スタイルガイド.md`、`アイコンシステム.md`
- `タイポグラフィシステム.md`、`レスポンシブデザインガイド.md`
- `Atomic Design実装.md`、`shadcn-ui統合ガイド.md`
- `Storybook運用ガイド.md`

**更新頻度**: 中程度（デザインシステム更新時）

**他ディレクトリとの関係**:

- `04_開発ガイド/` で実装方法を具体化
- `02_ドメイン設計/` の各ドメインで UI 実装パターンを適用

#### 04\_開発ガイド（実装 HOW）

**目的**: 開発者が**どうやって実装するか**を指南

**対象読者**: コードを実装する開発者、日常的な開発作業を行うエンジニア

**内容の特徴**:

- コード行レベルの具体例と実装パターン
- コーディング規約（TypeScript、React、Tailwind CSS）
- 開発ワークフローとツールの使い方
- トラブルシューティングと解決策

**配置するドキュメント**:

- `コーディング規約.md` - 日々のコーディングで参照する規約
- `環境設定ガイド.md` - 開発環境のセットアップ手順
- `大規模実装ガイド.md` - 実装手順とパターン
- `開発環境チューニング.md` - 開発環境の最適化

**更新頻度**: 高い（プロジェクト進行中に随時更新）

**他ディレクトリとの関係**:

- `03_デザインシステム/` の UI 原則を実装レベルで具体化
- `01_技術設計/` の設計決定を実装方法に変換
- `02_ドメイン設計/` の実装ガイドと連携

**実装例**:

```typescript
// ✅ 推奨
interface UserProps {
  id: string;
  name: string;
}

// ❌ 避ける
const user: any = { ... };
```

#### 05\_テスト戦略

**目的**: 単体/統合/E2E/ビジュアル等のテスト方針・運用を定義

**対象読者**: テストエンジニア、開発者、QA 担当者

**内容の特徴**:

- テスト戦略の全体像と各層の役割
- テストケース設計とテストデータ管理
- テスト自動化と CI/CD 連携
- 品質メトリクスと分析

**配置するドキュメント**:

- `ユニットテストガイド.md` - 単体テストの実装方法
- 各ドメインの `テスト/` ディレクトリと連携

**更新頻度**: 中程度（テスト戦略変更・新機能追加時）

**他ディレクトリとの関係**:

- `02_ドメイン設計/` の各ドメインでテストを実装
- `04_開発ガイド/` の実装パターンにテストを組み込み

#### 06\_運用保守

**目的**: デプロイ、バックアップ、トラブルシュート、監視・アラート、パフォーマンス運用

**対象読者**: インフラエンジニア、DevOps 担当者、運用担当者

**内容の特徴**:

- デプロイ手順と環境管理
- バックアップ・リストア手順
- 監視・アラート設定
- トラブルシューティングガイド

**配置するドキュメント**:

- `デプロイ運用ガイド.md` - デプロイ・運用の実践的手順
- `データベーストラブルシューティング.md` - データベース関連のトラブルシューティング

**更新頻度**: 低い（運用手順変更時）

**他ディレクトリとの関係**:

- `01_技術設計/` のインフラ設計を運用レベルで具体化

#### 07\_コンテンツ計画

**目的**: 記事やダッシュボードのコンテンツ企画・編集計画

**対象読者**: コンテンツ企画者、編集者、マーケター

**内容の特徴**:

- ランキング記事の企画・構成
- ダッシュボード記事の設計
- ブログ記事の計画・編集方針
- SEO・アクセス解析を考慮したコンテンツ戦略

**配置するドキュメント**:

- `01_ランキング記事/`、`02_ダッシュボード/`、`03_ブログ記事/`
- `リソース/` - 素材・参考資料（08\_リソースから移動）

**更新頻度**: 高い（コンテンツ企画・更新時）

**他ディレクトリとの関係**:

#### 99\_課題管理

### ファイル命名規則

#### 1. 基本ルール

- **日本語ファイル名を許可**（本リポジトリは日本語ドキュメントを第一読者と想定）
- **英語ファイル名も可**（外部公開やツール連携が必要な場合に適用）
- **命名は一貫性重視**: 同カテゴリ内で日本語/英語を混在させない
- **番号プレフィックス**: 順序が重要な場合に使用（例: `01_`, `02_`）
- **長さの目安**: 50 文字以内を推奨

補足指針:

- 検索性を高めるため、主要キーワードを先頭に置く
- OS/ツール互換性の観点から、半角英数字・アンダースコア・ハイフン中心を推奨（日本語も可）

#### 1.4 メタデータの追加（フロントマター）

**重要**: 全ての Markdown ドキュメントには、Obsidian 用の YAML フロントマターを必ず記載すること

##### 必須フィールド

```yaml
---
title: ドキュメントタイトル
created: 2024-01-01
updated: 2024-01-01
status: draft
tags:
  - stats47
  - domain/category
  - type/specification
author: 作成者名
---
```

##### フィールド定義

| フィールド | 必須 | 説明                          | 例                                            |
| ---------- | ---- | ----------------------------- | --------------------------------------------- |
| `title`    | ✅   | ドキュメントのタイトル        | `"カテゴリ管理システム仕様書"`                |
| `created`  | ✅   | 作成日（YYYY-MM-DD 形式）     | `2024-01-15`                                  |
| `updated`  | ✅   | 最終更新日（YYYY-MM-DD 形式） | `2024-01-20`                                  |
| `status`   | ✅   | ドキュメントの状態            | `draft`, `in-review`, `published`, `archived` |
| `tags`     | ✅   | タグのリスト                  | 下記タグ規則参照                              |
| `author`   | ✅   | 作成者名                      | `"開発チーム"`                                |

##### ステータスの定義

- **draft**: 作成中・下書き状態
- **in-review**: レビュー中
- **published**: 公開済み・使用可能
- **archived**: アーカイブ済み・非推奨

##### タグ規則

**階層構造を使用**: Obsidian のタグ階層機能を活用

```yaml
tags:
  - stats47 # プロジェクト名（必須）
  - domain/category # ドメイン名
  - type/specification # ドキュメントタイプ
  - tech/typescript # 技術スタック
  - priority/high # 優先度
```

**ドメインタグ**:

- `domain/auth` - 認証ドメイン
- `domain/blog` - ブログドメイン
- `domain/category` - カテゴリ管理ドメイン
- `domain/dashboard` - ダッシュボードドメイン
- `domain/database` - データベースドメイン
- `domain/estat-api` - e-Stat API ドメイン
- `domain/ranking` - ランキングドメイン
- `domain/visualization` - 可視化ドメイン

**技術スタックタグ**:

- `tech/typescript` - TypeScript
- `tech/react` - React
- `tech/nextjs` - Next.js
- `tech/d3js` - D3.js
- `tech/cloudflare` - Cloudflare

**優先度タグ**:

- `priority/high` - 高優先度
- `priority/medium` - 中優先度
- `priority/low` - 低優先度

## 共有ドメイン（Shared Kernel）の運用

**目的**: 真に横断的な契約・語彙・型を集中管理し、重複と循環依存を防ぐ。

- **配置先**: `docs/02_ドメイン設計/共有/`
- **含めてよい**: 共通 DTO/イベントスキーマ/値オブジェクト/語彙（2 ドメイン以上で実利用）
- **含めてはいけない**: 特定ドメインに依存する実装や詳細
- **依存ポリシー**: 各ドメイン → 共有（可）／共有 → 各ドメイン（不可）
- **昇格要件**:
  - 2 ドメイン以上での実利用
  - 仕様レビュー（影響分析/代替案/破壊的変更の有無）
  - 破壊的変更時は移行ガイドを付与

詳細: `docs/02_ドメイン設計/共有/README.md` を参照

## Markdown スタイルガイド

### 見出しの直後の空行

- すべての見出し（#〜######）の直後に 1 行の空行を入れること。
- 例外は箇条書き直後の見出し遷移などでも認めない（自動整形に任せる）。
- 自動検出は markdownlint の MD022 で担保する。

#### 例

```markdown
### セクション名

本文をここに書く。
```

### 自動チェック（Lint）

- ルートの `.markdownlint.json` に以下設定を追加（MD022 を有効化）
- 可能であれば CI で markdownlint を実行

```json
{
  "MD022": { "lines_above": 1, "lines_below": 1 },
  "MD041": false
}
```

### 整形ツール

- Prettier は補助的に使用（Markdown の文章整形）
- 将来的に remark-lint を導入する場合は同等ルールを適用
