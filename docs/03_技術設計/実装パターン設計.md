# 実装パターン設計（Fetcher/Formatter/Display 他）

> 本文書は設計（WHAT/WHY）の総論です。実装手順（HOW）は `docs/01_開発ガイド/大規模実装ガイド.md` を参照。

## 1. 基本方針

- DDD と 3 層（Data → Domain/Service → UI）の厳密な分離
- Next.js App Router で Server/Client 境界を明確化
- 型主導（Type-First）設計で契約を先に固める

## 2. コアパターン

### 2.1 Fetcher/Formatter/Display

```
Fetcher (外部/内部から取得)
  → Formatter (正規化・型付与・変換)
    → Display (UI 表示)
```

設計意図:
- 取得/変換/表示の責務を分離し、テスト容易性と再利用性を高める
- ドメイン知識は Display ではなく Formatter/Service に集約

### 2.2 Server/Client 分離

- Server Component: データ取得と初期レンダリングを担当
- Client Component: インタラクションと状態管理を担当

設計意図:
- 初速と SEO を最大化しつつ、対話性を担保

### 2.3 環境別データソース抽象

- Data Access Layer が `NEXT_PUBLIC_ENV` 等から自動選択（mock/local/staging/prod）
- UI・Page から環境分岐ロジックを排除

## 3. ディレクトリ/配置原則（設計観点）

```
src/
├ app/            # Server/route
├ components/     # UI
├ lib/
│  └ domain/
│     ├ types/
│     ├ fetcher.ts
│     ├ formatter.ts
│     └ service.ts
└ hooks/
```

- ドメイン型は各ドメイン直下にコロケーション
- UI は純粋化（ドメイン知識は Organisms 以上、Molecules/Atoms には持ち込まない）

## 4. 状態管理の設計

- 複雑な状態はカスタムフックへ分離し、副作用を一箇所に集約
- UI は props 駆動、ビジネスロジックは Service/Formatter に寄せる

## 5. 品質保全

- 型で契約を固定 → 並行開発の衝突を低減
- 単一責任の原則: ファイル/関数/コンポーネント単位で責務を限定
