## 型管理について

Stats47 プロジェクトでは、ドメイン駆動設計（DDD）の原則に基づいて型定義を管理しています。

### 基本方針

- **コロケーション原則**: 型定義は、それを使用するドメインロジックと同じディレクトリ内に配置
- **ドメイン境界の明確化**: 各ドメインの型は独立して管理
- **依存関係の方向性**: 上位ドメインから下位ドメインへの一方向依存を維持

### 型の配置

- **共有型**: `src/types/shared/` - 真に複数のドメインで共有される汎用型のみ
- **ドメイン型**: `lib/domain/types/` - 各ドメイン固有の型（各ドメイン内で管理）
- **外部型拡張**: `src/types/external/` - 外部ライブラリの型拡張

詳細は[型管理ガイド](./type-management-guide.md)を参照してください。

## コンポーネント設計について

アトミックデザインの原則に基づいてコンポーネントを階層化し、再利用性と保守性を向上させています。

### コンポーネント階層

- **Atoms**: 最小単位の UI コンポーネント
- **Molecules**: 複数の Atoms を組み合わせたコンポーネント
- **Organisms**: 複数の Molecules を組み合わせた複雑なコンポーネント
- **Templates**: ページレイアウトのテンプレート
- **Pages**: 実際のページコンポーネント

詳細は[コンポーネント設計ガイド](./component_guide.md)を参照してください。

## スタイリングについて

Tailwind CSS を使用した統一されたデザインシステムを採用しています。

### デザイン原則

- **一貫性**: プロジェクト全体で統一されたデザイン
- **アクセシビリティ**: WCAG 2.1 AA 準拠
- **レスポンシブ**: モバイルファーストアプローチ
- **ダークモード**: ライト・ダークモードの両方に対応

詳細は[スタイリングガイド](./styling_guide.md)を参照してください。

## 開発フロー

### 1. 機能開発

1. 要件を理解し、ドメイン設計を確認
2. 型定義を適切な場所に配置
3. コンポーネントをアトミックデザインに従って実装
4. テストを記述・実行
5. コードレビューを実施

### 2. バグ修正

1. 問題を再現し、原因を特定
2. 最小限の変更で修正
3. テストを更新・追加
4. コードレビューを実施

### 3. リファクタリング

1. 現状の問題点を整理
2. 改善案を検討
3. 段階的にリファクタリングを実施
4. テストを更新・追加
5. ドキュメントを更新
