# コンポーネント設計ガイド

## 1. 概要

このドキュメントでは、stats47 プロジェクトのコンポーネント設計の全体像を説明します。本プロジェクトは、e-Stat API から取得した統計データを視覚化し、ユーザーに直感的なデータ探索体験を提供することを目的としています。



### 設計原則

- **単一責任の原則**: 各コンポーネントは一つの明確な責任を持つ
- **再利用性**: 共通の UI パターンは共通コンポーネントとして実装
- **型安全性**: TypeScript の型定義を活用
- **アクセシビリティ**: WCAG 準拠の実装
- **パフォーマンス**: サーバーコンポーネントとクライアントコンポーネントの適切な分離

## 2. ページコンポーネント

### 2.1 カテゴリーページ

#### 概要

カテゴリーページ（`/[category]`）は、各統計カテゴリーのランディングページとして機能し、所属するサブカテゴリーへの導線を提供します。

#### 設計目標

- **明確なナビゲーション**: サブカテゴリーへの分かりやすい導線
- **視覚的な整理**: カテゴリーアイコンと色による識別
- **柔軟な対応**: サブカテゴリーの有無に応じた表示
- **一貫性**: 全カテゴリーで統一されたレイアウト

#### コンポーネント構造

### 2.2 サブカテゴリーページ

- **再利用性**: 共通レイアウトとコンポーネントの活用
- **拡張性**: 新しいサブカテゴリーの追加が容易

### 2.3 ダッシュボードページ

#### 概要

ダッシュボードページは、全国用と都道府県用に分離された新しいアーキテクチャを採用しています。各サブカテゴリーは2つの専用コンポーネントを持ち、`areaCode`に基づいて動的に選択されます。

#### アーキテクチャパターン

**複数コンポーネント + コンポーネント解決**

- **NationalDashboard**: 全国専用ダッシュボード（`areaCode === "00000"`）
- **PrefectureDashboard**: 都道府県専用ダッシュボード（`areaCode !== "00000"`）

#### コンポーネント解決システム

```typescript
// src/components/subcategories/index.tsx
export const getDashboardComponentByArea = (
  subcategoryId: string,
  areaCode: string,
  categoryId?: string
): React.ComponentType<SubcategoryDashboardPageProps> => {
  const subcategory = getSubcategoryInfo(subcategoryId, categoryId);

  if (!subcategory) {
    return DefaultDashboardPage;
  }

  const isNational = areaCode === "00000";

  // 新しいアーキテクチャ: 地域別コンポーネント
  if (isNational && subcategory.nationalDashboardComponent) {
    return (
      componentMap[subcategory.nationalDashboardComponent] ||
      DefaultDashboardPage
    );
  }

  if (!isNational && subcategory.prefectureDashboardComponent) {
    return (
      componentMap[subcategory.prefectureDashboardComponent] ||
      DefaultDashboardPage
    );
  }

  // フォールバック: 従来の単一コンポーネント
  if (subcategory.dashboardComponent) {
    return componentMap[subcategory.dashboardComponent] || DefaultDashboardPage;
  }

  return DefaultDashboardPage;
};
```

#### 命名規則

- **NationalDashboard**: `[Name]NationalDashboard.tsx`
- **PrefectureDashboard**: `[Name]PrefectureDashboard.tsx`
- **コンポーネント名**: `[Name]NationalDashboard`, `[Name]PrefectureDashboard`

#### コンポーネントマッピングシステム

**動的コンポーネント選択**

本プロジェクトでは、`categories.json` の設定に基づいて、サブカテゴリーごとに表示する React コンポーネントを動的に選択するシステムを採用しています。

1.  **コンポーネント集約**
    各カテゴリーディレクトリ（例: `landweather`, `population`）からエクスポートされたすべてのコンポーネントが、`src/components/subcategories/index.tsx` 内の `componentMap` オブジェクトに集約されます。

    ```typescript
    // src/components/subcategories/index.tsx
    import * as LandWeather from './landweather';
    import * as Population from './population';
    // ... 他のカテゴリーも同様にインポート

    const componentMap: Record<string, React.ComponentType<...>> = {
      ...LandWeather,
      ...Population,
      // ... 他のカテゴリーを展開
    };
    ```

2.  **JSON によるコンポーネント指定**
    `src/config/categories.json` ファイル内の各サブカテゴリー定義に、使用するコンポーネント名を `component` および `areaComponent` プロパティで指定します。

    ```json
    // src/config/categories.json
    {
      "id": "land-area",
      "name": "土地面積",
      "component": "LandAreaPage", // 都道府県ランキングページ用
      "areaComponent": "LandAreaDetailPage" // 都道府県別詳細ページ用
    }
    ```

3.  **動的なコンポーネント取得**
    `getSubcategoryComponent` 関数が、サブカテゴリー ID を基に `categories.json` を参照し、指定されたコンポーネント名のコンポーネントを `componentMap` から取得して返します。都道府県別ページも同様に `getAreaPageComponent` が対応します。

    ```typescript
    // src/components/subcategories/index.tsx
    export const getSubcategoryComponent = (subcategoryId: string): React.ComponentType<...> => {
      const subcategory = getSubcategoryInfo(subcategoryId);
      if (!subcategory?.component) return DefaultSubcategoryPage;

      return componentMap[subcategory.component] || DefaultSubcategoryPage;
    };

    export const getAreaPageComponent = (subcategoryId: string): React.ComponentType<...> => {
      const subcategory = getSubcategoryInfo(subcategoryId);
      if (!subcategory?.areaComponent) return DefaultSubcategoryPage;

      return componentMap[subcategory.areaComponent] || DefaultSubcategoryPage;
    };
    ```

**ページでの使用**

```typescript
// src/app/[category]/[subcategory]/page.tsx
import { getSubcategoryComponent } from '@/components/subcategories';

export default async function SubcategoryPage({ params, ... }) {
  const { subcategory: subcategoryId } = params;

  // サブカテゴリーIDに対応するコンポーネントを動的に取得
  const SubcategoryComponent = getSubcategoryComponent(subcategoryId);

  return <SubcategoryComponent {...props} />;
}
```

#### 新しいサブカテゴリーの追加方法

新しいサブカテゴリーページを追加する手順は以下の通りです。

**1. 専用コンポーネントの作成**

まず、サブカテゴリーに対応する新しい React コンポーネントを作成します。コンポーネントは、関連するカテゴリーのディレクトリ（例: `src/components/subcategories/population/`）内に配置します。

```tsx
// src/components/subcategories/population/PopulationCompositionPage.tsx
"use client";

import React from "react";
import { SubcategoryLayout } from "../SubcategoryLayout";
import { SubcategoryPageProps } from "@/types/subcategory";

export const PopulationCompositionPage: React.FC<SubcategoryPageProps> = ({
  category,
  subcategory,
  // ...その他のprops
}) => {
  return (
    <SubcategoryLayout category={category} subcategory={subcategory}>
      {/* ここにサブカテゴリー独自のレイアウトやコンポーネントを実装 */}
      <p>人口構成ページ</p>
    </SubcategoryLayout>
  );
};
```

**2. カテゴリーの `index.tsx` からエクスポート**

作成したコンポーネントを、所属するカテゴリーの `index.tsx` ファイルからエクスポートします。これにより、中央の `componentMap` から参照できるようになります。

```tsx
// src/components/subcategories/population/index.tsx
export { BasicPopulationDashboard } from "./basic-population";
export { PopulationCompositionPage } from "./PopulationCompositionPage"; // この行を追加
```

**3. `categories.json` にコンポーネント名を指定**

最後に、`src/config/categories.json` を開き、対応するサブカテゴリーの定義に `component` プロパティを追加し、作成したコンポーネントのファイル名を値として指定します。

```json
// src/config/categories.json
{
  "id": "population",
  "name": "人口・世帯",
  "subcategories": [
    // ...既存のサブカテゴリー
    {
      "id": "population-composition",
      "name": "人口構成",
      "component": "PopulationCompositionPage" // この行を追加
    }
  ]
}
```

以上の手順で、新しいサブカテゴリーページがシステムに統合され、`/population/population-composition` のような URL でアクセスできるようになります。

### SavedEstatMetainfoDisplay

保存された e-STAT メタデータを表示するクライアントコンポーネントです。

#### 設計方針

- **クライアントコンポーネント**: `"use client";` が指定されており、データの取得から表示まで、すべてクライアントサイドで完結します。
- **状態管理**: `useState` と `useEffect` を使用して、ローディング状態、エラー、取得したメタデータを管理します。
- **インタラクティブな操作**: データの再取得や更新などのアクションは、子コンポーネントの `EstatMetainfoActions` と連携して実行されます。

#### コンポーネント構成

1. **SavedEstatMetainfoDisplay** (クライアントコンポーネント)

   - `useEffect` 内で `/api/estat/metainfo/stats` エンドポイントからデータを非同期に取得します。
   - ローディング中、エラー発生時、データが空の場合の UI をそれぞれ表示します。
   - 取得したデータをテーブル形式で表示します。

2. **EstatMetainfoActions** (クライアントコンポーネント)
   - 更新ボタンや再試行ボタンを提供し、親コンポーネントのデータ取得関数をコールバックとして受け取ります。

#### メリット

- **動的なデータ更新**: ユーザーのアクション（例: 更新ボタンのクリック）に応じて、ページ全体をリロードすることなくデータを再取得して表示を更新できます。
- **リッチな UI**: ローディングスピナーやエラーメッセージなど、状態に応じたインタラクティブな UI を容易に実装できます。

## 7. データ統合

### e-Stat API 統合

#### 概要

e-Stat API からメタデータを取得し、Cloudflare D1 データベースに保存する機能を実装しています。

#### データフロー

1. **ユーザー入力** → 統計表 ID を入力
2. **API 呼び出し** → `/api/estat/metadata/save`に POST
3. **データ取得** → e-Stat API からメタデータを取得
4. **データ変換** → CSV 形式に変換
5. **D1 保存** → Cloudflare D1 に保存
6. **結果表示** → Message コンポーネントで成功/エラー表示

#### 実装場所

- **API ルート**: `src/app/api/estat/metadata/save/route.ts`
- **フロントエンド**: `src/components/estat/metadata/MetadataSaver.tsx`
- **データ変換**: `src/lib/estat/data-transformer.ts`
- **データベース操作**: `src/lib/estat/metadata-database.ts`

### Cloudflare D1 統合

#### データベース設定

- **データベース名**: `stats47`
- **バインディング**: `STATS47_DB`
- **スキーマ**: `database/schemas/main.sql`

#### セットアップ

```bash
# D1データベースの作成（初回のみ）
npx wrangler d1 create stats47

# リモートのD1データベースにスキーマを適用
npx wrangler d1 execute stats47 --remote --file=./database/schemas/main.sql
```

開発環境でも本番環境と同じリモートの D1 データベースを使用します。これにより、環境間の差異をなくし、一貫したデータ管理を実現しています。

### データ保存戦略

#### 開発環境

- **保存先**: Cloudflare D1（本番環境と同じ）
- **利点**:
  - 本番環境との一貫性
  - ローカル PC の負荷軽減
  - 実際の API 制限の確認
- **処理方式**: チャンク分割処理（50 件ずつ）
- **待機時間**: チャンク間で 200ms（API 制限対策）

#### 本番環境

- **保存先**: Cloudflare D1
- **利点**:
  - スケーラビリティ
  - グローバル配信
  - 高可用性
- **処理方式**: チャンク分割処理（50 件ずつ）
- **待機時間**: チャンク間で 200ms（API 制限対策）

#### チャンク処理の詳細

```typescript
// チャンクサイズ: 50件（Cloudflare D1の制限に最適化）
const CHUNK_SIZE = 50;

// チャンク分割
const chunks = [];
for (let i = 0; i < data.length; i += CHUNK_SIZE) {
  chunks.push(data.slice(i, i + CHUNK_SIZE));
}

// チャンクごとの処理
for (const [chunkIndex, chunk] of chunks.entries()) {
  // チャンク内で並列処理
  const chunkPromises = chunk.map(async (item) => {
    // 個別データの処理
  });

  // チャンク内の処理完了を待機
  const chunkResults = await Promise.all(chunkPromises);

  // 次のチャンクまで待機（API制限対策）
  if (chunkIndex < chunks.length - 1) {
    await new Promise((resolve) => setTimeout(resolve, 200));
  }
}
```

## 9. トラブルシューティング

### よくある問題

1. **スタイルが適用されない**

   - `useStyles`フックが正しくインポートされているか確認
   - Tailwind CSS の設定を確認

2. **Storybook でスタイルが表示されない**

   - `.storybook/storybook.css`の設定を確認
   - `tailwind.config.ts`に Storybook パスが含まれているか確認

3. **D1 データベース接続エラー**

   - `wrangler.toml`の設定を確認
   - ローカル D1 インスタンスが起動しているか確認

4. **e-Stat API エラー**
   - API キーが正しく設定されているか確認
   - 統計表 ID が正しいか確認

### デバッグ方法

1. **ブラウザの開発者ツール**: CSS と JavaScript のエラーを確認
2. **Storybook のコンソール**: コンポーネントのエラーを確認
3. **API レスポンス**: ネットワークタブで API 呼び出しを確認
4. **D1 ログ**: ローカル D1 インスタンスのログを確認
