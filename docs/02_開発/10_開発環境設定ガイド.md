# 🌏 Next.js × e-Stat API × Cloudflare（D1 / R2）

都道府県ランキング・ダッシュボード開発環境設計ガイド

---

## 🎯 プロジェクト概要

- **Next.js 15**: フロントエンドおよび API ルートを統合
- **e-Stat API**: 都道府県データを取得
- **Cloudflare D1**: 統計データ・ランキングキャッシュ管理（SQLite 互換）
- **Cloudflare R2**: 画像や JSON などの静的アセット保存
- **Cloudflare Pages**: デプロイ環境

複数の開発環境（API 接続あり／なし）を切り替えながら、デザイン、開発、ステージング、本番を安全に運用するためのベストプラクティスをまとめます。

---

## 🏗️ 環境構成

| 環境名          | 用途         | 特徴                                              |
| --------------- | ------------ | ------------------------------------------------- |
| **development** | ローカル開発 | e-Stat API 接続（またはモック）、ローカル D1 使用 |
| **mock**        | デザイン検証 | API 非接続、DB 非接続、モックデータ使用           |
| **staging**     | ステージング | D1/R2 接続あり、本番と同構成                      |
| **production**  | 本番         | 実運用データ、最適化ビルド                        |

## ⚙️ 設定コード例

### `/src/lib/config.ts`

```typescript
export const config = {
  env: process.env.NEXT_PUBLIC_ENV || "development",
  useMock: process.env.NEXT_PUBLIC_USE_MOCK === "true",

  estat: {
    baseUrl:
      process.env.ESTAT_API_BASE_URL || "https://api.e-stat.go.jp/rest/3.0/app",
    apiKey: process.env.ESTAT_API_KEY,
  },

  cloudflare: {
    d1: process.env.CLOUDFLARE_D1_DATABASE_ID,
    r2: process.env.CLOUDFLARE_R2_BUCKET,
    accountId: process.env.CLOUDFLARE_ACCOUNT_ID,
    apiToken: process.env.CLOUDFLARE_API_TOKEN,
  },

  mock: {
    dataPath: process.env.MOCK_DATA_PATH || "data/mock",
  },
} as const;

export type Config = typeof config;
```

### `/src/lib/api/estat.ts`

```typescript
import { config } from "@/lib/config";
import mockData from "@/data/mock/prefecture-stats.json";

export async function getPrefectureStats() {
  if (config.useMock) {
    console.log("Using mock data for development");
    return mockData;
  }

  const res = await fetch(
    `${config.estat.baseUrl}/getStatsData?appId=${config.estat.apiKey}&...`
  );

  if (!res.ok) {
    throw new Error("Failed to fetch e-Stat data");
  }

  return res.json();
}
```

## 🗺️ Cloudflare D1 設計

### 環境別構成

| 環境            | D1 の種類                 | 接続方法                      | データベース名    | データベース ID                        | 用途                 |
| --------------- | ------------------------- | ----------------------------- | ----------------- | -------------------------------------- | -------------------- |
| **development** | ローカル SQLite (D1 互換) | `--local`フラグ（デフォルト） | `stats47`         | `e6533698-d05a-475b-9f39-5558703feef7` | ローカル開発・テスト |
| **staging**     | Cloudflare D1（リモート） | `--remote`フラグ              | `stats47_staging` | `39f18714-83bf-423c-9956-dcc9e466affb` | 統合テスト・検証     |
| **production**  | Cloudflare D1（リモート） | `--remote`フラグ              | `stats47`         | `e6533698-d05a-475b-9f39-5558703feef7` | 本番運用             |

### 重要な注意事項

⚠️ **環境とデータベースの対応関係**

- **development 環境**: 常に**ローカル D1**を使用します。データは `.wrangler/state/v3/d1` に保存されます。
- **staging 環境**: 常に**リモート Cloudflare D1**（`stats47_staging`）を使用します。`--remote`フラグが必要です。
- **production 環境**: 常に**リモート Cloudflare D1**（`stats47`）を使用します。`--remote`フラグが必要です。

🔑 **`--remote`フラグの使い分け**

```bash
# ❌ 間違い: staging環境でローカルDBにアクセスしてしまう
npx wrangler d1 execute stats47_staging --env staging --command "SELECT * FROM users;"

# ✅ 正しい: staging環境のリモートDBにアクセス
npx wrangler d1 execute stats47_staging --env staging --remote --command "SELECT * FROM users;"

# ✅ development環境はローカルDB（--remoteフラグ不要）
npx wrangler d1 execute stats47 --local --command "SELECT * FROM users;"
```

### DB 接続例

```typescript
export const runtime = "edge";

export async function getDB() {
  if (process.env.NEXT_PUBLIC_ENV === "local") {
    return (await import("@cloudflare/workers-types")).env.STATS47_DB;
  } else {
    const response = await fetch(
      `https://api.cloudflare.com/client/v4/accounts/${process.env.CLOUDFLARE_ACCOUNT_ID}/d1/database/${process.env.CLOUDFLARE_D1_DATABASE_ID}/query`,
      {
        headers: {
          Authorization: `Bearer ${process.env.CLOUDFLARE_API_TOKEN}`,
        },
      }
    );
    return response;
  }
}
```

## 🧩 マイグレーション運用

| 項目             | 内容                               |
| ---------------- | ---------------------------------- |
| スキーマ管理     | `database/schemas/` を Git 管理    |
| Seed データ      | `database/seeds/`                  |
| マイグレーション | `npx wrangler d1 migrations apply` |
| 同期方向         | `local → staging → production`     |

> 本番環境では「自動 migrate 禁止」。  
> PR マージ後、CI/CD で明示的に実行。

### マイグレーション実行例

```bash
# Development環境（ローカルD1）
npx wrangler d1 migrations apply stats47 --local

# Staging環境（リモートD1） ⚠️ --remote フラグが必要
npx wrangler d1 migrations apply stats47_staging --env staging --remote

# Production環境（リモートD1） ⚠️ --remote フラグが必要
npx wrangler d1 migrations apply stats47 --remote
```

⚠️ **重要**: Staging/Production 環境では必ず`--remote`フラグを使用してください。フラグを忘れるとローカル DB にマイグレーションが適用されてしまいます。

## 🔄 データベース同期戦略

### スキーマの同期（マイグレーション）

#### ローカル → ステージング → 本番

```bash
# 1. ローカルでマイグレーションファイルを作成
npx wrangler d1 migrations create stats47 "add_new_column"

# 2. マイグレーションファイルを編集
# database/migrations/0001_add_new_column.sql

# 3. Development環境（ローカルD1）で適用・テスト
npx wrangler d1 migrations apply stats47 --local

# 4. Staging環境（リモートD1）に適用 ⚠️ --remote必須
npx wrangler d1 migrations apply stats47_staging --env staging --remote

# 5. Production環境（リモートD1）に適用（慎重に！） ⚠️ --remote必須
npx wrangler d1 migrations apply stats47 --remote
```

### データの同期（エクスポート/インポート）

#### 本番 → ステージングへのデータコピー（推奨方法）

本番データをステージング環境にコピーする場合：

```bash
# 1. 本番DBからデータをエクスポート（リモート）
npx wrangler d1 execute stats47 --remote --command "SELECT * FROM ranking_items;" --json > database/backups/$(date +%Y%m%d)/ranking_items.json

# 2. スクリプトでSQLに変換
node scripts/import-to-staging.js

# 3. Staging環境（リモートD1）にインポート ⚠️ --remote必須
npx wrangler d1 execute stats47_staging --env staging --remote --file=database/backups/$(date +%Y%m%d)/import_to_staging.sql
```

#### 方法 1: 特定テーブルのみエクスポート

```bash
# Development環境（ローカルD1）からエクスポート
npx wrangler d1 execute stats47 --local --command "SELECT * FROM ranking_items;" --json > database/backups/ranking_items.json

# Production環境（リモートD1）からエクスポート ⚠️ --remote必須
npx wrangler d1 execute stats47 --remote --command "SELECT * FROM ranking_items;" --json > database/backups/ranking_items_prod.json
```

⚠️ **注意**: `.dump`コマンドは Cloudflare D1 のリモート接続ではサポートされていません。テーブルごとにエクスポートする必要があります。

### データ同期の推奨フロー

#### 1. **Development 環境（ローカル D1）**

```
e-Stat API → ローカルD1 → 開発・テスト
```

- **データベース**: `.wrangler/state/v3/d1`に SQLite ファイル
- **接続方法**: `--local`フラグ（デフォルト）
- **用途**:
  - e-Stat API からデータ取得
  - ローカル D1 に保存
  - 機能開発・テスト実施

#### 2. **Staging 環境（リモート Cloudflare D1）**

```
Production DB → Staging DB (リモートコピー) → 統合テスト
```

- **データベース**: Cloudflare D1 `stats47_staging` (ID: `39f18714-83bf-423c-9956-dcc9e466affb`)
- **接続方法**: `--env staging --remote`フラグ **必須**
- **用途**:
  - 本番データのコピーで統合テスト
  - マイグレーションの検証
  - 本番環境と同じ構成でテスト

⚠️ **重要**: Staging 環境は本番データをコピーして使用します。ローカルデータは同期しません。

#### 3. **Production 環境（リモート Cloudflare D1）**

```
e-Stat API → Cloudflare D1 (production) → 本番運用
```

- **データベース**: Cloudflare D1 `stats47` (ID: `e6533698-d05a-475b-9f39-5558703feef7`)
- **接続方法**: `--remote`フラグ **必須**
- **用途**:
  - e-Stat API から直接データ取得
  - 本番サービス提供
  - 読み取りが中心（書き込みは慎重に）

### データベース管理スクリプト

```bash
# 本番DBからステージングDBへのデータコピー
node scripts/import-to-staging.js

# Development環境（ローカルD1）のマイグレーション
npx wrangler d1 migrations apply stats47 --local

# Staging環境（リモートD1）のマイグレーション ⚠️ --remote必須
npx wrangler d1 migrations apply stats47_staging --env staging --remote

# Production環境（リモートD1）のマイグレーション ⚠️ --remote必須
npx wrangler d1 migrations apply stats47 --remote
```

### 重要な注意事項

⚠️ **環境とデータベースの対応関係**

- **Development**: **ローカル D1 のみ**使用（`--local`フラグ）
- **Staging**: **リモート D1 のみ**使用（`--env staging --remote`フラグ **必須**）
- **Production**: **リモート D1 のみ**使用（`--remote`フラグ **必須**）

⚠️ **データ同期の原則**

- **スキーマ（マイグレーション）**: `Development → Staging → Production` の順で同期
- **データ**:
  - Development: e-Stat API から取得してローカル D1 に保存
  - Staging: 本番 DB からコピー（統合テスト用）
  - Production: e-Stat API から直接取得
- **ローカルデータを本番/ステージングに同期しない**: 各環境でデータソースが異なる

✅ **推奨される方法**

- Development 環境では`--local`フラグを使用
- Staging/Production 環境では**必ず**`--remote`フラグを使用
- Staging 環境のテストには本番データのコピーを使用
- マイグレーション（スキーマ変更）は段階的に適用
- データのバックアップは定期的に実施

❌ **避けるべき方法**

- Staging/Production 環境で`--remote`フラグを忘れる（ローカル DB に誤って適用される）
- ローカルの開発データを本番/ステージングにコピー
- 本番データをローカルで直接編集
- マイグレーションなしでのスキーマ変更
- 環境変数の誤設定

## 🧠 モックデータ構成

```
data/mock/
├── metainfo/                    # メタ情報モックデータ
│   ├── prefecture/              # 都道府県メタ情報
│   │   └── 0000010101.json      # 都道府県別人口データ
│   └── municipality/            # 市区町村メタ情報
│       └── 0000020201.json      # 市区町村別人口データ
├── statsdata/                   # 統計データモックデータ
│   ├── prefecture/              # 都道府県統計データ
│   │   └── 0000010101_A1101.json # 都道府県別人口統計
│   └── municipality/            # 市区町村統計データ
│       └── 0000020201_A1101.json # 市区町村別人口統計
├── statslist/                   # 統計表リストモックデータ（空）
├── catalog.json                 # データカタログ
└── README.md                    # モックデータ管理ガイド
```

`NEXT_PUBLIC_USE_MOCK=true` の場合のみ利用。  
Storybook やデザイン検証でも共通使用可能。

> **用語の統一**: 本ドキュメントでは「モックデータ」と「スタブデータ」は同じ意味で使用しています。

### モックデータの特徴

- **e-Stat API の実際のレスポンス形式**を再現
- **都道府県・市区町村別**の階層構造
- **メタ情報と統計データ**を分離管理
- **データベース接続不要**（ローカル JSON ファイルのみ）
- **Git 管理対象**（複数端末での共有）
- **デプロイ時は除外**（Cloudflare Pages）

### データフロー比較

| 環境            | データソース  | データベース  | 用途         |
| --------------- | ------------- | ------------- | ------------ |
| **development** | e-Stat API    | ローカル D1   | 開発・テスト |
| **mock**        | ローカル JSON | なし          | デザイン検証 |
| **staging**     | e-Stat API    | Cloudflare D1 | 統合テスト   |
| **production**  | e-Stat API    | Cloudflare D1 | 本番運用     |

## 🧭 CI/CD・環境切替例

### cross-env を使用した環境変数管理

本プロジェクトでは`cross-env`パッケージを使用して、クロスプラットフォーム対応の環境変数設定を実現しています。

#### cross-env を使用する理由

- **クロスプラットフォーム対応**: Windows、macOS、Linux で同じコマンドが動作
- **環境変数の統一**: OS ごとの環境変数設定の違いを吸収
- **Next.js 対応**: Next.js の`--env-file`オプション非対応を解決
- **シンプルな記述**: コマンドライン上で直接環境変数を設定

### 開発環境

```bash
# 開発（API接続あり）
npm run dev:api
# → cross-env NODE_ENV=development next dev --turbopack

# デザイン確認（モックデータ）
npm run dev:mock
# → cross-env NODE_ENV=development NEXT_PUBLIC_USE_MOCK=true next dev --turbopack

# ステージング環境での開発
npm run dev:staging
# → cross-env NODE_ENV=production NEXT_PUBLIC_ENV=staging next dev --turbopack

# 本番環境での開発
npm run dev:production
# → cross-env NODE_ENV=production NEXT_PUBLIC_ENV=production next dev --turbopack
```

### デプロイ

```bash
# ステージングデプロイ
npx wrangler pages deploy --env staging

# 本番デプロイ
npx wrangler pages deploy --env production
```

### データベース操作

```bash
# ローカルD1でスキーマ適用
npx wrangler d1 execute stats47 --local --file=./database/schemas/main.sql

# リモートD1でスキーマ適用
npx wrangler d1 execute stats47 --remote --file=./database/schemas/main.sql
```
