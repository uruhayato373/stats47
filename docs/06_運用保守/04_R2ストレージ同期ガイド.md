# R2ストレージ同期ガイド

## 概要

このドキュメントは、stats47プロジェクトのR2ストレージを環境間で同期するための詳細な手順を説明します。

開発環境で作成・更新した静的データ（地域マスタ、地理データ、ブログコンテンツ）をステージング環境、本番環境へ順次同期する方法を説明します。

## 同期フロー

```
開発環境 (stats47-development)
    ↓ [同期]
ステージング環境 (stats47-staging)
    ↓ [同期]
本番環境 (stats47)
```

## 同期対象データ

以下のデータのみを環境間で同期します：

| データ種類 | ディレクトリ | 説明 |
|-----------|------------|------|
| **静的マスタデータ** | `area/` | 都道府県・市区町村マスタ（全環境で同一であるべき） |
| **地理データ** | `geoshape/` | TopoJSONデータ（全環境で同一であるべき） |
| **ブログコンテンツ** | `content/blog/` | ローカルで更新したMDXファイル関連 |

以下のデータは同期しません（環境ごとに独立）：

| データ種類 | ディレクトリ | 理由 |
|-----------|------------|------|
| **APIキャッシュ** | `estat-api/` | 各環境でランタイムに生成される |
| **ランキングデータ** | `ranking/` | 各環境でランタイムに生成される |

## 前提条件

### 環境変数の設定

同期スクリプトを実行するには、以下の環境変数が必要です：

```bash
CLOUDFLARE_ACCOUNT_ID=your_account_id
CLOUDFLARE_R2_ACCESS_KEY_ID=your_access_key_id
CLOUDFLARE_R2_SECRET_ACCESS_KEY=your_secret_access_key
```

### R2バケットの確認

以下のバケットが存在することを確認してください：

- `stats47-development` - 開発環境用バケット
- `stats47-staging` - ステージング環境用バケット
- `stats47` - 本番環境用バケット

## 同期手順

### 開発→ステージング同期

#### 1. ドライランで確認

変更内容を確認するために、まずドライランを実行します：

```bash
npm run sync:r2:dev-to-staging:dry
```

ドライランでは実際の同期は行われず、同期対象のオブジェクト一覧が表示されます。

#### 2. 実際に同期を実行

ドライランの結果を確認後、実際に同期を実行します：

```bash
npm run sync:r2:dev-to-staging
```

#### 3. 同期結果の確認

同期が完了すると、以下の情報が表示されます：

- 合計オブジェクト数
- 同期成功数
- スキップ数
- 失敗数（エラーの詳細も表示）

### ステージング→本番同期

#### 1. ドライランで確認

本番環境への同期は慎重に行う必要があります。まずドライランで確認します：

```bash
npm run sync:r2:staging-to-prod:dry
```

#### 2. 実際に同期を実行

ステージング環境でテスト完了後、本番環境へ同期します：

```bash
npm run sync:r2:staging-to-prod
```

**注意**: 本番環境への同期は本番環境のデータを上書きします。必ず事前にドライランで確認してください。

## デプロイ時の同期フロー

### ステージング環境へのデプロイ

1. **コードをデプロイ**
   ```bash
   # コードをデプロイ
   npm run pages:build
   npx wrangler pages deploy .open-next --env staging
   ```

2. **D1データベースを同期**
   ```bash
   npm run db:sync:local-to-staging
   ```

3. **R2ストレージを同期**
   ```bash
   npm run sync:r2:dev-to-staging
   ```

### 本番環境へのデプロイ

1. **ステージング環境でテスト完了後、コードをデプロイ**
   ```bash
   # コードをデプロイ
   npm run pages:build
   npx wrangler pages deploy .open-next --env production
   ```

2. **R2ストレージを同期**
   ```bash
   npm run sync:r2:staging-to-prod
   ```

## トラブルシューティング

### エラー: 環境変数が設定されていません

**症状**:
```
❌ 環境変数が設定されていません:
   - CLOUDFLARE_ACCOUNT_ID
   - CLOUDFLARE_R2_ACCESS_KEY_ID
   - CLOUDFLARE_R2_SECRET_ACCESS_KEY
```

**対処法**:
1. 環境変数が設定されているか確認
2. `.env`ファイルまたは環境変数設定を確認
3. CloudflareダッシュボードでR2 APIトークンが作成されているか確認

### エラー: 認証エラー

**症状**:
```
❌ エラー: Access Denied
```

**対処法**:
1. R2 APIトークンの権限を確認（Read & Writeが必要）
2. トークンが有効期限内か確認
3. バケット名が正しいか確認

### エラー: バケットが見つかりません

**症状**:
```
❌ エラー: NoSuchBucket
```

**対処法**:
1. バケットが作成されているか確認
   ```bash
   npx wrangler r2 bucket list
   ```
2. バケット名のタイプミスがないか確認
3. バケットが削除されていないか確認

### エラー: 一部のオブジェクトの同期に失敗

**症状**:
```
失敗: 5件
エラー詳細:
  - area/prefectures.json: Access Denied
  ...
```

**対処法**:
1. エラー詳細を確認
2. 個別のオブジェクトの権限を確認
3. オブジェクトが存在するか確認
4. 必要に応じて、失敗したオブジェクトを個別に確認

## よくある質問

### Q: 同期はどのくらいの時間がかかりますか？

A: 同期対象のオブジェクト数とファイルサイズによって異なります。通常、数百件のオブジェクトであれば数分程度です。

### Q: 同期中に既存のデータは削除されますか？

A: いいえ。同期スクリプトは既存のオブジェクトを上書きするだけで、削除は行いません。ソースバケットに存在しないオブジェクトは、ターゲットバケットに残ります。

### Q: 部分的な同期は可能ですか？

A: はい。同期スクリプトは特定のプレフィックス（ディレクトリ）のみを同期するように設計されています。現在は `area/`、`geoshape/`、`content/blog/` のみが同期対象です。

### Q: 同期を自動化できますか？

A: はい。GitHub ActionsなどのCI/CDパイプラインに同期スクリプトを組み込むことができます。ただし、本番環境への同期は手動実行を推奨します。

### Q: 同期対象のデータを変更したい場合は？

A: `scripts/sync-r2-development-to-staging.ts` と `scripts/sync-r2-staging-to-production.ts` の `prefixes` 配列を編集してください。

## ベストプラクティス

### 1. 必ずドライランで確認

本番環境への同期前に、必ずドライランで変更内容を確認してください。

```bash
npm run sync:r2:staging-to-prod:dry
```

### 2. ステージング環境でテスト

本番環境へ同期する前に、ステージング環境で十分にテストしてください。

### 3. バックアップの取得

本番環境への同期前に、必要に応じてバックアップを取得してください。

### 4. エラーログの確認

同期が失敗した場合は、エラーログを確認し、問題を特定してください。

### 5. 段階的な同期

大量のデータを同期する場合は、段階的に同期することを検討してください。
