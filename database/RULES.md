# データベース管理ルール

このドキュメントは、プロジェクトのデータベース管理におけるルールとガイドラインを定義します。

## 🚨 **必須ルール**

### **1. スキーマ変更時のルール**

#### **1.1 変更前の必須作業**

- [ ] 現在のスキーマのバックアップを作成
- [ ] 変更内容をチームメンバーに通知
- [ ] 影響範囲の分析と文書化
- [ ] テスト環境での事前検証

#### **1.2 変更時の必須作業**

- [ ] 新しいマイグレーションファイルを作成
- [ ] 変更理由と影響範囲をコメントに記載
- [ ] ロールバック手順を文書化
- [ ] スキーマバージョンを更新

#### **1.3 変更後の必須作業**

- [ ] 本番環境での動作確認
- [ ] パフォーマンステストの実行
- [ ] チームメンバーへの変更内容共有
- [ ] ドキュメントの更新

### **2. ファイル命名ルール**

#### **2.1 スキーマファイル**

```
[機能名].sql
例: auth.sql, estat-metadata.sql, user-profile.sql
```

#### **2.2 マイグレーションファイル**

```
[連番]_[説明]_[日付].sql
例: 001_initial_schema_20241219.sql
     002_add_user_profile_20241220.sql
     003_modify_estat_index_20241221.sql
```

#### **2.3 バックアップファイル**

```
[テーブル名]_[日時].csv
例: users_20241219_143022.csv
     estat_metadata_20241219_143022.csv
```

### **3. SQL コーディングルール**

#### **3.1 基本ルール**

- すべての SQL ファイルは UTF-8 エンコーディングを使用
- インデントは 2 スペースを使用
- 各ステートメントの末尾にセミコロンを付ける
- 長い SQL は適切に改行して可読性を向上

#### **3.2 テーブル作成ルール**

```sql
-- テーブル作成時の必須項目
CREATE TABLE IF NOT EXISTS [テーブル名] (
  id INTEGER PRIMARY KEY AUTOINCREMENT,  -- 必須: 主キー
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,  -- 必須: 作成日時
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,  -- 必須: 更新日時
  -- その他のカラム
);

-- 必須: 更新日時のトリガー
CREATE TRIGGER IF NOT EXISTS [テーブル名]_updated_at
  AFTER UPDATE ON [テーブル名]
  BEGIN
    UPDATE [テーブル名] SET updated_at = CURRENT_TIMESTAMP WHERE id = NEW.id;
  END;
```

#### **3.3 インデックス作成ルール**

```sql
-- 基本インデックス（必須）
CREATE INDEX IF NOT EXISTS idx_[テーブル名]_[カラム名] ON [テーブル名]([カラム名]);

-- 複合インデックス（検索パフォーマンス向上）
CREATE INDEX IF NOT EXISTS idx_[テーブル名]_[用途] ON [テーブル名]([カラム1], [カラム2]);

-- ユニーク制約（データ整合性）
CREATE UNIQUE INDEX IF NOT EXISTS idx_[テーブル名]_unique ON [テーブル名]([カラム1], [カラム2]);
```

### **4. データ操作ルール**

#### **4.1 データ挿入・更新**

- 大量データの挿入時はバッチ処理を使用
- トランザクション内で実行し、エラー時はロールバック
- 外部キー制約を適切に設定

#### **4.2 データ削除**

- 物理削除ではなく論理削除を優先
- 削除前のバックアップ作成を必須とする
- 削除対象の確認を複数人で実施

#### **4.3 データ検索**

- パフォーマンスを考慮したクエリの作成
- 適切なインデックスの活用
- 結果件数の制限（LIMIT 句の使用）

### **5. セキュリティルール**

#### **5.1 アクセス制御**

- 必要最小限の権限のみを付与
- データベース接続情報の適切な管理
- 定期的な権限の見直し

#### **5.2 データ保護**

- 機密データの暗号化
- ログファイルへの機密情報出力禁止
- 定期的なセキュリティ監査

### **6. パフォーマンスルール**

#### **6.1 クエリ最適化**

- EXPLAIN QUERY PLAN の活用
- 不要な JOIN の回避
- 適切な WHERE 句の設定

#### **6.2 インデックス管理**

- 使用頻度の高いクエリに最適化
- 不要なインデックスの定期的な削除
- インデックスサイズの監視

## 🔧 **推奨プラクティス**

### **1. 開発フロー**

1. **設計**: スキーマ設計のレビュー
2. **実装**: マイグレーションファイルの作成
3. **テスト**: テスト環境での検証
4. **デプロイ**: 本番環境への適用
5. **監視**: パフォーマンスとエラーの監視

### **2. ドキュメント管理**

- 各テーブルの目的と用途を明確化
- リレーションシップの図解
- サンプルデータの提供
- よくある問題と解決方法の文書化

### **3. 監視とメンテナンス**

- 定期的なパフォーマンス監視
- データベースサイズの監視
- インデックス使用率の確認
- エラーログの定期的な確認

## 🚨 **禁止事項**

### **1. 絶対にやってはいけないこと**

- 本番環境での直接的なスキーマ変更
- バックアップなしでのデータ操作
- チームメンバーへの事前通知なしでの変更
- テスト環境での検証なしでのデプロイ

### **2. 避けるべきこと**

- 大量データの一括削除
- パフォーマンスを考慮しないクエリ
- 適切なインデックスのない検索
- 外部キー制約の無視
