# 環境変数管理の簡素化プラン

## 概要

環境を`development`、`staging`、`production`の 3 つに統一し、モックデータの使用可否は`NEXT_PUBLIC_USE_MOCK`フラグで制御するよう変更します。これにより Next.js の標準的な環境変数管理に従い、`.env.development`を使用できるようになります。

## 現状の問題

1. `NEXT_PUBLIC_ENV`で`mock`という環境を定義している
2. カスタム環境ファイル`.env.mock`を使用しているが、Next.js が標準で認識しない
3. `dotenv`を使った手動読み込みが必要で複雑
4. `AUTH_SECRET`が読み込まれずエラーが発生

## 新しい設計

### 環境の定義

- **development**: ローカル開発環境
- **staging**: ステージング環境
- **production**: 本番環境

### モックフラグ

- **NEXT_PUBLIC_USE_MOCK**: `true`でモックデータ使用、`false`で API/DB 使用

### 環境ファイル

Next.js の標準的な環境ファイルのみを使用：

- `.env.development` - development 環境用
- `.env.staging` - staging 環境用
- `.env.production` - production 環境用

### 3. src/infrastructure/env/index.ts の更新

```typescript
/**
 * 環境タイプの定義
 */
export type Environment = "development" | "staging" | "production";

/**
 * 現在の環境を取得
 */
export function getEnvironment(): Environment {
  const nodeEnv = process.env.NODE_ENV;

  if (nodeEnv === "production") {
    // NEXT_PUBLIC_ENVで staging/production を判定（オプション）
    return (process.env.NEXT_PUBLIC_ENV as Environment) || "production";
  }

  return "development";
}

/**
 * モックデータを使用するかどうか
 */
export function useMockData(): boolean {
  return process.env.NEXT_PUBLIC_USE_MOCK === "true";
}

/**
 * 環境別の設定を取得
 */
export function getEnvironmentConfig() {
  const env = getEnvironment();
  const isMock = useMockData();

  return {
    environment: env,
    isMock,
    isProduction: env === "production",
    isDevelopment: env === "development",
    isStaging: env === "staging",
  };
}
```

### 4. next.config.ts の更新

dotenv の手動読み込みを削除し、Next.js の標準機能に任せる：

```typescript
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  /* config options here */
  eslint: {
    ignoreDuringBuilds: true,
  },
  typescript: {
    ignoreBuildErrors: false,
  },

  // 本番ビルドではモックデータを除外
  webpack: (config, { dev, isServer }) => {
    if (!dev && !isServer) {
      config.resolve.alias = {
        ...config.resolve.alias,
        "@/data/mock": false,
      };
    }
    return config;
  },

  // 静的ファイルの最適化
  experimental: {
    optimizePackageImports: ["@/lib/mock-data"],
  },
};

export default nextConfig;
```

## メリット

1. **Next.js の標準に従う**: カスタム環境ファイルが不要
2. **シンプル**: `dotenv`の手動読み込みが不要
3. **明確**: 環境とモードを明確に分離
4. **保守性向上**: 標準的な方法で管理できる
5. **エラー解消**: `AUTH_SECRET`が`.env.development`から自動で読み込まれる

## デメリット（対応策あり）

1. **環境変数の切り替え**: スクリプトレベルで`NEXT_PUBLIC_USE_MOCK`を設定
2. **`.env.development`の共有**: 開発者間で共有（機密情報は`.env.development.local`で上書き可能）

## テスト手順

1. `.env.development`を作成
2. `npm run dev:mock`でモックデータが使用されることを確認
3. `npm run dev:api`で e-Stat API が使用されることを確認
4. `AUTH_SECRET`エラーが解消されることを確認
