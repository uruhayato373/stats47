# リファクタリングロードマップ

## 概要

stats47 プロジェクトの技術的負債を解消し、コードベースの品質と保守性を向上させるための包括的なリファクタリング計画です。

## 完了済みリファクタリング

### ✅ 1. 状態管理の統一（2025-10-16）

**問題**: `src/contexts` と `src/atoms` で状態管理が混在

**解決策**: Jotai への一本化

**成果**:

- 未使用の React Context API コードを削除
- 状態管理ライブラリを 1 つに統一
- パフォーマンスと開発体験の向上

**影響ファイル**:

- `src/contexts/ThemeContext.tsx` (削除)
- `src/contexts/` ディレクトリ (削除)
- `src/app/layout.tsx` (コメントアウト行削除)

### ✅ 2. API エンドポイントの統合（2025-10-16）

**問題**: `/api/ranking-items` と `/api/ranking` の重複

**解決策**: `/api/rankings/` への統合

**成果**:

- RESTful API 設計の一貫性向上
- ドメイン設計との整合性確保
- エンドポイントの明確化

**影響ファイル**:

- `src/app/api/ranking-items/` → `src/app/api/rankings/items/`
- `src/app/api/ranking/` → `src/app/api/rankings/`
- フロントエンドの API 呼び出し更新

### ✅ 3. ディレクトリ構造の最適化（2025-10-16）

**問題**: 不明確なディレクトリ名と配置

**解決策**: ドメイン設計に基づく再配置

**成果**:

- `src/lib/db/` → `src/lib/database/`
- `src/app/api/estat-api/data/` → `src/app/api/estat-api/stats-data/`
- `src/app/api/estat-api/ranking-values/` → `src/app/api/rankings/data/`

### ✅ 4. 環境変数の統一（2025-10-16）

**問題**: `NEXT_PUBLIC_USE_MOCK` と `NEXT_PUBLIC_ENV` の重複

**解決策**: `NEXT_PUBLIC_ENV` への統一

**成果**:

- 環境変数設定の簡素化
- 設定の一貫性向上
- ドキュメントの更新

### ✅ 5. ドキュメント管理の改善（2025-10-16）

**問題**: Obsidian での管理に必要なフロントマターが不足

**解決策**: 全 Markdown ファイルにフロントマター追加

**成果**:

- 124 ファイルにフロントマター追加
- タグベースの整理が可能
- ドキュメント検索性の向上

- ユーザーフレンドリーなエラーメッセージ

### 🔄 9. パフォーマンス最適化（優先度: 中）

### 🔄 10. テストカバレッジの向上（優先度: 中）

## リファクタリング原則

### 1. 段階的実行

- 小さな単位で実行
- 各段階でテストと検証
- ロールバック可能な設計

### 2. 後方互換性の維持

- 既存機能への影響を最小化
- 段階的な移行パスを提供
- 適切な通知とドキュメント

### 3. 品質の向上

- コードレビューの徹底
- 自動テストの充実
- パフォーマンス監視

### 4. ドキュメントの更新

- 変更内容の記録
- 移行ガイドの提供
- ベストプラクティスの共有

## メトリクスと KPI

### コード品質

- **複雑度**: サイクロマティック複雑度の削減
- **重複**: コード重複率の削減
- **保守性**: 保守性指数の向上

### パフォーマンス

- **バンドルサイズ**: 初期バンドルサイズの削減
- **ロード時間**: ページロード時間の短縮
- **ランタイム**: メモリ使用量の最適化

### 開発体験

- **ビルド時間**: 開発ビルド時間の短縮
- **テスト実行時間**: テストスイート実行時間の短縮
- **開発者満足度**: チーム内アンケート

## リスク管理

### 技術的リスク

- **回帰バグ**: 既存機能の破壊
- **パフォーマンス劣化**: リファクタリングによる性能低下
- **学習コスト**: 新しいパターンの習得

### 対策

- **段階的リリース**: フィーチャーフラグの活用
- **自動テスト**: 回帰テストの充実
- **監視**: パフォーマンス監視の強化
- **教育**: チーム内での知識共有
